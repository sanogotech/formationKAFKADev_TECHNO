Contexte & historique du GUCE CI

Le Guichet Unique du Commerce Extérieur de Côte d’Ivoire (GUCE CI) a été lancé officiellement le 1er juillet 2013 par l’État ivoirien pour dématérialiser l’ensemble des procédures douanières et portuaires. Avant cette plateforme, les opérateurs devaient se déplacer physiquement dans plusieurs administrations (Douanes, Port Autonome d’Abidjan, Ministère du Commerce, Trésor Public, etc.) pour déposer des documents papier – un processus pouvant prendre plusieurs semaines.

En 2020, le GUCE CI est devenu une société d’État à majorité publique (décret n°2020-xxx), renforçant la maîtrise nationale de l’outil. Depuis, la plateforme traite plus de 500 000 déclarations par an et génère plusieurs milliards de FCFA de recettes douanières. L’objectif affiché est de passer d’une simple dématérialisation à une plateforme intelligente, temps réel et orientée événements, capable de détecter la fraude et d’interopérer avec les ports ouest-africains.

Définitions fondamentales

DAU : Déclaration en Détail Unique – document douanier central pour importer/exporter.
API : Autorisation Préalable à l’Importation – délivrée par le Ministère du Commerce.
FDI : Fiche de Déclaration à l’Importation – générée automatiquement par le GUCE.
PCS : Port Community System – système d’échange d’informations portuaires.
e-Manifeste : déclaration électronique du chargement d’un navire.
Mainlevée : autorisation de sortie des marchandises après paiement des taxes.

Domain-Driven Design – Modélisation métier

Le Domain-Driven Design (DDD) est une approche de modélisation logicielle qui place le cœur métier au centre des décisions d’architecture. Pour le GUCE CI, cela signifie découper le système en contextes bornés (bounded contexts) alignés sur les processus réels du commerce extérieur : déclarations, manifestes, paiements, contrôles, etc. Chaque contexte borné devient un microservice autonome avec son propre langage ubiquitaire (ubiquitous language).

Pourquoi DDD ? Le GUCE CI implique plusieurs administrations (Douanes, Port, Commerce, Finances) aux vocabulaires différents. DDD évite les confusions sémantiques (ex. “validation” n’a pas le même sens pour un agent des douanes et un transitaire).

Ubiquitous Language
Langage commun partagé entre métier et technique.
Bounded Context
Limite explicite d’un modèle métier.
Aggregate
Groupe d’entités traitées comme une unité.
Domain Event
Événement métier important (ex. “DAU soumise”).

Tips terrain :
  • Commencez par l’Event Storming avec des scénarios réels (ex. “un transitaire soumet une DAU pour un conteneur de produits alimentaires”).
  • Ne cherchez pas la perfection dès le début – itérez sur la délimitation des contextes.
  • Utilisez un Glossaire partagé (accessible à tous) pour maintenir le langage ubiquitaire.
  • Documentez les contrats entre contextes (ex. API de messagerie asynchrone) avant d’implémenter.
  • Ne tombez pas dans le piège du “microservice trop petit” – un contexte borné doit avoir une taille cohérente (idéalement 2–4 personnes en maintenance).
Questions clés à se poser avant de découper les contextes
  1. Quels sont les processus métier qui changent le plus fréquemment ? (ceux-là méritent leur propre contexte)
  2. Existe-t-il des termes métier qui n’ont pas la même signification selon le service ? (ex. “valider” pour douane vs “valider” pour finance)
  3. Quels sous-domaines sont “critiques” (core domain) et lesquels sont “support” ? (le core domain mérite plus d’attention)
  4. Les équipes sont-elles organisées autour des contextes ? (loi de Conway)
  5. Quels sont les besoins d’intégration avec des systèmes externes (PCS, banques) ?

Architecture cible – Vue d’ensemble

L’architecture cible du GUCE CI est une architecture orientée événements (EDA), combinant des microservices, des flux asynchrones via Apache Kafka, et des composants spécialisés pour la capture de données (CDC), le traitement de flux, le cache et l’observabilité. Elle s’articule autour de 6 couches logicielles :

  1. Couche d’accès (Front) – Interfaces web, portails transitaires, applications mobiles.
  2. Couche API Gateway & microservices – Exposition des API REST, orchestration des commandes.
  3. Couche événementielle (colonne vertébrale) – Apache Kafka pour les flux asynchrones.
  4. Couche de traitement de flux – Kafka Streams (léger) et Apache Flink (lourd).
  5. Couche de persistance et cache – Bases transactionnelles (PostgreSQL), Redis, MinIO.
  6. Couche d’observabilité – ELK Stack pour logs, métriques, audit.
Flux typique : Un transitaire soumet une DAU via le Front → API REST → microservice déclaration → écriture en base + publication d’un événement Kafka → Debezium capture la modification (redondance) → service paiement réagit → mainlevée → notification au port via PCS.

Apache Kafka – Détails techniques
Bonnes pratiques
  • Utiliser des clés de partition métier (ex. declarationId) pour garantir l’ordre.

  • Rétention infinie (coût de stockage explosif).

Debezium – CDC pour bases transactionnelles

Debezium se connecte aux logs de transaction (binlog MySQL, WAL PostgreSQL, LogMiner Oracle) et transforme chaque insertion, mise à jour ou suppression en événement Kafka. Il est non intrusif (pas de code à modifier) et garantit exactly‑once avec Kafka.

Pattern associé : Transactional Outbox
Les microservices écrivent leurs événements dans une table outbox au sein de la même transaction métier. Debezium surveille cette table et publie les événements dans Kafka. Ainsi, on garantit l’atomicité : pas d’événement sans engagement en base.
Bonnes pratiques
  • Configurer la rétention des logs de la base (ex. 7 jours) pour permettre la reprise Debezium.
  • Utiliser un schéma Avro avec Schema Registry pour l’évolution des données.
  • Surveiller le lag du connecteur (offset vs dernière transaction).
Mauvaises pratiques
  • Supprimer trop tôt les logs de transaction (perte de données).
  • Appliquer des transformations complexes dans Debezium (préférer Kafka Streams).
  • Ignorer les événements de type DELETE (caches obsolètes).

ELK Stack – Supervision et audit

Logstash collecte les logs de tous les microservices (format JSON, avec correlationId), Elasticsearch les indexe, Kibana permet les tableaux de bord et les alertes. Cas d’usage :

  • Recherche centralisée : retrouver tous les événements liés à une déclaration en quelques secondes.
  • Audit métier : visualiser l’historique complet des modifications (grâce aux événements Kafka également stockés).
  • Supervision technique : latence des microservices, erreurs HTTP, lag Kafka.
  • Alerting : notifier l’équipe d’exploitation en cas de seuil dépassé (ex. plus de 5% d’erreurs paiement).
Bonnes pratiques ELK :
  • Structurer les logs en JSON, avec des champs obligatoires (timestamp, service, correlationId, level).
  • Utiliser des index rollover pour gérer la volumétrie.
  • Mettre en place un pipeline Logstash pour enrichir (géolocalisation, ajout de champs métier).
  • Limiter l’accès à Kibana par rôle (RBAC).

Bonnes pratiques pour utiliser les arbres de décision
  • Documentez chaque décision avec une Architecture Decision Record (ADR) en citant l’arbre utilisé.
  • Révisez périodiquement les arbres (tous les 6 mois) car les technologies et les besoins évoluent.
  • Impliquez les parties prenantes (métier, ops, dev) lors de l’application des arbres.
  • Ne suivez pas aveuglément un arbre ; tenez compte du contexte spécifique (ex. contraintes réglementaires).
  • Combinez plusieurs arbres pour des décisions complexes (ex. choix final d’une stack).
Checklist – Valider une décision architecturale avec les arbres
  • ✅ L’arbre a-t-il été parcouru collectivement (au moins 2 personnes) ?
  • ✅ Les réponses aux questions sont-elles basées sur des faits (pas des intuitions) ?
  • ✅ La décision est-elle documentée dans un ADR ?
  • ✅ Un prototype ou un POC a-t-il validé la branche choisie ?
  • ✅ Les impacts sur la sécurité, les coûts, la maintenance ont-ils été évalués ?
  • ✅ La décision est-elle réversible (coût de changement évalué) ?
Questions clés à se poser systématiquement
  1. Quel est le compromis entre performance et simplicité dans cette décision ?
  2. Quel est le coût total de possession (licences, infrastructure, formation) sur 3 ans ?
  3. Quelles sont les contraintes réglementaires (ex. traçabilité, hébergement des données) ?
  4. L’équipe actuelle a-t-elle les compétences pour maintenir la solution choisie ?
  5. Existe-t-il une solution plus simple (moins de technologies) qui pourrait suffire ?
  6. Quel est le plan de sortie (rollback) si la décision s’avère mauvaise ?
