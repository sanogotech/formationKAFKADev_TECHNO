Compréhension du workflow
  1. Émission : L’application cliente produit un événement TransactionCreated sur le topic banking-transactions.
  2. Détection fraude : Un consumer lit ce topic, applique des règles et écrit les alertes dans banking-fraud-alerts.
  3. Mise à jour ledger : Un second consumer met à jour le solde du compte et produit l’état final sur banking-ledger (topic compacté).

🎯 Objectif : Créer les sujets Kafka qui serviront de colonne vertébrale aux transactions bancaires.

📌 Bonnes pratiques : Préfixe métier (banking-), nombre de partitions adapté au débit attendu, réplication pour la tolérance de panne, et politique de rétention spécifique (compactage pour le ledger).

# Topic principal des transactions (débit élevé) – 3 partitions, réplication 1 (dev)
kafka-topics.sh --create --topic banking-transactions \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config retention.ms=604800000   # 7 jours

# Topic des alertes fraude (faible volume)
kafka-topics.sh --create --topic banking-fraud-alerts \
  --bootstrap-server localhost:9092 \
  --partitions 1

# Topic ledger compacté (garder dernière version par compte)
kafka-topics.sh --create --topic banking-ledger \
  --bootstrap-server localhost:9092 \
  --partitions 2 \
  --config cleanup.policy=compact \
  --config delete.retention.ms=86400000
Résultat attendu : Chaque commande retourne Created topic .... Vérifiez avec kafka-topics.sh --list --bootstrap-server localhost:9092.
Commentaire : Le partitionnement par clé (ex: transactionId ou accountId) garantit l’ordre des événements pour une même entité. C’est crucial pour les mises à jour de solde.

Cas d’usage : Une erreur de traitement est détectée (ex: mauvais calcul de frais). Grâce à Kafka, on peut rejouer les transactions depuis un offset spécifique.

# 1. Lire les offsets disponibles
kafka-run-class.sh kafka.tools.GetOffsetShell \
  --topic banking-transactions \
  --bootstrap-server localhost:9092

# 2. Consommer à partir d’un offset précis (ex: partition 0, offset 5)
kafka-console-consumer.sh --topic banking-transactions \
  --bootstrap-server localhost:9092 \
  --partition 0 \
  --offset 5 \
  --property print.key=true

# 3. Rejouer tout depuis le début vers un nouveau topic "banking-reprocessed"
kafka-console-consumer.sh --topic banking-transactions \
  --bootstrap-server localhost:9092 \
  --from-beginning | kafka-console-producer.sh --topic banking-reprocessed --bootstrap-server localhost:9092
REX (Retour d’expérience) : Dans une grande banque, le replay a permis de corriger un bug de conversion de devise sans interrompre les services. La rejouabilité est un argument clé pour les régulateurs (BCE, FDIC).

Compréhension du workflow
  1. Consommation : Le service lit les commandes depuis orders (group stock-service).

  • Gérer les réservations séparément (champ reserved) pour les commandes non encore payées.
  • Surveiller les alertes de rupture via un topic dédié pour déclencher des actions.

Les microservices ne communiquent plus via des appels HTTP synchrones, mais via des événements asynchrones dans Kafka. Résilience, scalabilité indépendante, et possibilité de rejouer l’historique complet.

Compréhension du workflow

Essentiel à retenir (top takeaways)

  • Kafka n’est pas une file de messages, c’est une plateforme de streaming.
  • Partitionner par clé métier (orderId, userId, deviceId) garantit l’ordre.
  • Exactly‑once est possible avec idempotence et transactions.
  • Le replay est une superpuissance : audit, correction, backtesting.
  • Les topics compactés sont idéaux pour les états (ledger, stock).
  • Toujours surveiller les lags des consumer groups.
  • Kafka Connect évite d’écrire des consumers/producers custom.
  • Dead Letter Topic (DLQ) indispensable pour les erreurs.
  • La rétention doit être adaptée au besoin métier (conformité).
  • Un schema registry (Avro) évite les cassures de contrat.

Top 20 bonnes pratiques Kafka

  1. Utiliser des clés de partition métier (ordre garanti).
  2. Activer l’idempotence sur les producers (`enable.idempotence=true`).
  3. Configurer `acks=all` et `min.insync.replicas=2` pour les données critiques.
  4. Définir une rétention adaptée (court pour les logs, long pour les événements métier).
  5. Utiliser la compression (snappy, lz4) pour réduire le volume.
  6. Surveiller les lags des consumer groups (alerte si > seuil).
  7. Séparer les environnements (dev, staging, prod) avec des préfixes de topics.
  8. Privilégier Kafka Connect pour l’intégration de sources/sinks.
  9. Mettre en place une Dead Letter Topic (DLQ) pour les erreurs.
  10. Utiliser un Schema Registry (Avro, Protobuf) pour l’évolution des schémas.
  1. Partitionner suffisamment (mais pas trop) pour le parallélisme.
  2. Répliquer les topics critiques (replication factor ≥ 2).
  3. Tester le replay en production sur un topic de validation.
  4. Documenter les noms de topics et leur schéma.
  5. Utiliser Kafka Streams ou ksqlDB pour les transformations.
  6. Isoler les consumer groups par application.
  7. Mettre en place une politique de sécurité (SSL, SASL, ACL).
  8. Chiffrer les données sensibles (PII, financières).
  9. Automatiser la création des topics (Infra as Code).
  10. Former les équipes aux concepts event‑driven.

Top 20 mauvaises pratiques Kafka (à éviter absolument)

  1. Utiliser un seul topic pour tous les événements (mono‑topic).
  2. Ignorer les clés de partition (ordre non garanti).
  3. Partitionner trop ou pas assez (déséquilibre de charge).
  4. Désactiver l’idempotence en production (risque de doublons).
  5. Ne pas surveiller les lags (découverte tardive des retards).
  6. Ne pas gérer les erreurs (pas de DLQ, pas de retry).
  7. Écrire des consumers sans commit explicite des offsets.
  8. Utiliser Kafka comme une base de données (requêtes aléatoires).
  9. Conserver indéfiniment des données sans rétention.
  10. Sous‑dimensionner le cluster (disque, CPU, réseau).
  1. Envoyer des messages trop gros (batching mal configuré).
  2. Mélanger producteurs et consommateurs dans la même application.
  3. Ignorer les schémas (évolution incompatible).
  4. Ne pas tester le replay (surprise en production).
  5. Oublier la haute disponibilité (replication factor = 1).
  6. Mettre tous les consumer groups sur les mêmes offsets.
  7. Réinitialiser les offsets sans comprendre les conséquences.
  8. Utiliser Kafka sans monitoring (Kafka Exporter, Prometheus).
  9. Négliger la sécurité (accès anonyme en production).
  10. Penser que Kafka résout tout (ne remplace pas une base transactionnelle).
