📌 Apache Kafka – Concepts fondamentaux

Kafka est une plateforme de streaming distribuée. Elle permet de publier, stocker et traiter des flux d'événements en temps réel.

Topic Flux de messages, partitionné et répliqué.
Producer Publie des messages sur un topic.
Consumer Lit les messages d'un topic.
Broker Serveur Kafka (cluster).
Zookeeper/KRaft Coordination et métadonnées.
Streams API Traitement de flux (Java).
ksqlDB Moteur SQL pour streaming.
Kafka Connect Intégration de données.

Use case typique : pipeline de logs, détection de fraude, analytics temps réel, CDC.

⚙️ Kafka Streams en Java

Bibliothèque légère pour transformer des flux de données directement dans votre application Java.

Exemple : compteur de mots (WordCount)

// Dépendance Maven : org.apache.kafka:kafka-streams
Properties props = new Properties();
props.put(StreamsConfig.APPLICATION_ID_CONFIG, "wordcount-app");
props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");

StreamsBuilder builder = new StreamsBuilder();
KStream<String, String> textLines = builder.stream("input-topic");

KTable<String, Long> wordCounts = textLines
    .flatMapValues(line -> Arrays.asList(line.toLowerCase().split("\\W+")))
    .groupBy((key, word) -> word)
    .count();

wordCounts.toStream().to("output-topic", Produced.with(Serdes.String(), Serdes.Long()));

KafkaStreams streams = new KafkaStreams(builder.build(), props);
streams.start();

Bonnes pratiques : Utiliser des Serdes personnalisés, gérer les fenêtres de temps (TimeWindows), activer exactly-once semantics.

🐍 Kafka Streams en Python (Faust)

Faust est une bibliothèque Python inspirée de Kafka Streams, avec support asynchrone.

Exemple : compteur de mots

import faust

app = faust.App('wordcount', broker='kafka://localhost:9092')
input_topic = app.topic('input', value_type=str)
counts = app.Table('wordcount', default=int)

@app.agent(input_topic)
async def process(stream):
    async for value in stream:
        for word in value.lower().split():
            counts[word] = counts.get(word, 0) + 1
            print(f"{word}: {counts[word]}")

if __name__ == '__main__':
    app.main()

Avantages : syntaxe Python, asynchrone, gestion d'état avec RocksDB. Convient pour des prototypages rapides.

Limitation : Moins performant que Java pour les très gros volumes, mais très productif.

📊 ksqlDB – Streaming SQL pour Kafka

ksqlDB est un moteur de traitement de flux basé sur SQL. Permet de faire des requêtes temps réel sans coder.

Installation rapide (Docker)

docker run -d --name ksqldb-server \
  -e KSQL_BOOTSTRAP_SERVERS=localhost:9092 \
  confluentinc/ksqldb-server:latest

Exemple : compteur de mots avec ksqlDB

-- Créer un stream à partir du topic
CREATE STREAM text_input (line VARCHAR) WITH (KAFKA_TOPIC='input', VALUE_FORMAT='JSON');

-- FlatMap et comptage
CREATE TABLE wordcount AS
  SELECT EXPLODE(SPLIT(LCASE(line), ' ')) AS word,
         COUNT(*) AS count
  FROM text_input
  GROUP BY EXPLODE(SPLIT(LCASE(line), ' '))
  EMIT CHANGES;

Fonctionnalités : Jointures de streams, fenêtrage, agrégations, push/pull queries.

🔌 Kafka Connect – Intégration de données

Framework pour importer/exporter des données entre Kafka et d'autres systèmes (bases de données, S3, etc.).

Exemple : Connecteur source JDBC (PostgreSQL -> Kafka)

{
  "name": "jdbc-source-connector",
  "config": {
    "connector.class": "io.confluent.connect.jdbc.JdbcSourceConnector",
    "connection.url": "jdbc:postgresql://localhost:5432/mydb",
    "mode": "incrementing",
    "incrementing.column.name": "id",
    "topic.prefix": "db-",
    "table.whitelist": "users,orders"
  }
}

Connecteurs populaires : JDBC, Elasticsearch, S3, MongoDB, Debezium (CDC).

Bonnes pratiques : Utiliser le mode distribué, surveiller avec JMX, configurer les conversions (Avro, JSON Schema).

🔄 Debezium – Change Data Capture

Debezium est un ensemble de connecteurs Kafka Connect qui capturent les changements (insert, update, delete) dans les bases de données et les publient dans Kafka.

Exemple : MySQL CDC

{
  "name": "mysql-connector",
  "config": {
    "connector.class": "io.debezium.connector.mysql.MySqlConnector",
    "database.hostname": "localhost",
    "database.port": "3306",
    "database.user": "debezium",
    "database.password": "dbz",
    "database.server.id": "184054",
    "topic.prefix": "mysql-server",
    "table.include.list": "inventory.customers"
  }
}

Chaque événement contient before, after, op (c/u/d). Les messages sont stockés dans un topic Kafka.

Cas pratique : synchronisation entre microservices

Capture des changements d'une base relationnelle → Kafka → services consommateurs mettent à jour leurs propres stores (CQRS/Event Sourcing).

📋 Cas pratique #1 – Analyse de logs HTTP

Objectif : Compter les erreurs 404 par IP en temps réel.

Architecture :

Logs → Filebeat → Kafka → Kafka Streams → Table d'état → topic alerte

Code Java (Kafka Streams) :

KStream<String, String> logStream = builder.stream("access-logs");
KTable<String, Long> errorCount = logStream
    .filter((k, v) -> v.contains(" 404 "))
    .map((k, v) -> KeyValue.pair(extractIp(v), 1L))
    .groupByKey()
    .count();
errorCount.toStream().filter((ip, count) -> count > 10)
    .to("alert-topic");

Bonnes pratiques : Fenêtrage temporel pour éviter des alertes sur des cumuls anciens : TimeWindows.of(Duration.ofMinutes(5)).

🔄 Cas pratique #2 – Materialized Views avec Debezium + ksqlDB

Contexte : Une base PostgreSQL, on veut une vue enrichie (jointure entre customers et orders) mise à jour en temps réel.

Étapes :

  1. Démarrer Debezium PostgreSQL connector → topics dbserver1.public.customers et dbserver1.public.orders.
  2. Dans ksqlDB : créer des streams à partir de ces topics.
  3. Faire une jointure et matérialiser une table.
CREATE STREAM customers (id INT, name VARCHAR) WITH (KAFKA_TOPIC='...', VALUE_FORMAT='AVRO');
CREATE STREAM orders (order_id INT, cust_id INT, total DECIMAL) WITH (...);

CREATE TABLE enriched_orders AS
  SELECT o.order_id, c.name, o.total
  FROM orders o
  JOIN customers c ON o.cust_id = c.id
  EMIT CHANGES;

Résultat : une table Kafka qui s’enrichit en continu, consultable via requêtes push/pull.

✅ Bonnes pratiques générales

Partitionnement – Choisir une clé de partition pour l'ordre et le parallélisme.
Sérialisation – Utiliser Avro ou Protobuf avec Schema Registry.
Exactly-once – Activer les sémantiques exactly-once pour les applications critiques.
Monitoring – JMX, Prometheus, Grafana pour surveiller latence, débit, état des connecteurs.
Rétention – Configurer log.retention.hours selon les besoins de rejeu.
Gestion des erreurs – Dead Letter Queue (DLQ) pour les messages malformés.

Pour les connecteurs :

  • Toujours spécifier tasks.max pour le parallélisme.
  • Utiliser transforms pour le routage ou l’enrichissement léger.
  • Surveiller les délais de rebalancement.

🌍 Cas réels d'entreprise (Worldflik – style Netflix)

Worldflik utilise Kafka pour alimenter son moteur de recommandations, sa détection de fraudes et son monitoring.

Recommandation temps réel – Flux d’événements "visionnage" → Kafka Streams calcule les similarités entre utilisateurs → met à jour une table Redis.
Fraude aux essais gratuits – Debezium capture les inscriptions en base → ksqlDB détecte les multi-comptes (même adresse IP ou carte bancaire) → alerte Kafka.
Pipeline analytique – Kafka Connect S3 sink pour déposer les logs dans un data lake (Parquet).
Monitoring SLAs – Agrégations par fenêtre tumbling sur les temps de réponse des APIs.

Leçons tirées : scalabilité horizontale, exactly-once pour la facturation, gestion des schémas avec Schema Registry.
