📌 Introduction à Apache Kafka – Guide complet pour débutant

Apache Kafka est une plateforme de streaming distribuée open-source. Elle permet de publier, stocker, traiter et consommer des flux d'événements en temps réel, à grande échelle.

🎯 Pourquoi Kafka ? – Problèmes résolus

Avant Kafka, les applications communiquaient souvent via des API synchrones (REST, RPC) ou des files de messages classiques (RabbitMQ, ActiveMQ). Ces approches montrent leurs limites quand :

  • Le volume de données explose (millions de messages par seconde).
  • Plusieurs consommateurs doivent lire les mêmes données sans se ralentir mutuellement.
  • On a besoin de rejouer des messages historiques.
  • Il faut intégrer des données hétérogènes (logs, BDD, IoT) dans un seul pipeline.

Kafka résout ces problèmes grâce à une approche log structuré et persistant.

🏗️ Architecture de Kafka – Les composants clés

📦 Topic
C'est un flux de messages, comparable à une table de base de données ou un dossier. Chaque topic est divisé en partitions (unités de parallélisme).
✍️ Producteur (Producer)
Une application qui publie des messages sur un topic. Elle choisit la partition (par défaut round-robin ou via une clé).
📖 Consommateur (Consumer)
Lit les messages d’un topic. Il peut faire partie d’un groupe de consommateurs pour se répartir les partitions.
🖥️ Broker
Un serveur Kafka. Un cluster est composé de plusieurs brokers. Chaque partition est répliquée sur plusieurs brokers (tolérance aux pannes).
🧩 Zookeeper / KRaft
Zookeeper (ancien) gère les métadonnées du cluster. Depuis Kafka 3.x, KRaft (Kafka Raft) remplace Zookeeper pour simplifier l’opération.
🧠 Streams API
Librairie Java pour transformer des flux (filtres, agrégations, jointures) directement dans votre application.
🧬 ksqlDB
Moteur SQL pour streaming. Permet d’écrire des requêtes temps réel sans coder.
🔌 Kafka Connect
Framework pour importer/exporter des données (bases de données, S3, etc.) de manière fiable.

🔁 Flow des messages – Étape par étape

  1. Un producteur envoie un message à un topic. Il peut spécifier une clé (key) ; la même clé va toujours dans la même partition.
  2. Le message est écrit dans une partition du topic. Chaque message a un offset unique dans la partition.
  3. Le message est répliqué sur plusieurs brokers (selon le facteur de réplication). Le leader de la partition gère les écritures/lectures, les followers copient passivement.
  4. Un consommateur (d’un groupe) se connecte et lit les messages. Il commit son offset (position de lecture) pour ne pas relire les mêmes messages.
  5. Plusieurs groupes de consommateurs peuvent lire indépendamment le même topic. Kafka conserve les messages selon une politique de rétention (durée ou taille).

📈 Cas d’usage typiques de Kafka en entreprise

  • Pipeline de logs et métriques : collecte centralisée des logs d’applications → Kafka → Elasticsearch / Splunk.
  • Change Data Capture (CDC) : capture des modifications de bases de données (via Debezium) → propagation en temps réel vers des caches, data lakes, ou microservices.
  • Messagerie orientée événements : communication asynchrone entre microservices (découplage, résilience).
  • Analytique en temps réel : agrégations, fenêtres glissantes pour détection de fraudes, recommandations, monitoring IoT.
  • Event Sourcing / CQRS : stockage de l’historique complet des événements d’un domaine.

🧪 Première application Python (producteur/consommateur simple)

Installez d’abord kafka-python : pip install kafka-python.

# producer.py
from kafka import KafkaProducer
producer = KafkaProducer(bootstrap_servers='localhost:9092',
                         value_serializer=lambda v: v.encode('utf-8'))
producer.send('test', value='Hello depuis Python')
producer.flush()
print("Message envoyé")

# consumer.py
from kafka import KafkaConsumer
consumer = KafkaConsumer('test', bootstrap_servers='localhost:9092',
                         auto_offset_reset='earliest',
                         value_deserializer=lambda v: v.decode('utf-8'))
for msg in consumer:
    print(f"Reçu : {msg.value}")

📚 Concepts avancés (pour comprendre la suite)

  • Offset : position unique d’un message dans une partition. Le consommateur commit son offset pour ne pas tout relire au redémarrage.
  • Groupe de consommateurs : plusieurs consommateurs partagent la charge des partitions (chaque partition est assignée à un seul consommateur du groupe).
  • Réplication : chaque partition a un leader et des followers. Si le leader tombe, un follower devient leader (assure la haute disponibilité).
  • acks : paramètre du producteur (0, 1, all) qui contrôle la garantie de persistance.
  • Rétention : les messages ne sont pas supprimés après consommation. Kafka conserve par défaut 7 jours (configurable).

✅ Bonnes pratiques pour débuter

  • Commencez avec un seul broker en local pour apprendre, puis passez en cluster.
  • Utilisez des noms de topics explicites (ex: order.created, user.logins).
  • Préférez Avro ou Protobuf avec Schema Registry pour l’évolution des schémas.
  • Surveillez la taille des partitions (ne pas dépasser quelques Go par partition).
  • Testez la résilience en tuant un broker (si réplication > 1).

⚙️ Kafka Streams en Java – Guide complet pas à pas

Kafka Streams est une bibliothèque Java légère (pas de cluster externe) pour construire des applications de traitement de flux en temps réel. Elle s’appuie directement sur Kafka comme source et sink.

🎯 Pourquoi Kafka Streams ?

  • Intégration native : utilise les topics Kafka comme source et destination, sans système externe.
  • État local : les agrégations (compteurs, jointures) sont stockées dans des stores RocksDB intégrés, ce qui est très performant.
  • Exactly-once semantics : garantie que chaque message est traité une seule fois, même en cas de panne.
  • Fenêtrage : support des fenêtres glissantes, tumbling, hopping, session.
  • Scalabilité : parallélisme automatique via les partitions Kafka.

📦 Prérequis et configuration du projet (Maven)

Créez un projet Java avec Maven. Ajoutez la dépendance suivante dans pom.xml :

<dependency>
    <groupId>org.apache.kafka</groupId>
    <artifactId>kafka-streams</artifactId>
    <version>3.6.0</version>
</dependency>
<dependency>
    <groupId>org.slf4j</groupId>
    <artifactId>slf4j-simple</artifactId>
    <version>2.0.9</version>
</dependency>

Assurez-vous que Kafka tourne sur localhost:9092 et que le topic input-topic existe (ou sera créé automatiquement).

📝 Premier exemple : compteur de mots (WordCount) – étape par étape

Nous allons lire un topic contenant des phrases, extraire les mots, et compter leurs occurrences dans une table (KTable).

Étape 1 : Configuration des propriétés

import org.apache.kafka.streams.StreamsConfig;
import java.util.Properties;

Properties props = new Properties();
props.put(StreamsConfig.APPLICATION_ID_CONFIG, "wordcount-app"); // Identifiant unique de l'application
props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG, Serdes.String().getClass());
props.put(StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG, Serdes.String().getClass());
// Option exactly-once (depuis Kafka 3.0)
props.put(StreamsConfig.PROCESSING_GUARANTEE_CONFIG, StreamsConfig.EXACTLY_ONCE_V2);

Étape 2 : Créer le builder et le stream source

import org.apache.kafka.streams.StreamsBuilder;
import org.apache.kafka.streams.kstream.KStream;

StreamsBuilder builder = new StreamsBuilder();
KStream<String, String> textLines = builder.stream("input-topic");

Étape 3 : Transformation flatMap – éclater chaque phrase en mots

import org.apache.kafka.streams.kstream.ValueMapper;
import java.util.Arrays;

KStream<String, String> words = textLines.flatMapValues(
    line -> Arrays.asList(line.toLowerCase().split("\\W+"))
);

Étape 4 : GroupBy et comptage (agrégation avec état)

import org.apache.kafka.streams.kstream.Grouped;
import org.apache.kafka.streams.kstream.Materialized;
import org.apache.kafka.streams.state.KeyValueStore;

KTable<String, Long> wordCounts = words
    .groupBy((key, word) -> word, Grouped.with(Serdes.String(), Serdes.String()))
    .count(Materialized.<String, Long, KeyValueStore<org.apache.kafka.common.utils.Bytes>>as("counts-store"));

Étape 5 : Envoyer les résultats dans un topic de sortie

wordCounts.toStream().to("output-topic", Produced.with(Serdes.String(), Serdes.Long()));

Étape 6 : Lancer l'application

import org.apache.kafka.streams.KafkaStreams;
import org.apache.kafka.streams.Topology;

Topology topology = builder.build();
KafkaStreams streams = new KafkaStreams(topology, props);
streams.start();

// Ajoutez un hook d'arrêt gracieux
Runtime.getRuntime().addShutdownHook(new Thread(streams::close));

⏱️ Fenêtrage temporel – Compter les mots par minute

Au lieu d’un compteur global, on veut le nombre d’occurrences par fenêtre de 1 minute (tumbling window).

import org.apache.kafka.streams.kstream.TimeWindows;
import java.time.Duration;

KTable<Windowed<String>, Long> windowedCounts = words
    .groupBy((key, word) -> word)
    .windowedBy(TimeWindows.of(Duration.ofMinutes(1)))
    .count();

// Pour envoyer dans un topic, il faut un Serde pour Windowed<String>
windowedCounts.toStream().to("windowed-output", Produced.with(WindowedSerdes.timeWindowedSerdeFrom(String.class), Serdes.Long()));

🔗 Jointure entre deux flux (exemple : enrichir des clics avec des profils utilisateurs)

KStream<String, Click> clicks = builder.stream("clicks", Consumed.with(Serdes.String(), clickSerde));
KTable<String, UserProfile> profiles = builder.table("user-profiles", Consumed.with(Serdes.String(), profileSerde));
KStream<String, EnrichedClick> enriched = clicks.join(profiles,
    (click, profile) -> new EnrichedClick(click, profile),
    Joined.with(Serdes.String(), clickSerde, profileSerde));

🧠 Concepts avancés – Topologie, Stores, Serdes personnalisés

  • Topologie : graphe de nœuds (sources, processeurs, sinks). Visualisable via topology.describe().
  • State Store : stockage local (RocksDB) pour les tables. Peut être persisté sur disque et sauvegardé dans un topic changelog.
  • Serdes personnalisés : pour vos classes métier, implémentez Serializer<T> / Deserializer<T> ou utilisez Avro.
  • Transformers : accès bas niveau aux stores et horloge.

✅ Bonnes pratiques pour Kafka Streams en production

  • Définissez num.stream.threads pour utiliser plusieurs threads dans une même instance.
  • Surveillez les métriques (JMX) : taux de traitement, taille des stores, temps de commit.
  • Configurez commit.interval.ms pour équilibrer latence et performance.
  • Utilisez exactly-once pour les cas critiques (paiements, inventaire).
  • Testez la résilience : tuez une instance et vérifiez la reprise (rebalancement).
  • Ne mettez pas trop de données par clé (évitez les "hot keys").

📊 Monitoring avec Kafka Streams

Exposez les métriques via JMX, par exemple :

props.put(StreamsConfig.METRICS_RECORDING_LEVEL_CONFIG, "INFO");
// Puis activez un collecteur comme Prometheus JMX exporter.

Métriques clés : kafka.streams:type=stream-metrics (taux de messages), kafka.streams:type=processor-node (latence), kafka.streams:type=rocksdb (utilisation disque).

🔧 Dépannage fréquent

  • Erreur de sérialisation : vérifiez les Serdes (clé/valeur).
  • Rebalancement trop long : ajustez session.timeout.ms et max.poll.interval.ms.
  • State store corrompu : supprimez le répertoire local (par défaut /tmp/kafka-streams) et redémarrez.
  • Messages ignorés : activez un DeserializationExceptionHandler personnalisé.

🐍 Kafka Streams en Python avec Faust – Guide complet

Faust est une bibliothèque Python pour le traitement de flux, inspirée de Kafka Streams (Java). Elle est asynchrone (basée sur async/await) et permet de construire des applications de streaming robustes avec un état persistant.

🎯 Pourquoi Faust plutôt que Kafka Streams Java ?

  • Productivité Python : syntaxe simple, pas besoin de compiler.
  • Asynchrone par nature : idéal pour l'I/O (Kafka, bases de données, HTTP).
  • Intégration facile : avec FastAPI, SQLAlchemy, etc.
  • État persistant : utilise RocksDB localement, avec changelog dans Kafka.
  • Support des fenêtres (tumbling, hopping, session).
  • Moins performant que Java pour les très gros volumes (mais largement suffisant pour la plupart des cas).

📦 Installation et prérequis

Python 3.8+ requis. Kafka doit tourner sur localhost:9092.

# Créer un environnement virtuel (recommandé)
python -m venv faust-env
source faust-env/bin/activate  # ou .\faust-env\Scripts\activate sur Windows

# Installer Faust et kafka-python
pip install faust kafka-python

Vérifiez l'installation : faust --version.

📝 Premier exemple : compteur de mots (WordCount) – étape par étape

Créez un fichier wordcount_faust.py.

Étape 1 : Imports et configuration de l'application

import faust
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

app = faust.App(
    'wordcount-app',                    # Identifiant unique
    broker='kafka://localhost:9092',    # Serveur Kafka
    value_serializer='raw',             # Pas de sérialisation JSON
    store='rocksdb://',                 # Stockage local
    topic_partitions=3,                 # Parallélisme
    topic_replication_factor=1,
)

Étape 2 : Définir le topic source et la table de comptage

input_topic = app.topic('input-topic', value_type=str)
counts_table = app.Table('wordcount', default=int)

Étape 3 : Créer l'agent (le processeur de flux)

@app.agent(input_topic)
async def process(stream):
    """Pour chaque phrase, extrait les mots et incrémente la table."""
    async for value in stream:
        words = value.lower().split()
        for word in words:
            current = counts_table[word]
            counts_table[word] = current + 1
            logger.info(f"{word} : {counts_table[word]}")

Étape 4 : Lancer l'application

if __name__ == '__main__':
    app.main()

Pour exécuter : python wordcount_faust.py worker -l info

⏱️ Fenêtrage temporel – Compter les mots par minute

Faust supporte les fenêtres via Table avec paramètre window. Exemple : fenêtre tumbling de 60 secondes.

from faust.windows import TumblingWindow

# Table avec fenêtre de 60 secondes
counts_windowed = app.Table(
    'wordcount_windowed',
    default=int,
    window=TumblingWindow(60, 10),  # taille 60s, avancement 10s (hopping)
    help="Comptage par fenêtre"
)

@app.agent(input_topic)
async def process_windowed(stream):
    async for value in stream:
        words = value.lower().split()
        for word in words:
            # La clé devient (word, window_start)
            current = counts_windowed[word] or 0
            counts_windowed[word] = current + 1
            logger.info(f"{word} (window): {counts_windowed[word]}")

🔗 Jointure entre un stream et une table (exemple : enrichir des événements)

Supposons un topic user-events (stream) et une table user-profiles (table).

events_topic = app.topic('user-events', value_type=dict)
profiles_table = app.Table('user-profiles', default=dict)

@app.agent(events_topic)
async def enrich(stream):
    async for event in stream:
        user_id = event['user_id']
        profile = profiles_table.get(user_id, {})
        enriched = {**event, 'profile': profile}
        logger.info(f"Enriched event: {enriched}")

Pour alimenter la table, un autre agent peut lire un topic de mise à jour des profils.

🧠 Concepts avancés Faust

  • Agents multiples : vous pouvez chaîner les agents en utilisant app.agent(autre_topic).
  • Stateful vs stateless : une table rend l'agent stateful. Faust gère la restauration de l'état après redémarrage.
  • Partitionnement : les clés des tables doivent correspondre au partitionnement du topic source. Utilisez key_type et value_type dans la table.
  • Exactly-once : Faust ne le supporte pas nativement, mais avec Kafka idempotent et des commits adaptés, on peut approcher "au moins une fois". Pour exactly-once, préférez Kafka Streams Java.

✅ Bonnes pratiques pour Faust en production

  • Utilisez rocksdb:// comme store (par défaut). Évitez memory:// pour des volumes importants.
  • Configurez le nombre de partitions du topic source en fonction du parallélisme souhaité.
  • Surveillez la taille du dossier RocksDB (faust-env/var/).
  • Activez les logs détaillés (faust --debug) pour le débogage.
  • Utilisez app.conf.table_cleanup_interval pour nettoyer les vieilles entrées des fenêtres.
  • Déployez avec des process managers (systemd, supervisord) car Faust ne tourne pas en daemon par défaut.

📊 Monitoring et debugging

Faust expose une interface web sur le port 6066 (par défaut) : http://localhost:6066. Vous y verrez les agents, tables, et métriques de base.

Pour activer les métriques Prometheus :

app = faust.App(..., web_port=6066, web_bind='0.0.0.0')
# Ensuite, configurez Prometheus pour scraper /metrics

🔧 Dépannage fréquent

  • Erreur de connexion à Kafka : vérifiez que Kafka tourne et que broker est correct.
  • Table vide après redémarrage : le topic changelog interne a été supprimé. Ne supprimez pas les topics commençant par wordcount-app-.
  • Agent ne consomme pas : vérifiez les permissions des topics, et que le groupe de consommateurs n'est pas en état "rebalancing" trop longtemps.
  • Performance lente : augmentez topic_partitions et lancez plusieurs workers (chaque worker utilise un sous-ensemble des partitions).

📊 ksqlDB – Le moteur SQL pour Kafka Streaming

ksqlDB est une plateforme de traitement de flux temps réel construite sur Kafka. Elle permet d’écrire des requêtes SQL sur des flux de données, sans avoir à coder en Java ou Python. C’est un outil puissant pour l’analyse en continu, la transformation, l’agrégation et la jointure de données.

🎯 Pourquoi ksqlDB ?

  • Déclaratif : écrivez du SQL, pas du code impératif.
  • Temps réel : les résultats sont mis à jour à chaque nouvel événement.
  • Intégration native : utilise directement les topics Kafka comme sources et sinks.
  • Fenêtrage : support de fenêtres tumbling, hopping, session.
  • Jointures : entre streams, entre streams et tables, entre tables.
  • Scalabilité : s’exécute en cluster, parallélise automatiquement.
  • Cas d’usage : tableaux de bord temps réel, détection de fraudes, monitoring IoT, enrichissement de données.

📝 Concepts de base : STREAM vs TABLE

  • STREAM : flux immuable d’événements. Chaque ligne est un événement. Supporte les opérations de filtrage, projection, flatMap.
  • TABLE : vue matérialisée d’un flux (équivalent d’une KTable). Représente l’état le plus récent d’une clé. Supporte les mises à jour (UPSERT).
  • Changelog : les tables sont soutenues par des topics Kafka (compacts).

📋 Première requête – Créer un STREAM à partir d’un topic

Supposons un topic Kafka pageviews contenant des lignes au format CSV : user,page,ts.

-- Créer un stream en spécifiant le topic et le format
CREATE STREAM pageviews (
    user VARCHAR,
    page VARCHAR,
    ts BIGINT
) WITH (
    KAFKA_TOPIC = 'pageviews',
    VALUE_FORMAT = 'DELIMITED',  -- CSV (ou JSON, AVRO)
    KEY_FORMAT = 'NONE'
);

-- Afficher les 10 derniers événements (push query)
SELECT * FROM pageviews EMIT CHANGES LIMIT 10;

-- Filtre : ne garder que les vues de la page 'home'
CREATE STREAM home_views AS
    SELECT user, ts
    FROM pageviews
    WHERE page = 'home'
    EMIT CHANGES;

📊 Comptage simple – Grouper par utilisateur (Table)

-- Compter le nombre de vues par utilisateur (table mise à jour en continu)
CREATE TABLE view_counts AS
    SELECT user,
           COUNT(*) AS views
    FROM pageviews
    GROUP BY user
    EMIT CHANGES;

-- Interroger la table (pull query)
SELECT * FROM view_counts WHERE user = 'alice';

Les changements sont propagés dans un topic interne VIEW_COUNTS.

⏱️ Fenêtrage temporel – Fenêtres tumbling (par minute)

-- Fenêtre tumbling de 1 minute
CREATE TABLE views_per_minute AS
    SELECT user,
           COUNT(*) AS views,
           WINDOWSTART AS window_start,
           WINDOWEND AS window_end
    FROM pageviews
    WINDOW TUMBLING (SIZE 1 MINUTE)
    GROUP BY user
    EMIT CHANGES;

Autres types de fenêtres : HOPPING (glissante), SESSION (basée sur inactivité).

🔗 Jointures : Stream-Stream, Stream-Table, Table-Table

Exemple : enrichir un flux de clics avec une table de profils utilisateurs.

-- Table des profils (topic "user_profiles" clé=user)
CREATE TABLE user_profiles (
    user VARCHAR PRIMARY KEY,
    name VARCHAR,
    region VARCHAR
) WITH (KAFKA_TOPIC='user_profiles', VALUE_FORMAT='JSON');

-- Jointure entre le stream pageviews et la table
CREATE STREAM enriched_views AS
    SELECT pv.user, up.name, up.region, pv.page, pv.ts
    FROM pageviews pv
    LEFT JOIN user_profiles up ON pv.user = up.user
    EMIT CHANGES;

🔄 Insertion dans un nouveau topic (sink)

Par défaut, chaque CREATE STREAM ... AS ou CREATE TABLE ... AS crée un topic de sortie. Vous pouvez spécifier le nom du topic :

CREATE STREAM enriched_views
    WITH (KAFKA_TOPIC='enriched-pageviews', VALUE_FORMAT='JSON') AS
    SELECT ... ;

🧠 Fonctions avancées et UDF

  • Fonctions scalaires : UCASE, LCASE, CONCAT, SUBSTRING, EXTRACTJSONFIELD, etc.
  • Fonctions d’agrégation : COUNT, SUM, AVG, TOPKDISTINCT, EARLIEST_BY_OFFSET.
  • UDF : vous pouvez écrire vos propres fonctions en Java (ou via le SDK ksqlDB).

📡 Push queries vs Pull queries

  • Push query : SELECT ... EMIT CHANGES → flux continu de résultats (comme une souscription).
  • Pull query : SELECT ... WHERE key = ... → requête ponctuelle sur une table (state lookup).

Exemple : pour obtenir la vue actuelle d’un utilisateur, utilisez une pull query.

✅ Bonnes pratiques pour ksqlDB en production

  • Définissez des propriétés de streaming : SET 'auto.offset.reset' = 'earliest';
  • Utilisez Avro ou JSON Schema avec Schema Registry pour gérer l’évolution des données.
  • Surveillez les performances avec les métriques JMX de ksqlDB.
  • Pour les grandes tables, utilisez le tombstone pour supprimer des clés (envoyer null).
  • Séparez les requêtes longues et les requêtes courtes (différents clusters ksqlDB si besoin).
  • Activez la persistance des requêtes pour qu’elles redémarrent automatiquement après un crash.

📊 Monitoring et gestion des erreurs

  • Commandes utiles : SHOW STREAMS;, SHOW TABLES;, SHOW QUERIES;, DESCRIBE <nom>;
  • Terminer une requête : TERMINATE <query_id>;
  • Visualiser le plan d’exécution : EXPLAIN <query_id>;
  • Surveiller les erreurs de désérialisation : configurer ksql.streams.processing.guarantee=exactly_once_v2 et des deserialization.exception.handler.

🔧 Dépannage fréquent

  • Erreur "Topic does not exist" : ksqlDB peut auto-créer les topics si la propriété ksql.topic.auto.create est vraie (par défaut oui). Sinon, créez-les manuellement.
  • Requête qui ne produit rien : vérifiez que le topic source a bien des données et que le format de valeur correspond (JSON, DELIMITED, etc.).
  • Jointure qui ne retourne rien : assurez-vous que les clés de jointure sont correctement définies (dans un stream, la clé est ROWKEY).
  • Performance lente : augmentez le nombre de threads (ksql.streams.num.stream.threads) et le nombre de partitions des topics sources.

📦 Exemple concret : Détection de clics suspects (fraude)

Objectif : déclencher une alerte si un même utilisateur clique plus de 10 fois en 5 secondes.

-- Fenêtre hopping de 5 secondes
CREATE TABLE suspicious_clicks AS
    SELECT user,
           COUNT(*) AS clicks
    FROM pageviews
    WINDOW HOPPING (SIZE 5 SECONDS, ADVANCE BY 1 SECOND)
    GROUP BY user
    HAVING COUNT(*) > 10
    EMIT CHANGES;

-- Créer un stream d'alertes vers un topic 'alerts'
CREATE STREAM alerts AS
    SELECT user, clicks, WINDOWSTART, WINDOWEND
    FROM suspicious_clicks
    EMIT CHANGES;

🔌 Kafka Connect – Intégration de données sans code

Kafka Connect est un framework intégré à Apache Kafka pour importer (source) et exporter (sink) des données entre Kafka et d’autres systèmes (bases de données, fichiers, cloud, etc.). Il permet de construire des pipelines de données fiables et scalables sans écrire une ligne de code.

🎯 Pourquoi Kafka Connect ?

  • Simplifie l’intégration : plus besoin d’écrire des producteurs/consommateurs personnalisés.
  • Scalabilité : les connecteurs s’exécutent dans des workers, parallélisables via les tâches (tasks).
  • Fault-tolerance : reprise automatique, rééquilibrage des tâches en cas de panne.
  • Transformations (SMT) : modifiez les données à la volée (filtrage, masquage, routage).
  • Écosystème riche : des centaines de connecteurs prêts à l’emploi (JDBC, S3, Elasticsearch, MongoDB, Debezium, etc.).

🏗️ Architecture de Kafka Connect

  • Worker : processus JVM qui exécute les connecteurs et leurs tâches. Peut fonctionner en mode standalone (un seul worker) ou distribué (cluster).
  • Connector : instance d’un connecteur (ex: jdbc-source). Chaque connector a une configuration.
  • Task : unité de travail d’un connector. Un connector peut avoir plusieurs tâches pour paralléliser le traitement.
  • Converter : sérialise/désérialise les données (JSON, Avro, Protobuf).
  • Transformation (SMT) : étape de manipulation des messages (ex: renommer un champ, ajouter un timestamp).
  • Offset : stocke la position de lecture (source) ou d’écriture (sink) pour éviter de traiter deux fois les données.

📦 Installation et démarrage (standalone) – Pas à pas

Kafka Connect est inclus dans la distribution Kafka. Après avoir démarré Kafka et Zookeeper (ou KRaft) :

# Démarrer Kafka Connect en mode standalone (avec connecteur file source)
bin/connect-standalone.sh config/connect-standalone.properties config/connect-file-source.properties config/connect-file-sink.properties

Le fichier connect-standalone.properties contient la configuration de base (brokers, conversion, etc.). Exemple minimal :

bootstrap.servers=localhost:9092
key.converter=org.apache.kafka.connect.json.JsonConverter
value.converter=org.apache.kafka.connect.json.JsonConverter
key.converter.schemas.enable=false
value.converter.schemas.enable=false
offset.storage.file.filename=/tmp/connect.offsets

Le fichier connect-file-source.properties :

name=file-source
connector.class=FileStreamSource
tasks.max=1
file=/tmp/test.txt
topic=connect-test

Placez des lignes dans /tmp/test.txt et consommez le topic connect-test pour les voir.

🌐 Mode distribué (pour la production)

Le mode distribué permet d’exécuter plusieurs workers, d’équilibrer la charge, d’assurer la tolérance aux pannes.

# Démarrer un premier worker
bin/connect-distributed.sh config/connect-distributed.properties
# Démarrer un second worker (même configuration, sur un autre port)
bin/connect-distributed.sh config/connect-distributed-2.properties

Configuration typique (connect-distributed.properties) :

bootstrap.servers=localhost:9092
group.id=connect-cluster
key.converter=org.apache.kafka.connect.json.JsonConverter
value.converter=org.apache.kafka.connect.json.JsonConverter
config.storage.topic=connect-configs
offset.storage.topic=connect-offsets
status.storage.topic=connect-status
offset.storage.partitions=25
status.storage.partitions=5

Les topics de configuration, offsets et status sont créés automatiquement (avec facteur de réplication par défaut 3).

🔄 Single Message Transforms (SMT) – Modifier les données à la volée

Les SMT permettent de transformer chaque message avant de l’écrire dans Kafka (source) ou avant de l’envoyer au système externe (sink).

{
  "config": {
    "transforms": "renameField,addPrefix",
    "transforms.renameField.type": "org.apache.kafka.connect.transforms.ReplaceField$Value",
    "transforms.renameField.renames": "oldName:newName",
    "transforms.addPrefix.type": "org.apache.kafka.connect.transforms.InsertField$Value",
    "transforms.addPrefix.topic.field": "source_topic",
    "transforms.addPrefix.static.field": "pipeline_id",
    "transforms.addPrefix.static.value": "my_pipeline"
  }
}

SMT courants : RegexRouter (renommer topic), TimestampRouter, MaskField, DropHeaders, InsertField, ValueToKey, ExtractKey.

✅ Bonnes pratiques pour Kafka Connect en production

  • Utilisez le mode distribué avec au moins 2 workers pour la tolérance aux pannes.
  • Définissez un tasks.max approprié : généralement égal au nombre de partitions du topic source (pour les sinks) ou au nombre de tables/partitions (pour les sources).
  • Surveillez les métriques : taux de messages, temps de traitement, état des tâches, erreurs de conversion.
  • Utilisez des convertisseurs Avro avec Schema Registry pour garantir l’évolution des schémas.
  • Activez la journalisation des erreurs : errors.tolerance=all ou errors.deadletterqueue.topic.name pour rediriger les messages problématiques.
  • Testez la reprise après panne : tuez un worker et vérifiez le rééquilibrage.
  • Configurez les délais d’expiration (consumer.max.poll.interval.ms) pour éviter les rebalances intempestives.
  • Utilisez offset.flush.interval.ms pour contrôler la fréquence de commit des offsets.

📊 Monitoring et métriques

Kafka Connect expose des métriques JMX. Les plus importantes :

  • kafka.connect:type=connector-metrics,connector="{name}" : nombre d’enregistrements traités, durée moyenne des tâches.
  • kafka.connect:type=source-task-metrics,task="{id}" : débit source, temps de poll.
  • kafka.connect:type=sink-task-metrics,task="{id}" : débit sink, temps d’écriture.
  • kafka.consumer:type=consumer-fetch-manager-metrics (pour les sinks) : détail sur la consommation.

Vous pouvez aussi utiliser l’API REST /connectors/{name}/status et /connectors/{name}/tasks/{id}/status.

🔧 Dépannage fréquent

  • Le connecteur reste en state "RUNNING" mais aucune donnée : vérifiez les logs du worker. Souvent un problème de permissions, de requête SQL, ou de topic inexistant.
  • Erreur de conversion de schéma : désactivez schemas.enable ou utilisez Avro.
  • Tasks échouent avec "Task is being killed" : augmentez task.timeout.ms ou réduisez le nombre de tâches.
  • Rebalancement perpétuel : vérifiez les délais de session (session.timeout.ms).
  • Topic de configuration introuvable : créez-le manuellement avec --config cleanup.policy=compact.
  • Conflit de versions de connecteurs : utilisez le répertoire plugin.path séparément pour chaque connecteur.

🔄 Debezium – Change Data Capture (CDC) pour Kafka

Debezium est une plateforme open-source de Change Data Capture (CDC) basée sur Kafka Connect. Elle capture les modifications (insert, update, delete) dans les bases de données (MySQL, PostgreSQL, MongoDB, SQL Server, etc.) et les publie en temps réel dans des topics Kafka.

🎯 Pourquoi Debezium ?

  • Capture en temps réel : chaque changement est détecté immédiatement, sans polling.
  • Non intrusif : utilise les journaux de transaction (binlog, WAL, oplog) de la base de données.
  • Exactly-once : grâce à Kafka Connect et aux offsets, chaque événement est livré une seule fois.
  • Structure riche : chaque événement contient les valeurs avant/après, l’opération, les métadonnées.
  • Cas d’usage : synchronisation de microservices, mise à jour de caches, analyse en temps réel, event sourcing, data lake.

🏗️ Architecture Debezium

  • Connector Debezium : un connecteur source Kafka Connect spécifique à une base de données.
  • Kafka Connect Worker : exécute le connecteur et ses tâches (distribué recommandé).
  • Topic Kafka : chaque table capturée a son propre topic (ou un topic par base).
  • Database log : Debezium se connecte au binlog (MySQL), WAL (PostgreSQL), etc. pour lire les changements.
  • Snapshot initial : Debezium peut d’abord prendre un snapshot de l’état existant de la table avant de capturer les changements en continu.

📦 Prérequis et installation

Debezium est distribué sous forme de connecteurs Kafka Connect. La manière la plus simple est d’utiliser Docker.

📊 Structure d’un événement Debezium

Chaque message Kafka contient une clé (généralement la clé primaire de la ligne) et une valeur complexe :

  • before : l’état de la ligne avant la modification (null pour insert).
  • after : l’état après modification (null pour delete).
  • source : métadonnées (nom de la base, table, timestamp, position dans le log, etc.).
  • op : type d’opération : c (create), u (update), d (delete), r (read, lors du snapshot initial).
  • ts_ms : timestamp de l’événement dans Debezium.

🔧 Configuration avancée de Debezium

  • snapshot.mode : initial (prend un snapshot puis CDC), never (pas de snapshot), when_needed, etc.
  • table.include.list / table.exclude.list : filtrer les tables à capturer.
  • column.include.list : capturer uniquement certaines colonnes.
  • decimal.handling.mode : comment sérialiser les décimaux (string, double, precise).
  • tombstones.on.delete : envoyer un message tombstone (valeur null) après un delete pour indiquer la suppression dans les tables compactées.
  • key.converter / value.converter : utiliser Avro avec Schema Registry pour une meilleure gestion des schémas.

Exemple de configuration enrichie :

{

📋 Cas pratique #1 – Analyse de logs HTTP en temps réel

Ce cas pratique vous guide dans la construction d’un pipeline de traitement de logs (par exemple, les accès à un serveur web Nginx/Apache) pour détecter les erreurs, compter les requêtes par IP, et déclencher des alertes en temps réel – le tout avec Kafka, Kafka Streams (Java) et ksqlDB.

🎯 Objectifs pédagogiques

  • Comprendre comment ingérer des logs dans Kafka (producteur).
  • Appliquer des transformations de flux (filtrage, extraction, comptage).
  • Utiliser les fenêtres temporelles pour des agrégations glissantes.
  • Détecter des comportements suspects (ex: IP avec trop d’erreurs 404).
  • Produire des alertes dans un topic dédié.

🏗️ Architecture du pipeline

Nous allons simuler des logs d’accès HTTP et les traiter en temps réel.

Source de logs (fichier ou générateur) → Producteur Kafka → Topic "raw-logs"
→ Kafka Streams / ksqlDB (traitement) → Topic "alerts" → Consommateur d’alertes

Composants :

  • Producteur : envoie chaque ligne de log dans Kafka.
  • Topic raw-logs : stocke les logs bruts (format texte).
  • Application de streaming : lit, parse, filtre, agrège.
  • Topic alerts : reçoit les événements suspects (ex: IP avec >10 erreurs 404 en 5 minutes).
  • Consommateur d’alertes : affiche ou envoie des notifications.

📦 Étape 1 – Génération de logs simulés (producteur)

Créez un script Python qui produit des lignes de log au format Common Log Format (CLF) dans Kafka.

📝 Étape 2 – Traitement avec Kafka Streams (Java)

Nous allons implémenter un pipeline Kafka Streams qui parse les logs, filtre les erreurs 4xx/5xx, et compte par IP sur une fenêtre glissante de 5 minutes, puis alerte si le seuil dépasse 10.

Dépendances Maven

Code complet

import org.apache.kafka.streams.*;
import org.apache.kafka.streams.kstream.*;
import java.time.Duration;
import java.util.Properties;
import java.util.regex.Matcher;
import java.util.regex.Pattern;

public class LogAnalysisStream {
    public static void main(String[] args) {
        Properties props = new Properties();
        props.put(StreamsConfig.APPLICATION_ID_CONFIG, "log-analyzer");
        props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG, Serdes.String().getClass());
        props.put(StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG, Serdes.String().getClass());

        StreamsBuilder builder = new StreamsBuilder();
        KStream<String, String> rawLogs = builder.stream("raw-logs");

        // 1. Parse la ligne de log pour extraire IP, status, etc.
        Pattern logPattern = Pattern.compile("^(\\S+) .+ \\[(.+)\\] \"\\w+ (\\S+) .+\" (\\d+) \\d+$");
        KStream<String, LogEntry> parsed = rawLogs
            .filter((k, v) -> v != null && !v.isEmpty())
            .mapValues(value -> {
                Matcher m = logPattern.matcher(value);
                if (m.find()) {
                    return new LogEntry(m.group(1), m.group(2), m.group(3), Integer.parseInt(m.group(4)));
                }
                return null;
            })
            .filter((k, v) -> v != null);

        // 2. Ne garder que les erreurs (status >= 400)
        KStream<String, LogEntry> errors = parsed.filter((k, v) -> v.status >= 400);

        // 3. Compter les erreurs par IP sur une fenêtre tumbling de 5 minutes
        KTable<Windowed<String>, Long> errorCounts = errors
            .groupBy((k, v) -> v.ip, Grouped.with(Serdes.String(), customSerde(LogEntry.class)))
            .windowedBy(TimeWindows.of(Duration.ofMinutes(5)))
            .count();

        // 4. Alerter si le compteur dépasse 10 (ex: IP suspecte)
        KStream<String, String> alerts = errorCounts
            .toStream()
            .filter((windowedIp, count) -> count != null && count > 10)
            .map((windowedIp, count) -> {
                String ip = windowedIp.key();
                long start = windowedIp.window().start();
                long end = windowedIp.window().end();
                String alert = String.format("IP %s a eu %d erreurs entre %d et %d", ip, count, start, end);
                return KeyValue.pair(ip, alert);
            });

        alerts.to("alerts", Produced.with(Serdes.String(), Serdes.String()));

        KafkaStreams streams = new KafkaStreams(builder.build(), props);
        streams.start();

        Runtime.getRuntime().addShutdownHook(new Thread(streams::close));
    }
}

// Classe LogEntry et sérialiseur simplifié (à adapter)
class LogEntry {
    String ip, timestamp, url;
    int status;
    LogEntry(String ip, String ts, String url, int status) { this.ip = ip; this.timestamp = ts; this.url = url; this.status = status; }
}

📊 Étape 2bis – Version plus simple avec ksqlDB

Si vous préférez SQL, voici la même logique avec ksqlDB.

-- Créer un stream à partir du topic raw-logs
CREATE STREAM raw_logs (log_line VARCHAR) WITH (KAFKA_TOPIC='raw-logs', VALUE_FORMAT='DELIMITED');

-- Parser la ligne et extraire IP et status (expression régulière)
CREATE STREAM parsed_logs AS
  SELECT
    REGEXP_EXTRACT(log_line, '^(\\S+)', 1) AS ip,
    CAST(REGEXP_EXTRACT(log_line, '\\s(\\d{3})\\s', 1) AS INT) AS status
  FROM raw_logs
  WHERE log_line IS NOT NULL
  EMIT CHANGES;

-- Filtrer les erreurs (status >= 400)
CREATE STREAM error_logs AS
  SELECT ip, status
  FROM parsed_logs
  WHERE status >= 400
  EMIT CHANGES;

-- Fenêtre tumbling de 5 minutes, compter les erreurs par IP
CREATE TABLE error_counts AS
  SELECT ip,
         COUNT(*) AS errors,
         WINDOWSTART AS window_start,
         WINDOWEND AS window_end
  FROM error_logs
  WINDOW TUMBLING (SIZE 5 MINUTES)
  GROUP BY ip
  EMIT CHANGES;

-- Alerter si le nombre d'erreurs dépasse 10
CREATE STREAM alerts AS
  SELECT ip, errors, window_start, window_end
  FROM error_counts
  WHERE errors > 10
  EMIT CHANGES;

Les alertes sont alors écrites dans le topic ALERTS (par défaut).

                         value_deserializer=lambda v: v.decode('utf-8'))
for msg in consumer:
    print(f"[ALERTE] {msg.value}")

🔍 Améliorations possibles

  • Enrichissement : joindre avec une table d’IPs connues (malveillantes, internes).
  • Fenêtre hopping : au lieu de tumbling, une fenêtre glissante qui se déclenche toutes les minutes.
  • Plusieurs seuils : alerte critique si >50 erreurs, warning si >10.
  • Stockage des métriques : envoyer les compteurs dans une base de données (via Kafka Connect JDBC Sink).
  • Tableau de bord temps réel : utiliser un sink Elasticsearch + Kibana pour visualiser les erreurs par IP.

✅ Bonnes pratiques pour ce cas pratique

  • Utilisez Avro plutôt que du texte brut pour éviter de re-parser à chaque fois.
  • Gérez les erreurs de parsing via une Dead Letter Queue (topic séparé pour les logs mal formés).
  • Surveillez la latence du pipeline avec les métriques Kafka Streams ou ksqlDB.
  • Configurez la rétention du topic raw-logs à quelques jours pour économiser de l’espace.
  • Pour des volumes très importants, augmentez le nombre de partitions des topics.

Définissez les streams à partir des topics Debezium. Notez que les messages sont au format JSON avec une structure before/after.

-- Définir un stream pour les customers (nous ne prenons que le payload.after)
CREATE STREAM customers_raw (before STRUCT<...>, after STRUCT<id INT, first_name VARCHAR, last_name VARCHAR, email VARCHAR, created_at VARCHAR>, op VARCHAR)
WITH (KAFKA_TOPIC='dbserver1.public.customers', VALUE_FORMAT='JSON');

-- Créer un stream propre avec seulement les données après modification (insert/update)
CREATE STREAM customers AS
  SELECT after->id AS id,
         after->first_name AS first_name,
         after->last_name AS last_name,
         after->email AS email,
         after->created_at AS created_at
  FROM customers_raw
  WHERE op IN ('c', 'u')
  EMIT CHANGES;

-- Créer une table matérialisée des customers (état le plus récent)
CREATE TABLE customers_table AS
  SELECT id,
         LATEST_BY_OFFSET(first_name) AS first_name,
         LATEST_BY_OFFSET(last_name) AS last_name,
         LATEST_BY_OFFSET(email) AS email
  FROM customers
  GROUP BY id
  EMIT CHANGES;

Même chose pour les commandes (orders) :

CREATE STREAM orders_raw (before STRUCT<...>, after STRUCT<id INT, customer_id INT, product VARCHAR, quantity INT, price DECIMAL(10,2), order_date VARCHAR>, op VARCHAR)
WITH (KAFKA_TOPIC='dbserver1.public.orders', VALUE_FORMAT='JSON');

CREATE STREAM orders AS
  SELECT after->id AS order_id,
         after->customer_id AS customer_id,
         after->product AS product,
         after->quantity AS quantity,
         after->price AS price,
         after->order_date AS order_date
  FROM orders_raw
  WHERE op IN ('c', 'u')
  EMIT CHANGES;

CREATE TABLE orders_table AS
  SELECT order_id,
         LATEST_BY_OFFSET(customer_id) AS customer_id,
         LATEST_BY_OFFSET(product) AS product,
         LATEST_BY_OFFSET(quantity) AS quantity,
         LATEST_BY_OFFSET(price) AS price
  FROM orders
  GROUP BY order_id
  EMIT CHANGES;

🔗 Étape 4 – Créer une vue matérialisée : jointure customers + orders

Nous voulons une table qui pour chaque commande affiche les informations du client (dénormalisation).

CREATE TABLE customer_orders AS
  SELECT o.order_id,
         o.customer_id,
         c.first_name,
         c.last_name,
         o.product,
         o.quantity,
         o.price
  FROM orders_table o
  LEFT JOIN customers_table c ON o.customer_id = c.id
  EMIT CHANGES;

Cette table se met à jour en temps réel : si un client change son nom, la jointure reflète immédiatement le changement. Vous pouvez interroger cette table :

CREATE TABLE customer_orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    first_name VARCHAR(100),
    last_name VARCHAR(100),
    product VARCHAR(200),
    quantity INT,
    price DECIMAL(10,2)
);

Ensuite, installez le connecteur JDBC Sink (si non présent). Pour Debezium Connect, ajoutez le plugin. Avec notre image debezium/connect:latest, il n’est pas inclus par défaut. Vous pouvez utiliser l’image confluentinc/cp-kafka-connect:latest avec les plugins appropriés. Pour simplifier, nous allons utiliser une approche alternative : créer un stream dans ksqlDB qui écrit dans un topic, puis un connecteur sink. Ou bien utiliser ksqlDB directement pour écrire dans une base via un connecteur intégré (non disponible). Je vais plutôt montrer la configuration d’un connecteur JDBC sink séparé.

Créez d’abord un topic à partir de la table ksqlDB :

-- Dans ksqlDB, créer un stream qui réplique la table (pour la rendre disponible en tant que topic)
CREATE STREAM customer_orders_stream AS
  SELECT * FROM customer_orders
  EMIT CHANGES;

Ce stream écrit dans un topic CUSTOMER_ORDERS_STREAM. Maintenant, configurez un connecteur JDBC sink (vous aurez besoin de télécharger le connecteur JDBC et de le placer dans le plugin.path du worker). Exemple de configuration :

curl -X POST -H "Content-Type: application/json" --data '{
  "name": "jdbc-sink",
  "config": {
    "connector.class": "io.confluent.connect.jdbc.JdbcSinkConnector",
    "connection.url": "jdbc:postgresql://postgres-target:5432/ecommerce_target",
    "connection.user": "postgres",
    "connection.password": "postgres",
    "topics": "CUSTOMER_ORDERS_STREAM",
    "table.name.format": "customer_orders",
    "insert.mode": "upsert",
    "pk.mode": "record_key",
    "pk.fields": "order_id",
    "auto.create": false,
    "auto.evolve": false
  }
}' http://localhost:8083/connectors

Désormais, chaque insertion ou modification dans la base source (customers ou orders) se propage instantanément vers la base cible.

🔄 Gestion des suppressions (tombstones)

Pour gérer les suppressions, vous pouvez configurer Debezium pour envoyer des messages tombstone (valeur null). Dans la table ksqlDB, les suppressions apparaîtront comme op='d'. Vous devez alors propager la suppression au sink. C’est plus avancé, mais possible.

✅ Bonnes pratiques pour ce cas pratique

  • Utilisez Avro plutôt que JSON pour les événements Debezium, afin de bénéficier de Schema Registry.
  • Surveillez le lag des connecteurs Debezium et JDBC Sink.
  • Testez la résilience : arrêtez et redémarrez les workers Connect, vérifiez que la synchronisation reprend sans perte.
  • Filtrez les colonnes sensibles dans Debezium (ex: email, mots de passe) via column.exclude.list.
  • Pour de très gros volumes, utilisez des partitions multiples sur les topics sources (Debezium respecte la clé primaire).
  • Activez le compactage des topics (cleanup.policy=compact) pour les tables ksqlDB, afin d’éviter l’accumulation infinie.

📊 Monitoring du pipeline

  • curl localhost:8083/connectors/postgres-connector/status et /connectors/jdbc-sink/status.
  • Dans ksqlDB : SHOW STREAMS; SHOW TABLES; SHOW QUERIES;.
  • Surveillez la taille des topics dbserver1.public.customers et autres.

🔧 Dépannage fréquent

  • Le connecteur Debezium ne capture pas les changements : vérifiez que wal_level=logical est actif dans PostgreSQL.
  • Les événements n’apparaissent pas dans ksqlDB : assurez-vous que les streams sont créés avec le bon format et la bonne structure (consultez un message brut pour voir la structure exacte).
  • La jointure ne fonctionne pas : vérifiez que les clés de jointure sont correctement typées et que les streams/tables sont correctement partitionnées (clé primaire).
  • Le sink JDBC échoue avec des erreurs de duplication : configurez insert.mode=upsert et une clé primaire.

✅ Bonnes pratiques pour l’écosystème Kafka

Cette section rassemble les meilleures pratiques pour concevoir, développer, déployer et maintenir des applications Kafka robustes, performantes et sécurisées. Ces recommandations s’appliquent aux producteurs, consommateurs, Kafka Streams, ksqlDB, Kafka Connect, Debezium, et à l’administration du cluster.

📦 1. Producteurs Kafka

  • Utilisez des clés de partition pertinentes : une clé détermine la partition. Pour garantir l’ordre des messages d’une même entité (ex: commande), choisissez une clé métier (ex: order_id).
  • Activez l’idempotence : enable.idempotence=true évite les doublons en cas de retry.
  • Configurez acks=all pour garantir la persistance sur tous les réplicas (sauf si la latence est critique).
  • Gérez les retries avec backoff : retries=Integer.MAX_VALUE et retry.backoff.ms=100 pour une reprise progressive.
  • Utilisez des sérialiseurs performants : Avro, Protobuf ou JSON Schema avec Schema Registry plutôt que JSON brut (sans schéma).
  • Compressez les messages : compression.type=snappy ou lz4 pour réduire la bande passante et le stockage.
  • Évitez les messages trop gros : la limite par défaut est 1 Mo. Pour des messages plus gros, utilisez un référentiel externe (S3, HDFS) et envoyez l’URI.
  • Surveillez les métriques : request-latency, record-error-rate, bytes-out.

📖 2. Consommateurs Kafka

  • Choisissez le bon auto.offset.reset : earliest pour relire depuis le début, latest pour ne lire que les nouveaux messages.
  • Utilisez le commit manuel des offsets (enable.auto.commit=false) pour contrôler exactement quand marquer un message comme traité.
  • Gérez le rebalancement : implémentez ConsumerRebalanceListener pour sauvegarder l’état avant la perte de partitions.
  • Augmentez max.poll.interval.ms si le traitement de chaque lot peut prendre du temps (sinon, le consommateur sera considéré comme mort).
  • Limitez la taille des lots avec max.poll.records pour éviter de surcharger la mémoire.
  • Utilisez des groupes de consommateurs pour paralléliser la consommation : le nombre de consommateurs dans un groupe ne doit pas dépasser le nombre de partitions.
  • Surveillez le consumer lag : écart entre le dernier message produit et le dernier message consommé. Outils : kafka-consumer-groups, JMX, Burrow.

⚙️ 3. Kafka Streams (Java)

  • Définissez un application.id unique et cohérent : il sert à identifier l’application dans le cluster et pour le stockage d’état.
  • Utilisez exactly-once pour les transactions critiques : processing.guarantee=exactly_once_v2.
  • Choisissez le nombre de threads (num.stream.threads) égal au nombre de partitions pour un parallélisme maximal sur une seule instance.
  • Surveillez la taille des stores RocksDB : les stores peuvent devenir très gros. Configurez rocksdb.block.cache.size et rocksdb.write.buffer.size.
  • Utilisez des window stores avec une rétention adaptée : windowstore.changelog.additional.retention.ms pour éviter une accumulation infinie.
  • Testez la topologie avec TopologyTestDriver pour valider le comportement sans déployer de cluster.
  • Gérez les erreurs de désérialisation : implémentez DeserializationExceptionHandler pour envoyer les mauvais messages vers une Dead Letter Queue.
  • Activez la métrologie JMX pour surveiller le débit, la latence et l’état des stores.

🐍 4. Faust (Python)

  • Utilisez rocksdb:// comme store (par défaut). Évitez memory:// en production.
  • Configurez topic_partitions sur l’application pour correspondre aux partitions du topic source.
  • Surveillez la mémoire : Faust étant asynchrone, les agents peuvent accumuler des messages non traités. Utilisez app.conf.stream_buffer_max_size.
  • Utilisez les timers avec parcimonie : trop de timers peuvent bloquer le event loop.
  • Déployez plusieurs workers pour la haute disponibilité, mais assurez-vous que chaque worker ait une configuration web_port différente.
  • Activez les logs détaillés : faust --debug pour le débogage, mais désactivez en production.

📊 5. ksqlDB

  • Utilisez des clés primaires dans les tables : définissez PRIMARY KEY pour permettre les mises à jour et les suppressions.
  • Évitez les requêtes EMIT CHANGES non filtrées sur de gros volumes : elles peuvent submerger le client.
  • Préférez les pull queries pour les requêtes ponctuelles plutôt que les push queries.
  • Surveillez les requêtes actives : SHOW QUERIES; et TERMINATE <query_id>; si nécessaire.
  • Activez le schéma Avro avec Schema Registry pour garantir l’évolution.
  • Définissez des propriétés de streaming appropriées : auto.offset.reset=earliest ou latest selon le besoin.

🔌 6. Kafka Connect

  • Utilisez le mode distribué en production avec au moins 2 workers pour la tolérance aux pannes.
  • Configurez tasks.max en fonction du parallélisme souhaité : pour un connecteur source JDBC, tasks.max peut être le nombre de tables ou de partitions.
  • Utilisez les transformations SMT avec modération : elles s’exécutent dans le worker et peuvent devenir un goulot d’étranglement.
  • Activez la Dead Letter Queue : errors.tolerance=all et errors.deadletterqueue.topic.name pour rediriger les messages problématiques.
  • Surveillez l’état des connecteurs et des tâches via l’API REST et les métriques JMX.
  • Séparez les connecteurs par répertoire plugin.path pour éviter les conflits de version de dépendances.

🔄 7. Debezium (CDC)

  • Assurez-vous que la base source conserve les logs assez longtemps : MySQL binlog retention, PostgreSQL wal_keep_size.
  • Utilisez snapshot.mode=initial pour la première exécution, puis never si vous recréez le connecteur.
  • Filtrez les colonnes sensibles avec column.exclude.list ou masquez via SMT.
  • Surveillez la latence : Debezium ajoute un champ source.ts_ms. Comparez-le avec ts_ms pour détecter les retards.
  • Activez les tombstones pour les suppressions : tombstones.on.delete=true pour que les tables compactées reflètent la suppression.

🔐 8. Sécurité

  • Activez SSL/TLS pour le chiffrement des communications entre clients et brokers, et entre brokers.
  • Utilisez SASL (SCRAM, PLAIN, Kerberos) pour l’authentification des clients.
  • Appliquez des ACLs : contrôlez qui peut lire, écrire, créer des topics, etc.
  • Séparez les environnements (développement, test, production) avec des clusters distincts ou des namespaces.
  • Gérez les secrets avec des providers externes (Vault, AWS Secrets Manager) plutôt qu’en clair dans les fichiers de configuration.
  • Auditez les accès : activez les logs d’autorisation (authorizer.logger).

📈 9. Monitoring et alerting

  • Exposez les métriques JMX et utilisez Prometheus + Grafana.
  • Surveillez les métriques clés :
    • Pour les brokers : kafka.server:type=BrokerTopicMetrics (messages in/out), kafka.controller:type=ControllerStats (état du contrôleur).
    • Pour les consommateurs : consumer lag (via kafka-consumer-groups).
    • Pour Kafka Streams : taux de traitement, taille des stores, temps de commit.
    • Pour Connect : nombre d’enregistrements traités, tâches en échec.
  • Configurez des alertes : lag > seuil, broker down, tâches Connect défaillantes, erreurs de désérialisation.
  • Utilisez des outils comme Burrow, Cruise Control, Confluent Control Center pour la surveillance avancée.

🚀 10. Administration du cluster

  • Dimensionnez correctement le nombre de partitions : trop de partitions augmente la latence et la mémoire, trop peu limite le parallélisme. Une règle : 2-3 partitions par broker pour commencer.
  • Configurez la rétention des données : log.retention.hours ou log.retention.bytes. Évitez la rétention infinie.
  • Activez la compaction pour les topics de type changelog (ex: tables Kafka Streams) : cleanup.policy=compact.
  • Surveillez l’utilisation du disque : les partitions peuvent grossir rapidement.
  • Planifiez la répartition des partitions : utilisez kafka-reassign-partitions pour équilibrer la charge.
  • Mettez à jour Kafka progressivement : testez d’abord sur un environnement de staging, puis procédez broker par broker avec rolling upgrade.

🧪 11. Tests et validation

  • Testez unitairement les topologies Kafka Streams avec TopologyTestDriver.
  • Utilisez des conteneurs Docker (Testcontainers) pour les tests d’intégration avec un vrai broker Kafka.
  • Simulez des pannes : tuez des brokers, des workers Connect, des consommateurs, et vérifiez la reprise.
  • Vérifiez la sémantique exactly-once en injectant des erreurs et en redémarrant les applications.

📝 12. Documentation et gestion des schémas

  • Utilisez Schema Registry pour stocker et versionner les schémas Avro/Protobuf/JSON.
  • Adoptez des conventions de nommage des topics : <environnement>.<domaine>.<entité>.<type> (ex: prod.ecommerce.orders.v1).
  • Documentez les flux de données : quels producteurs écrivent dans quels topics, quels consommateurs lisent, quels schémas.
  • Versionnez vos configurations (Connect, Streams) et utilisez Git.

🎓 Récapitulatif des erreurs courantes à éviter

  • Négliger le dimensionnement des partitions → déséquilibre ou limitation de débit.
  • Ne pas surveiller le lag consommateur → données périmées.
  • Utiliser le commit automatique sans comprendre les risques de perte ou double traitement.
  • Ignorer les erreurs de désérialisation → blocage silencieux du pipeline.
  • Configurer une rétention trop courte pour les topics de changelog → perte d’état après redémarrage.
  • Ne pas sécuriser le cluster en production → risques de fuite de données.

🌍 Cas réels d’entreprise – Worldflik (plateforme de streaming)

Worldflik est une plateforme de streaming vidéo qui compte des millions d’utilisateurs. Elle utilise Apache Kafka comme épine dorsale pour traiter plus de 1 million d’événements par seconde : clics, lectures, recherches, paiements, logs système. Voici quatre cas d’usage réels déployés en production, avec les architectures, les patterns, les extraits de code et les leçons apprises.

📺 1. Moteur de recommandation en temps réel

Problème : Proposer des films/séries personnalisés à chaque utilisateur en fonction de ses dernières actions (visionnages, likes, recherches). L’algorithme doit réagir en quelques secondes.

Solution : Kafka Streams + Redis (cache des similarités) + modèle ML précalculé.

Architecture

Actions utilisateur (applis, web) → Topic "user-events" → Kafka Streams (agrégation par fenêtre glissante) → Mise à jour du profil utilisateur dans Redis → API de recommandation lit Redis + modèle ML.

Extrait Kafka Streams (Java)

KStream<String, UserAction> actions = builder.stream("user-events");
KTable<String, UserProfile> profiles = actions
    .groupByKey()
    .windowedBy(SessionWindows.with(Duration.ofMinutes(5)))
    .aggregate(
        () -> new UserProfile(),
        (key, action, profile) -> profile.update(action),
        Materialized.as("user-profile-store")
    );

Le profil est ensuite poussé dans Redis via un sink Kafka Connect Redis (ou un consommateur dédié).

Leçons apprises :

  • Utiliser des fenêtres de session pour capturer l’activité par session utilisateur.
  • Scaler les instances de Kafka Streams en augmentant les partitions du topic source.
  • Mettre en cache les profils dans Redis pour réduire la latence de l’API.

🛡️ 2. Détection de fraudes aux essais gratuits

Problème : Des utilisateurs malveillants créent des comptes multiples avec de fausses identités ou cartes bancaires pour profiter indéfiniment des essais gratuits.

Solution : Debezium capture les inscriptions en base PostgreSQL → ksqlDB analyse les patterns (même adresse IP, même carte bancaire, même appareil) → alerte en temps réel.

Architecture

Base de données utilisateurs (PostgreSQL) → Debezium CDC → Topic "postgres.public.users"
Topic "postgres.public.payments" → ksqlDB (jointure + fenêtre tumbling) → Topic "fraud-alerts" → Service d’alerte (email, SMS, blocage).

Requête ksqlDB pour détecter les multi-comptes par IP

CREATE STREAM users_raw (before ..., after ...) WITH (...);
CREATE STREAM users AS SELECT after->id AS user_id, after->ip_address AS ip FROM users_raw WHERE op='c';
CREATE TABLE fraud_count AS
    SELECT ip,
           COUNT(DISTINCT user_id) AS distinct_users,
           WINDOWSTART AS window_start
    FROM users
    WINDOW TUMBLING (SIZE 1 HOUR)
    GROUP BY ip
    HAVING COUNT(DISTINCT user_id) > 5
    EMIT CHANGES;

CREATE STREAM fraud_alerts AS
    SELECT ip, distinct_users, 'multi-account' AS reason
    FROM fraud_count
    EMIT CHANGES;

Leçons apprises :

  • Les fenêtres tumbling courtes (1 heure) permettent de réagir vite.
  • Utiliser des jointures entre plusieurs topics (inscriptions + paiements) pour une détection plus fine.
  • Mettre en place une Dead Letter Queue pour les événements mal formés.

📊 3. Pipeline analytique vers un data lake (S3)

Problème : Les équipes data doivent analyser des volumes massifs de logs (visionnages, clics, erreurs) pour des rapports quotidiens et de l’apprentissage automatique. Le stockage dans Kafka n’est que temporaire (7 jours).

Solution : Kafka Connect S3 Sink pour archiver les événements au format Parquet, partitionné par date.

Configuration du connecteur S3 Sink

{
  "name": "s3-sink",
  "config": {
    "connector.class": "io.confluent.connect.s3.S3SinkConnector",
    "s3.region": "us-east-1",
    "s3.bucket.name": "worldflik-data-lake",
    "topics": "user-events,server-logs,payments",
    "flush.size": 10000,
    "rotate.interval.ms": 3600000,
    "partitioner.class": "io.confluent.connect.storage.partitioner.TimeBasedPartitioner",
    "path.format": "'year'=YYYY/'month'=MM/'day'=dd/'hour'=HH",
    "locale": "en-US",
    "timezone": "UTC",
    "format.class": "io.confluent.connect.s3.format.parquet.ParquetFormat"
  }
}

Les données sont ensuite interrogables via Athena, Spark, ou Trino.

Leçons apprises :

  • Utiliser Parquet pour de meilleures performances de lecture et la compression.
  • Partitionner par date/heure pour faciliter les requêtes et la gestion des coûts S3.
  • Surveiller le lag du connecteur S3 (les écritures S3 peuvent être lentes).

⏱️ 4. Monitoring des SLAs et alerting

Problème : Worldflik doit garantir un temps de chargement des vidéos inférieur à 2 secondes (SLA). Les équipes d’infrastructure veulent des alertes immédiates en cas de dégradation.

Solution : Les services backend envoient des métriques (temps de réponse, code erreur) dans un topic Kafka. Une application Kafka Streams calcule des percentiles sur fenêtre glissante de 1 minute et déclenche des alertes.

Exemple Kafka Streams (calcul du p95)

KStream<String, Metric> metrics = builder.stream("service-metrics");
KTable<Windowed<String>, Double> p95 = metrics
    .groupBy((k, v) -> v.serviceName)
    .windowedBy(TumblingWindows.of(Duration.ofMinutes(1)))
    .aggregate(
        () -> new ArrayList<Double>(),
        (key, metric, list) -> { list.add(metric.latency); return list; },
        Materialized.as("latency-store")
    )
    .mapValues(list -> calculatePercentile(list, 95.0));

Si p95 > 2000 ms, on écrit un message dans le topic sla-alerts.

Leçons apprises :

  • Les fenêtres tumbling sont faciles à comprendre pour les SLAs.
  • Attention à la mémoire utilisée pour stocker les listes de latences – utiliser des structures optimisées (ex: HdrHistogram).
  • Intégrer les alertes avec PagerDuty ou Slack via un consommateur dédié.

🧠 Patterns réutilisables dans Worldflik

  • Event Sourcing : tous les changements d’état (inscription, visionnage, like) sont stockés dans Kafka comme source de vérité.
  • CQRS : les commandes (écritures) et les requêtes (lectures) sont séparées. Les vues matérialisées (tables ksqlDB, Redis) sont mises à jour via les événements.
  • CDC : Debezium pour capturer les modifications des bases de données transactionnelles et les propager dans Kafka.
  • Stateful processing : utilisation intensive de Kafka Streams et ksqlDB pour maintenir des agrégations temps réel.

📈 Leçons apprises en production

  • Partitionnement : Worldflik utilise 36 partitions pour les topics à fort débit (ex: user-events) afin de paralléliser les consommateurs.
  • Exactly-once : Activé pour les pipelines critiques (facturation, recommandation) pour éviter les doublons ou pertes.
  • Schema Registry : Tous les événements utilisent Avro avec évolution de schéma rétrocompatible.
  • Monitoring : Utilisation de Prometheus + Grafana pour les métriques Kafka (lag, débit, état des connecteurs) et des dashboards dédiés par équipe.
  • Tests de résilience : Chaos Monkey arrête aléatoirement des brokers Kafka et des workers Connect pour valider la tolérance aux pannes.
  • Gestion des coûts : Rétention configurée à 7 jours pour les topics opérationnels, 30 jours pour les topics d’audit, et archiving vers S3 pour l’historique.

  • Producteurs (20%) : configuration, sérialisation, partitionnement, idempotence, transactions, gestion des erreurs.

  • Kafka Streams (25%) : topologie, KStream vs KTable, fenêtrage (tumbling, hopping, session), états (RocksDB), exactly-once, jointures.
  • ksqlDB (10%) : streams vs tables, push/pull queries, agrégations, fenêtres, jointures.
  • Kafka Connect (5%) : concepts de base, transformations SMT, Dead Letter Queue.

Exemple de question (style examen)

Question : Une application Kafka Streams doit compter les mots dans un flux de phrases et émettre les résultats toutes les 5 secondes (fenêtre hopping). Quelle API utiliser ?
A) TimeWindows.of(Duration.ofSeconds(5)).advanceBy(Duration.ofSeconds(1))
B) JoinWindows.of(Duration.ofSeconds(5))
C) SessionWindows.with(Duration.ofSeconds(5))
D) TumblingWindows.of(Duration.ofSeconds(5))

Réponse : A (Hopping window avec avancement).

🖥️ 2. Confluent Certified Administrator (CCAAK)

Cette certification se concentre sur l’installation, la configuration, la surveillance, la sécurisation et l’optimisation d’un cluster Kafka en production.

Domaines du programme

  • Installation et configuration (20%) : brokers, Zookeeper/KRaft, configuration des logs, réplication, quotas.
  • Administration et opérations (25%) : ajout/suppression de brokers, réaffectation des partitions, équilibrage de charge, MirrorMaker 2.
  • Surveillance et métriques (20%) : JMX, métriques clés (ISR, débit, lag, taille des partitions), outils (Burrow, Cruise Control).
  • Sécurité (20%) : SSL/TLS, SASL (SCRAM, PLAIN, Kerberos), ACLs, rôles, configuration sécurisée.
  • Gestion des données (15%) : rétention, compaction, nettoyage des logs, stratégies de partitionnement.

Exemple de question (style examen)

Question : Un broker Kafka tombe en panne. Quelle condition doit être vraie pour qu’une partition reste disponible ?
A) Le nombre de réplicas ISR est supérieur ou égal à 1
B) min.insync.replicas est strictement inférieur au nombre de réplicas
C) Le leader est toujours sur le broker tombé
D) Le facteur de réplication est supérieur à 1

Réponse : D (facteur de réplication > 1 permet au cluster d’élire un nouveau leader parmi les followers).

🧠 Conseils pour le jour de l’examen

  • L’examen est en ligne, surveillé par un proctor. Assurez-vous d’avoir une connexion stable, une webcam, et un environnement calme.
  • Lisez attentivement chaque question – certaines comportent des pièges (ex: “Kafka Streams ne peut pas faire X” alors que si).
  • Pour les questions de code, vérifiez les imports et les types (Serde, Duration, etc.).
  • Gérez votre temps : 1 minute par question en moyenne. Marquez les questions difficiles pour y revenir.
  • Ne changez pas une réponse sauf si vous êtes certain d’une erreur – l’intuition initiale est souvent bonne.

Administrateur
