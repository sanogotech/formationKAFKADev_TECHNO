De la source DB → Kafka → Sink

Fil conducteur : Catalogue e-commerce en temps réel

Cas pratique : Une marketplace souhaite synchroniser son catalogue produits (PostgreSQL) avec un moteur de recherche Elasticsearch et un cache Redis, sans réécrire d’application.

  1. Debezium PostgreSQL connector capture chaque INSERT/UPDATE/DELETE sur la table produits.
  2. Les événements CDC sont poussés dans le topic Kafka catalogue.public.produits.
  3. Un sink connector Elasticsearch consomme ce topic et indexe les documents.
  4. Un second sink Redis (via Redis Sink) met à jour le cache.
  5. Les transformations SMT (Single Message Transform) masquent les données sensibles et ajoutent un champ timestamp_index.
Pour l’examen / production : Ce pattern CDC évite la double écriture, garantit l’ordre et la résilience grâce à Kafka.

Cas complet : Pipeline CDC de bout en bout

Objectif : Mettre en place un flux de données continu depuis une base PostgreSQL vers Elasticsearch et Redis, avec tolérance aux pannes, monitoring et exactly-once (idempotent). Voici les étapes détaillées d’un projet réel.

Étape 1 – Préparation de la base source (PostgreSQL)
  • Activer la réplication logique : wal_level = logical dans postgresql.conf.
  • Créer un utilisateur dédié : CREATE USER debezium WITH REPLICATION LOGIN PASSWORD 'dbz'.
  • Donner les droits : GRANT SELECT ON ALL TABLES IN SCHEMA public TO debezium;.
Étape 2 – Déployer Kafka Connect (cluster distribué)
  • Démarrer 3 workers Kafka Connect (ex: avec Docker Compose). Configuration clé : group.id=connect-cluster, config.storage.topic=connect-configs, offset.storage.topic=connect-offsets, status.storage.topic=connect-status.
  • Installer le plugin Debezium (JARs) dans plugin.path de chaque worker.
  • Ajouter le sink Elasticsearch (confluentinc-elasticsearch-connector) et Redis (Redis Sink).
Étape 3 – Créer le connecteur source Debezium (via API REST)
curl -X POST http://connect:8083/connectors \
  -H "Content-Type: application/json" \
  -d '{
    "name": "catalogue-postgres-connector",
    "config": {
      "connector.class": "io.debezium.connector.postgresql.PostgresConnector",
      "database.hostname": "postgres",
      "database.port": "5432",
      "database.user": "debezium",
      "database.password": "dbz",
      "database.dbname": "catalogue",
      "database.server.name": "catalogue",
      "table.include.list": "public.produits",
      "topic.prefix": "catalogue",
      "snapshot.mode": "initial",
      "decimal.handling.mode": "string",
      "key.converter": "io.confluent.connect.avro.AvroConverter",
      "value.converter": "io.confluent.connect.avro.AvroConverter",
      "key.converter.schema.registry.url": "http://schema-registry:8081",
      "value.converter.schema.registry.url": "http://schema-registry:8081",
      "transforms": "maskPrice,addTimestamp",
      "transforms.maskPrice.type": "org.apache.kafka.connect.transforms.MaskField$Value",
      "transforms.maskPrice.fields": "prix_achat",
      "transforms.addTimestamp.type": "org.apache.kafka.connect.transforms.InsertField$Value",
      "transforms.addTimestamp.timestamp.field": "capture_timestamp"
    }
  }'

Le connecteur va d’abord faire un snapshot complet de la table produits, puis passer en streaming des changements (insert/update/delete).

Étape 4 – Créer le sink Elasticsearch
curl -X POST http://connect:8083/connectors \
  -H "Content-Type: application/json" \
  -d '{
    "name": "elasticsearch-sink",
    "config": {
      "connector.class": "io.confluent.connect.elasticsearch.ElasticsearchSinkConnector",
      "connection.url": "http://elasticsearch:9200",
      "type.name": "_doc",
      "topics": "catalogue.public.produits",
      "key.ignore": "false",
      "schema.ignore": "false",
      "behavior.on.null.values": "delete",
      "write.method": "upsert",
      "transforms": "extractKey",
      "transforms.extractKey.type": "org.apache.kafka.connect.transforms.ExtractField$Key",
      "transforms.extractKey.field": "id"
    }
  }'

Ici, on utilise upsert pour garantir l’idempotence : si le même document est réindexé plusieurs fois, Elasticsearch ne crée pas de doublon.

Étape 5 – Créer le sink Redis pour le cache
curl -X POST http://connect:8083/connectors \
  -H "Content-Type: application/json" \
  -d '{
    "name": "redis-sink",
    "config": {
      "connector.class": "com.github.jcustenborder.kafka.connect.redis.RedisSinkConnector",
      "redis.host": "redis",
      "redis.port": "6379",
      "topics": "catalogue.public.produits",
      "key.converter": "org.apache.kafka.connect.storage.StringConverter",
      "value.converter": "io.confluent.connect.avro.AvroConverter",
      "value.converter.schema.registry.url": "http://schema-registry:8081",
      "redis.command": "HSET",
      "redis.key.format": "product:{{key}}",
      "redis.field.format": "data"
    }
  }'

Redis stocke chaque produit sous forme de hash product:<id>. En cas de suppression (op = d), on peut ajouter une SMT pour envoyer une commande DEL.

Étape 6 – Vérifier le flux et surveiller
  • Consulter les topics : kafka-console-consumer --bootstrap-server localhost:9092 --topic catalogue.public.produits --from-beginning (avec Avro, utiliser kafka-avro-console-consumer).
  • Surveiller les métriques : API REST /connectors/catalogue-postgres-connector/status, /connectors/elasticsearch-sink/status.
  • Vérifier le lag : kafka-consumer-groups --bootstrap-server localhost:9092 --group connect-cluster --describe.
  • Simuler une modification : UPDATE produits SET prix = 19.99 WHERE id = 42; → constater l’indexation quasi instantanée dans Elasticsearch et Redis.
Étape 7 – Gérer les erreurs et la reprise
  • Dead Letter Queue (DLQ) : configurer errors.tolerance=all et errors.deadletterqueue.topic.name=catalogue-dlq sur les connecteurs sink.
  • Redémarrage après crash : Les offsets sont stockés dans Kafka, le connecteur Debezium reprend exactement là où il s’était arrêté.
  • Réinitialisation du snapshot : Modifier snapshot.mode=never après le premier run, ou utiliser incremental snapshot via Debezium.
Résultat : Un pipeline entièrement découplé, scalable, tolérant aux pannes. Aucune ligne de code métier n’a été écrite pour la synchronisation – uniquement de la configuration.

18 concepts Kafka Connect & Debezium – version immersive

Analogies métier, extraits techniques, et liens avec le fil conducteur (catalogue produit).

Connect Worker
Ouvrier polyvalent
📦 Analogie : Un ouvrier dans un atelier (cluster Connect). Chaque worker exécute des tâches (tasks) et gère la configuration.
💡 Technique : Workers distribués (mode distribué) ou standalone. Le leader coordonne la répartition des tâches via le topic de configuration interne connect-configs.
🔑 Dans l’exemple : 3 workers distribuent les tâches source (Debezium) et sink (Elasticsearch).
Connector
Adaptateur source ou puit
📦 Analogie : Une prise électrique européenne vers américaine : le connector transforme le protocole externe en format Kafka.
💡 Examen : Source connector = lit depuis une source externe (DB, fichier). Sink connector = écrit vers un système externe. connector.class définit l’implémentation.
🔑 Notre pipeline : DebeziumPostgresConnector (source) + ElasticsearchSinkConnector.
Task
Tâche parallèle
📦 Analogie : Un ouvrier peut gérer plusieurs chaînes en parallèle. Une task = une unité de travail exécutée par un worker.
💡 Technique : tasks.max définit le parallélisme. Pour Debezium, une task par table surveillée (ou partitionnement manuel).
🔑 Bonne pratique : Augmenter tasks.max si le volume de changements est élevé, mais Debezium est souvent monothread par table (garantie d’ordre).
Converter
Traducteur de données
📦 Analogie : Un interprète qui traduit le dialecte PostgreSQL (format interne) en JSON ou Avro avant de l’envoyer dans Kafka.
💡 Examen : key.converter et value.converter. Avec Schema Registry, on utilise io.confluent.connect.avro.AvroConverter.
🔑 Dans le catalogue : Converter Avro + Schema Registry pour gérer l’évolution du schéma produit.
SMT (Transformations)
Petite modification locale
📦 Analogie : Un tampon encreur qui ajoute une mention "Confidentiel" sur chaque document avant de l’archiver.
💡 Exemples : MaskField (cacher email), InsertField (ajouter timestamp), DropHeaders, RegexRouter (renommer topic).
🔑 Dans notre flux : SMT MaskField$Value pour masquer le champ prix_achat interne.
Debezium (CDC)
Œil permanent sur la base
📦 Analogie : Un notaire qui enregistre chaque modification dans un registre (binlog/WAL) et les rédige en messages Kafka.
💡 Fonctionnement : Se connecte au mécanisme de réplication native (PostgreSQL logical decoding, MySQL binlog, MongoDB oplog). Pas de polling, événementiel pur.
🔑 Avantage : Faible impact sur la source, capture before/after, opération c,u,d,r.
Nommage des topics
Serveur.base.table
📦 Analogie : Adresse postale : serveur.schema.table. Exemple : postgres.catalogue.produits.
💡 Configuration : topic.prefix ou database.server.name. Le message contient l’enveloppe before/after, op, ts_ms.
🔑 Filtre : table.include.list pour ne capturer que certaines tables.
Snapshot initial
Photo de la base au démarrage
📦 Analogie : Avant de noter les changements, on recopie l’état complet de la base (première photographie).
💡 Paramètre : snapshot.mode=initial (par défaut) prend un snapshot puis passe en streaming. never pour démarrer sans snapshot.
🔑 Incremental snapshot : Permet de re-snapshotter des tables sans arrêter le connecteur (Debezium 1.6+).
Offsets Debezium
Marque-page du binlog
📦 Analogie : Un signet qui indique la dernière ligne lue dans le journal de modifications (WAL).
💡 Technique : Les offsets sont stockés dans un topic Kafka interne connect-offsets ou un fichier (standalone). Permet de reprendre après redémarrage exactement là où on s’était arrêté.
🔑 Danger : Si on supprime ce topic, le connecteur refait un snapshot complet.
Dead Letter Queue
Poubelle des erreurs
📦 Analogie : Un bac à rebuts où l’on met les documents illisibles au lieu de bloquer toute la chaîne.
💡 Connect : errors.tolerance=all et errors.deadletterqueue.topic.name=my-dlq. Les messages en erreur (désérialisation, transformation) y sont déposés.
🔑 Dans notre pipeline : Si Elasticsearch rejette un document, il atterrit dans DLQ pour analyse.
Configuration JSON
Recette du connector
📦 Analogie : Une fiche de réglage qui indique à l’ouvrier quel adaptateur utiliser, avec quels paramètres.
💡 Exemple Debezium : {"name": "catalogue-connector", "config": {"connector.class": "io.debezium.connector.postgresql.PostgresConnector", "database.hostname": "postgres", ...}}
🔑 API REST : POST /connectors pour créer, PUT /connectors/.../config pour mettre à jour.
API REST & JMX
Console de pilotage
📦 Analogie : Un panneau de commande central pour voir l’état des ouvriers, démarrer/arrêter des chaînes.
💡 Endpoints utiles : /connectors, /connectors/{name}/status, /connectors/{name}/tasks. JMX expose métriques (taux d’erreur, temps de traitement).
🔑 Supervision : Surveiller le statut RUNNING / FAILED et le lag du connecteur source.
Sink Connector
Puit de données
📦 Analogie : Un entonnoir qui récupère les messages Kafka et les verse dans un lac (Elasticsearch, S3).
💡 Exemple : ElasticsearchSinkConnector : indexe chaque message avec key = ID document. Supporte upsert.
🔑 Dans le catalogue : Sink Elasticsearch pour rendre les produits recherchables instantanément après modification.
SMT Avancées
Modif à la volée
📦 Analogie : Un tampon qui ajoute automatiquement la date et l’expéditeur sur chaque enveloppe.
💡 SMT courantes : ExtractTopic$Key, ReplaceField, TimestampConverter, ValueToKey.
🔑 Bonne pratique : Chaîner plusieurs SMT (transforms=addTS,maskCard). L’ordre compte.
Évolution de schéma
Agrandir le formulaire
📦 Analogie : On ajoute une colonne "couleur" au formulaire produit. Les anciens lecteurs continuent de fonctionner (compatibilité backward).
💡 Avec Schema Registry : Le converter Avro vérifie la compatibilité. Debezium gère l’ajout/suppression de colonnes sans interruption.
🔑 Conseil certif : Toujours définir une stratégie de compatibilité (BACKWARD, FULL) pour éviter les ruptures.
Rebalance des tasks
Répartition dynamique
📦 Analogie : Si un ouvrier quitte l’atelier, le chef réaffecte ses tâches aux autres automatiquement.
💡 Connect : Lorsqu’un worker rejoint/quitte le groupe, les tâches sont rééquilibrées. Les connecteurs source/sink peuvent être redémarrés sur d’autres workers.
🔑 Pour Debezium : Une task par table ne peut pas être rééquilibrée en cours de snapshot – mieux vaut éviter les rebalances fréquentes.
Exactly-once dans Connect
Au moins une fois par défaut
📦 Analogie : Un facteur peut livrer deux fois le même colis (at-least-once). Pour exactement une fois, il faut un sink idempotent.
💡 Connect source : Les offsets sont commités périodiquement, possible doublon en cas de crash. Sink idempotent (ex: upsert sur clé primaire) permet d’absorber les doublons.
🔑 Debezium : Les messages contiennent des métadonnées lsn (PostgreSQL) et peuvent être dédupliqués à la consommation.
Monitoring & santé
Tableau de bord vital
📦 Analogie : Un ECG qui surveille le rythme cardiaque de l’atelier.
💡 Métriques clés : source-record-poll-rate (Debezium), sink-record-send-rate, task-errors-total, lag du connecteur source (différence entre dernière écriture Kafka et dernière transaction DB).
🔑 Outils : Prometheus + JMX Exporter, Confluent Control Center, ou API REST.

Bonnes pratiques & réussite projet

Pour Debezium en production :
  • Droits spécifiques : créer un utilisateur dédié avec les privilèges de réplication (ex: REPLICATION pour PostgreSQL).
  • Gérer le snapshot : planifier le snapshot initial sur des heures creuses ou utiliser snapshot.mode=never si la base est déjà préchargée.
  • Surveiller le lag : io.debezium.connector.common.CdcSourceTask expose la métrique MilliSecondsBehindSource.
  • Taille des messages : éviter les lignes trop larges (ajuster max.request.size).
  • Topic compaction : activer la compaction pour les tables de référence (KTable).
Kafka Connect – astuces avancées
  • Configurations clés : offset.flush.interval.ms, task.shutdown.graceful.timeout.ms, consumer.auto.offset.reset pour les sinks.
  • Isolation des connecteurs : utiliser connector.client.config.override.policy=All avec prudence.
  • Déploiement des plugins : placez les JARs Debezium dans plugin.path sur chaque worker (ou partage NFS).
  • REST API pour CI/CD : versionnez les configurations JSON, utilisez curl pour déployer automatiquement.
  • Transformations SMT : préférez plusieurs petites transformations à une seule complexe pour la maintenabilité.
Pour la certification Confluent (CCDAK / confluent developer) : Les sujets Connect incluent la différence entre source/sink, le rôle des workers, la configuration des convertisseurs, les SMT, et les patterns de tolérance aux pannes. Debezium est abordé via le CDC et la capture des changements sans perte.

Mémorisez : “Worker + Task + Connector = pipeline ; Debezium = binlog → Kafka ; SMT = petite touche magique ; Schema Registry = l’arme de l’évolution.”
