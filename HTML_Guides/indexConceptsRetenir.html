Qu'est-ce qu'Apache Kafka ?

Kafka est une plateforme de streaming distribuée, open-source, conçue pour gérer des flux de données à très haut débit, en temps réel, avec une tolérance aux pannes exceptionnelle. Initialement développé par LinkedIn en 2011, puis donné à la Fondation Apache. Il combine les fonctionnalités d’un message broker, d’un stockage durable (commit log) et d’un système de traitement de flux.

Historique & évolution

  • 2011 - LinkedIn crée Kafka pour remplacer leurs pipelines de données hétérogènes.
  • 2012 - Apache Incubator, devient top-level project.
  • 2017 - Kafka Streams mature, KSQL apparaît.
  • 2020+ - KRaft (remplacement ZooKeeper), exactly-once semantics, eco-système Confluent.

Aujourd'hui : utilisé par 80% du Fortune 500, cœur de l'architecture data-driven.

21 concepts Kafka - Version ultra-détaillée

Chaque carte contient : analogie approfondie, exemples concrets, astuces techniques pour dev/archi, et mnémonique renforcée.

Topic
Mnémo : T comme Tiroir de classement
📦 Analogie détaillée : Un topic est comme un classeur entier dans un bureau. Chaque classeur possède un nom unique (ex : « Factures », « Logs applicatifs ») et regroupe des documents de même nature. Si vous devez archiver toutes les commandes clients, vous créez un classeur « Commandes ».
💻 Exemple concret : Dans une plateforme de livraison, topic orders contient tous les événements de commande. Un autre topic payments stocke les paiements. Les producteurs écrivent dans ces topics, les consommateurs s’abonnent aux topics pertinents.
🧠 Pour l’architecte : Les topics sont logiques et partitionnés. Ils supportent la rétention et la réplication. Un cluster peut avoir des milliers de topics. Nommez-les avec une convention (ex: env.service.donnee).
🔁 Rappel mnémotechnique : « Topic = Tiroir » : chaque tiroir a un nom et contient des messages classés.
Partition
Mnémo : P comme Page numérotée
📦 Analogie : Si le topic est un classeur, la partition est un sous-dossier à onglet numéroté (0,1,2…). Chaque sous-dossier contient des fiches dans l’ordre chronologique. Kafka garantit l’ordre à l’intérieur d’une partition mais pas entre partitions.
💻 Exemple : Topic clics-utilisateurs avec 4 partitions. Le producteur avec clé = user_123 ira toujours dans la même partition (hash). Ainsi tous les clics d’un même utilisateur sont ordonnés.
🧠 Technique : Le parallélisme d’un topic est égal au nombre de partitions. Un consumer group peut avoir au maximum autant de consommateurs que de partitions. Plus de partitions = plus de throughput, mais aussi plus de fichiers ouverts et de latence de métadonnées.
Offset
Mnémo : O comme Ordre absolu
📦 Analogie : Le numéro de page dans un sous-dossier. La première fiche porte l’offset 0, la suivante 1, etc. Ce numéro est immuable et incrémental. On ne peut pas modifier un offset existant.
💻 Exemple : Partition 0, offsets 0 à 99. Un consommateur lit offset 45 à 78, puis reprend plus tard à 79. Kafka utilise les offsets pour suivre la progression de lecture (committed offset).
🧠 Pour le développeur : L’offset est stocké dans un topic interne (__consumer_offsets). Permet de « rejouer » des messages en remettant l’offset à zéro. Attention : les offsets ne sont pas uniques globalement, seulement par partition.
Producer
Mnémo : Pousse les messages (push)
📦 Analogie : Un employé qui écrit une fiche et la dépose dans le classeur. Il peut choisir dans quel sous-dossier (partition) la mettre, ou laisser Kafka décider via une clé.
💻 Exemple : Application backend qui, après validation d’un paiement, produit un message {"orderId":123, "status":"paid"} vers le topic payments.
🧠 Configuration clé : acks=all (attente réplication), compression.type=gzip, retries, idempotence=true. Le producteur peut envoyer des batchs pour optimiser le débit.
Consumer
Mnémo : Consulte (pull)
📦 Analogie : Un employé qui lit une fiche dans le classeur, la traite, puis laisse sa place. Il tire les messages (pull) à son propre rythme, contrairement à certains brokers push.
💻 Exemple : Service de notification qui consomme le topic orders, et pour chaque commande envoie un email au client. Si le service est lent, Kafka ne le surcharge pas.
🧠 Bonnes pratiques : Gérer la session.timeout.ms et heartbeat. Utiliser enable.auto.commit=false et commiter manuellement après traitement pour éviter les pertes ou doublons. Un consumer appartient à un consumer group.
Broker
Mnémo : Boîtier serveur
📦 Analogie : Une étagère dans la bibliothèque. Chaque broker (serveur Kafka) stocke plusieurs partitions de différents topics. Un broker unique peut gérer des téraoctets de données.
💻 Exemple : Un cluster de 3 brokers. Broker 1 héberge la partition leader du topic A, partition follower du topic B. Broker 2 héberge d’autres partitions.
🧠 Architecture : Chaque broker a un ID unique. Les clients se connectent à un broker (bootstrap) qui leur renvoie la métadonnée complète du cluster. Un broker peut être leader ou follower pour différentes partitions. La montée en charge se fait en ajoutant des brokers.
Cluster
Mnémo : Collectif de brokers
📦 Analogie : La bibliothèque entière composée de plusieurs étagères (brokers). Toutes travaillent ensemble pour offrir un service unifié, tolérant aux pannes et scalable.
💻 Exemple : Cluster de 5 brokers, facteur de réplication 3. Même si 2 brokers tombent, le cluster reste opérationnel (car les partitions leaders sont sur les brokers restants).
🧠 Pour architecte : Le cluster est identifié par un cluster.id. La communication interne utilise le protocole Kafka (binaire). Les outils d’administration (Kafka AdminClient, kafka-topics) agissent au niveau cluster.
Consumer Group
Mnémo : Coopération de groupe
📦 Analogie : Une équipe de lecteurs qui se répartissent les sous-dossiers d’un même classeur. Chaque lecteur est responsable d’un sous-dossier (partition). Si un lecteur tombe malade, ses sous-dossiers sont attribués à un autre.
💻 Exemple : Topic avec 6 partitions. Un groupe avec 3 consommateurs : chacun prend 2 partitions. Si un consommateur s’arrête, les 2 partitions sont rééquilibrées entre les 2 restants.
🧠 Important : Dans un groupe, chaque partition est consommée par un seul consommateur. Le nombre de consommateurs ne peut pas dépasser le nombre de partitions, sinon certains restent inactifs. Le rééquilibrage (rebalance) peut causer une courte pause.
Réplication
Mnémo : Redondance photocopie
📦 Analogie : Faire des photocopies de chaque fiche et les ranger sur d’autres étagères (brokers différents). Si l’original brûle, on utilise une copie.
💻 Exemple : Facteur de réplication = 3 pour le topic transactions. Chaque message est écrit sur 3 brokers différents. Un broker peut tomber sans perte de données.
🧠 Détail technique : Une partition a un leader (lecture/écriture) et des followers (qui copient passivement). Les followers demandent les données au leader via des requêtes fetch. La réplication asynchrone peut mener à des pertes si le leader meurt avant que les followers aient copié (config min.insync.replicas).
Leader / Follower
Mnémo : L comme Lecture/écriture
📦 Analogie : Un chef d’équipe (leader) gère toutes les demandes d’écriture et de lecture pour sa partition. Ses adjoints (followers) ne font que copier les données du leader pour être prêts à prendre la relève.
💻 Exemple : Partition 0 du topic logs : le leader est Broker 2, les followers sont Brokers 1 et 3. Tous les producteurs écrivent sur Broker 2. Si Broker 2 tombe, un follower (Broker 1) devient automatiquement leader.
🧠 Pour l’architecte : L’élection du leader est gérée par le contrôleur (un broker spécial ou via KRaft). Les followers ne servent que pour la réplication et la haute disponibilité. Un broker peut être leader pour certaines partitions et follower pour d’autres.
ZooKeeper / KRaft
Mnémo : Secrétaire (Zoo) ou KRaft intégré
📦 Analogie : Un secrétaire qui connaît l’emplacement de toutes les étagères, qui désigne les chefs (leaders) et tient à jour le registre du cluster. Depuis Kafka 3.x, le secrétaire est intégré dans le cluster lui-même : c’est KRaft.
💻 Exemple : ZooKeeper stocke les métadonnées : liste des brokers, configuration des topics, quotas, ACL. Avec KRaft, Kafka utilise un protocole Raft pour élire un contrôleur.
🧠 Évolution : KRaft supprime la dépendance externe, simplifie l’exploitation, améliore la scalabilité (plus de limite de 200k partitions). Pour les nouvelles installations, préférez KRaft (Kafka 3.3+).
Message / Record
Mnémo : Missive complète
📦 Analogie : Une fiche avec plusieurs champs : en-tête (clé), corps (valeur), horodatage, et métadonnées optionnelles. C’est l’unité de base transportée par Kafka.
💻 Exemple : key="user_42", value="{\"action\":\"click\", \"page\":\"home\"}", timestamp=1698765432000, headers={"source":"web"}.
🧠 Technique : La taille maximale par défaut est 1 Mo (configurable). Les messages peuvent être compressés (batch). Les en-têtes (headers) sont utiles pour du routage ou métadonnées sans modifier la valeur.
Key (Clé)
Mnémo : Kabinet de classement
📦 Analogie : Une étiquette de couleur sur la fiche. Cette couleur détermine automatiquement dans quel sous-dossier (partition) la fiche sera rangée. Toutes les fiches avec la même couleur vont dans le même sous-dossier.
💻 Exemple : Clé = customerId. Tous les événements d’un même client vont dans la même partition, donc ils sont lus dans l’ordre par un consommateur. Sans clé, les messages sont répartis en round-robin.
🧠 Pour le développeur : La clé est optionnelle. Si vous avez besoin d’ordre strict par entité, fournissez une clé non nulle. Le partitionnement par défaut : hash(key) % nombre_partitions. Vous pouvez implémenter un partitionneur personnalisé.
Sérialisation / Désérialisation
Mnémo : S comme Schéma
📦 Analogie : Traduire un message en langage binaire pour l’envoyer sur le réseau (sérialiser), puis le retraduire en objet lisible (désérialiser). Sans traduction, les ordinateurs ne se comprennent pas.
💻 Exemple : Un objet Java Order est sérialisé en JSON (texte), puis Kafka le transforme en bytes. À la réception, le consommateur désérialise les bytes en JSON, puis en objet Order.
🧠 Bonnes pratiques : Utilisez Avro avec Schema Registry pour gérer l’évolution des schémas. Les sérialiseurs courants : String, JSON, Avro, Protobuf. La clé et la valeur peuvent avoir des sérialiseurs différents.
Commit Log
Mnémo : Comptable immuable
📦 Analogie : Un registre comptable où chaque écriture est ajoutée à la fin, et on ne peut jamais effacer ou modifier une ligne existante. Kafka stocke tous les messages dans l’ordre d’arrivée, comme un journal immuable.
💻 Exemple : Le répertoire de stockage sur disque contient des fichiers .log où chaque message est ajouté séquentiellement. Une lecture aléatoire est possible mais optimisée via index.
🧠 Architecture : Cette conception permet des lectures séquentielles ultra-rapides, et une réplication simple. La suppression se fait par « compaction » ou par expiration (suppression de vieux segments). Pas de mise à jour sur place.
Retention Policy
Mnémo : Retrait des anciens
📦 Analogie : Un videur qui jette les fiches trop vieilles (par durée) ou lorsque le classeur est trop plein (par taille). Ainsi la bibliothèque ne sature pas.
💻 Exemple : Topic avec retention.ms=604800000 (7 jours) et retention.bytes=1073741824 (1 Go). Les messages plus vieux que 7 jours sont supprimés, et si la taille totale dépasse 1 Go, les plus anciens sont éliminés.
🧠 Pour l’architecte : Deux politiques : delete (suppression) ou compact (compaction, garde seulement la dernière valeur par clé). La compaction est utile pour les tables de référence (KTable).
Compression
Mnémo : Compact et rapide
📦 Analogie : Aspirer l’air d’un coussin pour le rendre plus petit à transporter. Le destinataire le regonfle (décompresse) pour l’utiliser. Gain de place et de bande passante.
💻 Exemple : Activer compression.type=gzip sur un topic de logs texte. La taille passe de 100 Mo à 30 Mo. Les producteurs compressent par batch, les consommateurs décompressent automatiquement.
🧠 Types supportés : gzip, snappy, lz4, zstd. Snappy est rapide, gzip meilleur taux. La compression est transparente pour le consommateur. Attention : un broker peut recompresser si les configurations diffèrent.
Idempotent Producer
Mnémo : Interdit de dupliquer
📦 Analogie : Un facteur qui note le numéro de chaque lettre dans son carnet. S’il doit réexpédier une lettre à cause d’un problème réseau, il vérifie qu’il ne l’a pas déjà envoyée. Pas de doublon.
💻 Exemple : Producer avec enable.idempotence=true. Même après un timeout et une retransmission, Kafka garantit que le message n’est écrit qu’une seule fois dans le log.
🧠 Technique : Basé sur les PID (Producer ID) et les numéros de séquence par partition. Nécessite max.in.flight.requests.per.connection <= 5 et acks=all. Idempotence est un prérequis pour Exactly-Once Semantics.
Exactly-once (EOS)
Mnémo : Exact One Shot
📦 Analogie : Un transfert bancaire parfait : ni perte, ni double dépense, ni omission. Chaque message est traité exactement une fois, même en cas de panne ou de rejeu.
💻 Exemple : Kafka Streams avec processing.guarantee=exactly_once_v2. Un consommateur lit un topic, transforme, et produit dans un autre topic le tout dans une transaction atomique.
🧠 Pour architecte : Combine l’idempotence du producteur et les transactions (atomicité lecture-écriture). Les transactions Kafka permettent de committer les offsets du consommateur et les messages produits ensemble. Impact sur les performances.
Kafka Streams
Mnémo : Stream = ruisseau continu
📦 Analogie : Une chaîne de montage qui lit les messages à la volée, les transforme (filtre, agrandit, joint, compte) et les réécrit dans un autre topic. Tout cela en continu, sans base externe.
💻 Exemple : Lire un topic clics, faire une fenêtre de 5 secondes, compter les clics par utilisateur, écrire dans stats-clics. Code Java avec KStream et KTable.
🧠 Pour le développeur : Bibliothèque (pas de serveur séparé) intégrable dans n’importe quelle app JVM. Supporte l’état (state stores) via RocksDB. Garanties exactly-once, tolérance aux pannes, répartition automatique.
Kafka Connect
Mnémo : Câble universel
📦 Analogie : Un adaptateur universel qui branche Kafka à des sources (bases de données, fichiers, IoT) ou des puits (Elasticsearch, S3, HDFS) sans écrire une seule ligne de code.
💻 Exemple : Connecteur JDBC Source : toutes les 5 minutes, il lit les nouvelles lignes d’une table MySQL customers et les publie dans le topic mysql.customers. Connecteur S3 Sink : écrit tous les messages du topic logs dans des fichiers Parquet sur S3.
🧠 Mode distribué : Connect s’exécute en cluster (workers) pour la scalabilité et la tolérance. Supporte les transformations simples (Single Message Transforms). Idéal pour les pipelines ETL sans code.
Mémo global de la formation : Bibliothèque (cluster) → étagères (brokers) → classeurs (topics) → sous-dossiers (partitions) → pages numérotées (offsets). Producteurs écrivent, groupes de consommateurs lisent, photocopies (réplication) sécurisent. Les 21 concepts forment le socle de l’architecture event-driven moderne.
