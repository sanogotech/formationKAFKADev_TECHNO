Architectures Event-Driven : Panorama stratégique pour DSI

Dans les systèmes d'information modernes, l'approche event-driven (pilotée par les événements) transforme la façon dont les microservices communiquent, réagissent en temps réel et garantissent la scalabilité. Ce guide pose les bases pour choisir entre CQRS, Kafka-first, Event Sourcing, Outbox, CDC et les architectures synchrones.

Pourquoi ce sujet est critique ?

Les DSI doivent absorber des volumes massifs de données (IoT, logs, clics) tout en maintenant une cohérence forte pour les transactions financières ou réglementaires. L'architecture event-driven n'est pas un pattern unique mais une boîte à outils dont le mauvais choix peut entraîner pertes de données, latences ou surcoûts.

Tendance 2025 : hybride dominant

Selon une étude récente, 78% des grandes DSI utilisent au moins deux patterns event-driven combinés (ex. CQRS + Outbox + CDC). L'approche "Kafka-first pure" reste minoritaire (12%) en raison des risques de cohérence.

Checklist : votre DSI est-elle prête pour l'event-driven ?

  • Capacité à gérer l'eventual consistency acceptée par le métier
  • Mise en place d’un Schema Registry (évolution des événements)
  • Outillage de replay et de reconstruction d’états
  • Monitoring des décalages (lag) des consommateurs Kafka
  • Stratégie de gestion des doublons (idempotence)
  • Auditabilité des événements critiques (optionnel)
Point critique : si vous ne maîtrisez pas le versioning des schémas, préférez d'abord un pattern Outbox ou CDC avant de passer à Kafka-first.

Questions clés à se poser avant de choisir une architecture event-driven

🔹 Métier : Acceptez-vous une cohérence à terme (secondes/minutes) ?
🔹 Réglementation : Existe-t-il une obligation d’audit exhaustif ? (RGPD, SOX, MIFID)
🔹 Volume : Plus de 100 000 événements/seconde ? → Kafka-first ou CDC.
🔹 Legacy : Pouvez-vous modifier les applications pour publier des événements ? Sinon CDC.
🔹 Équipes : Avez-vous l'expertise pour gérer Event Sourcing (complexité) ?
🔹 Infrastructure : Kafka cluster avec réplication inter-AZ ?

Anti-patterns fréquents à éviter

  • Kafka comme base de données : requêter directement les topics (sauf pour très petits volumes).
  • Événements sans version → rupture silencieuse des consommateurs.
  • Ignorer la gestion des doublons → double écriture dans les vues.
  • Consumer unique pour tout faire → réintroduit du couplage.
  • Pas de plan de replay → impossibilité de reconstruire après corruption.

Signes que vous avez bien architecturé

  • ✔ Le monitoring montre un lag proche de zéro en régime normal.
  • ✔ Vous pouvez rejouer les événements d'une journée sans impacter la production.

CQRS + DB Write : l'architecture de référence pour SI critiques

Le pattern CQRS (Command Query Responsibility Segregation) associé à une base de données d’écriture centrale et à Kafka comme bus d’événements est la colonne vertébrale des systèmes d’information exigeants (finance, énergie, ERP). La base de données reste la source de vérité pour les commandes, tandis que Kafka assure la diffusion asynchrone vers des modèles de lecture optimisés.

Flux typique
API Command → Validation métier → Write DB (transaction) → Publication événement Kafka → Consumers → Read DB / Cache / DWH
Quand l'utiliser ?
  • Transactions financières (billing, paiements)
  • Systèmes réglementés (audit obligatoire)
  • Domaines où la cohérence forte prime sur la latence
  • Refonte d’un monolithe avec besoin de découplage progressif

Avantages opérationnels

  • Source de vérité unique – la Write DB (ACID) garantit l’intégrité.
  • Cohérence forte immédiate – après validation, la commande est enregistrée.
  • Auditabilité simple – requêtage direct de la base transactionnelle.
  • Découplage lecture/écriture – les vues Read DB peuvent être adaptées aux besoins.
  • Résilience – si Kafka est indisponible, l’écriture en DB reste possible (mode dégradé).
  • Évolutivité verticale et horizontale – réplication lecture, sharding écriture possible.

Inconvénients et défis

  • Double écriture (DB + Kafka) – risque de désynchronisation sans pattern Outbox.
  • Latence supplémentaire – l’événement n’est disponible qu’après commit DB.
  • Complexité de synchronisation – garantir l’ordre des événements.
  • Consommation de ressources – maintenir à jour les vues de lecture.
  • Gestion des transactions distribuées – souvent déléguée à des patterns comme Outbox ou Saga.

Checklist pour implémenter CQRS + DB Write avec Kafka

  • Séparer les modèles – Command model (write) ≠ Query model (read).
  • Utiliser un pattern Outbox pour publier les événements de façon transactionnelle.
  • Garantir l’idempotence des consommateurs pour supporter les rejeux Kafka.
  • Versionner les événements (Schema Registry) pour évoluer sans casser les consommateurs.
  • Monitorer le lag Kafka – alerter si les consumers accusent un retard.
  • Prévoir un mécanisme de replay pour reconstruire les Read DB.
  • Tester la résilience – simuler une panne Kafka, vérifier le mode dégradé.
  • Documenter les événements métier – contrat explicite.

Questions clés à se poser

  • La cohérence immédiate est-elle une exigence métier (ex: solde bancaire) ?
  • Acceptez-vous une latence de quelques centaines de ms entre écriture et disponibilité dans les vues de lecture ?
  • Avez-vous des contraintes réglementaires d’audit qui nécessitent une base de données traçable ?
  • Votre équipe maîtrise-t-elle les patterns de synchronisation DB → Kafka (Outbox, CDC) ?
  • Le volume d’écriture dépasse-t-il 10 000 transactions/seconde ? (si oui, prévoir du sharding)

Bonnes pratiques exclusives à CQRS + DB Write

  • Ne pas exposer la Write DB en lecture – toujours passer par la Read DB ou des vues matérialisées.
  • Utiliser un aggregate ID pour partitionner Kafka et garantir l’ordre des événements.
  • Limiter la taille des transactions – éviter les locks longs.
  • Mettre en place des snapshots pour les agrégats volumineux.
  • Séparer physiquement les bases de lecture (voire moteurs différents).
  • Automatiser la création des topics Kafka à partir des événements.

Anti-patterns courants

  • Transaction distribuée naïve (2PC) entre DB et Kafka – préférer l’Outbox.
  • Read DB mise à jour directement par l’API – casse le CQRS.
  • Pas de monitoring du lag – les vues de lecture deviennent obsolètes.
  • Événements trop fins – submergent Kafka.
  • Consumer unique pour toutes les Read DB – goulot d’étranglement.

  • Confluent Control Center (lag, throughput)
  • Prometheus + Grafana (métriques DB, Kafka)
  • Debezium UI pour CDC Outbox
  • OpenTelemetry pour traçage distribué
Recommandation : toujours avoir un plan de replay des événements pour reconstruire les Read DB.

Kafka-first : l'architecture événementielle pure où Kafka devient la source de vérité

Dans une architecture Kafka-first, il n’existe pas de base de données centrale d’écriture. Kafka est le cœur du système : tous les événements métier sont publiés directement dans des topics, et chaque consommateur (microservice, base de données, cache, lake) construit son propre état à partir du flux. C’est le modèle ultime de découplage et de scalabilité temps réel.

Flux typique
API Command → Kafka (topic) → Consumers multiples → DB X / DB Y / Cache / Analytics / AI

Pas de base de données centrale : chaque consommateur est autonome.

Quand l'utiliser ?
  • Ingestion massive de données IoT (smart meters, capteurs)
  • Clickstream web/mobile, logs système, tracking utilisateur
  • Analytique temps réel, dashboards, machine learning
  • Architectures où la cohérence à terme (eventual consistency) est acceptable
  • Systèmes où la scalabilité horizontale prime sur la fiabilité transactionnelle

Avantages opérationnels

  • Scalabilité extrême – jusqu’à 10M événements/seconde avec un cluster Kafka bien dimensionné.
  • Découplage total – les producteurs ne connaissent pas les consommateurs.
  • Faible latence – publication immédiate, pas d’écriture DB préalable.
  • Rejouabilité native – rejouer les événements depuis n’importe quel offset.
  • Pas de single point of failure au niveau applicatif – Kafka est clusterisé.
  • Adapté aux architectures Data Mesh – chaque domaine consomme ses sujets.

Inconvénients et risques critiques

  • Kafka devient point de vérité unique – très risqué sans gouvernance stricte.
  • Cohérence faible (eventual consistency) – chaque base peut diverger.
  • Rebuild complexe – en cas de bug dans un consumer, la donnée peut être perdue si le topic est purgé.
  • Audit difficile – pas de source unique fiable (il faut interroger tous les consumers).
  • Gouvernance des événements critique – schema evolution obligatoire (Schema Registry).
  • Requêtage limité – Kafka n’est pas une base de données (pas de requêtes ad-hoc efficaces).
  • Coût infrastructure élevé – stockage des événements sur longue durée (tiered storage coûteux).

Checklist pour implémenter Kafka-first avec succès

  • Définir une politique de rétention – combien de temps garder les événements ? (heures, jours, indéfini ?)
  • Mettre en place un Schema Registry – versioning et compatibilité.
  • Implémenter l’idempotence des producteurs (enable.idempotence=true) pour éviter les doublons.
  • Garantir l’idempotence des consommateurs – gérer les rejeux.
  • Planifier la relecture (replay) – capacité à reconstruire un consommateur depuis zéro.
  • Monitorer le lag consommateur – alerter si dépassement de seuil.
  • Sécuriser l’accès aux topics – ACLs, TLS, authentification.
  • Prévoir des tests de résilience (chaos engineering sur brokers).

Questions clés à se poser

  • L’eventual consistency est-elle acceptable pour votre domaine métier ? (oui pour analytics/IoT, non pour transactions financières)
  • Avez-vous besoin de rejouer des événements sur plusieurs mois ? (nécessite un stockage durable et coûteux)
  • Votre équipe maîtrise-t-elle les concepts Kafka (partitions, offsets, rééquilibrage) ?
  • Acceptez-vous qu’un consommateur défaillant doive se reconstruire en relisant des jours d’événements ?
  • Quel est votre RPO/RTO en cas de corruption d’un consumer ?

Bonnes pratiques exclusives à Kafka-first

  • Utiliser des clés de partition pertinentes – garantir l’ordre par agrégat.
  • Configurer la rétention en fonction des besoins de replay (pas trop longue, pas trop courte).
  • Mettre en place une stratégie de compactage pour les événements de référence (ex: dernière valeur d’un capteur).
  • Isoler les topics sensibles avec des ACLs.
  • Prévoir des consumers idempotents avec stockage des offsets externalisé (ex: base de déduplication).
  • Documenter le schéma sémantique de chaque topic.

Anti-patterns courants en Kafka-first

  • Kafka utilisé comme base de données – faire des requêtes sur les topics (inefficace).
  • Topics uniques pour tous les événements – difficile à gérer, préférer plusieurs topics par domaine.
  • Pas de gestion des doublons – les consumers doivent être idempotents.
  • Consumer unique pour plusieurs stores – crée un goulot d’étranglement.
  • Rétention infinie sans tiered storage – explosion des coûts.
  • Ignorer le rééquilibrage (rebalancing) – impact sur la latence.

Outillage recommandé pour Kafka-first

  • Schema Registry – Confluent, Apicurio
  • Stream Processing – ksqlDB, Kafka Streams, Flink
  • Monitoring – Confluent Control Center, Prometheus + JMX exporter, Burrow
  • Test de résilience – Kafka Doctor, Chaos Mesh
  • Migration / Mirroring – MirrorMaker 2, Confluent Replicator
Attention : ne jamais déployer Kafka-first sans un plan de reprise (backup des topics, réplication inter-AZ).

Conclusion : Kafka-first n’est pas universel mais puissant

L’architecture Kafka-first est idéale pour les systèmes à très haut volume où la cohérence à terme est acceptable. Elle nécessite une gouvernance rigoureuse (schémas, monitoring, rejeu). Dans les DSI critiques (finance, santé), elle est souvent combinée avec d’autres patterns (CQRS+DB, Outbox) pour les sous-domaines transactionnels. À utiliser avec discernement.

Stratégie hybride : le meilleur des deux mondes

De nombreuses DSI adoptent une architecture hybride où certains flux critiques utilisent CQRS+DB, tandis que les flux non critiques ou massifs passent par Kafka-first. Exemple typique :

Flux transactionnel (paiement) → CQRS+DB + Outbox → Kafka → Consumer facturation (DB)
Flux IoT (télémétrie) → Kafka-first (pas de DB intermédiaire) → Data Lake + Analytics

Avantage : on bénéficie de la cohérence pour le cœur de métier et de la scalabilité pour le reste. Inconvénient : deux paradigmes à maintenir.

Recommandation DSI : commencez par CQRS+DB pour le périmètre critique, puis ajoutez Kafka-first pour les flux périphériques à très haut volume.

5 questions pour départager les deux architectures

  1. Quel est votre RTO/RPO en cas de panne ? Si vous ne pouvez pas perdre une seule transaction → CQRS+DB.
  2. Acceptez-vous des lectures éventuellement obsolètes de quelques secondes ? Oui → Kafka-first.
  3. Devez-vous conserver l’historique des modifications pendant 10 ans ? CQRS+DB plus simple à auditer.
  4. Votre pic d’écriture dépasse-t-il 100k requêtes/seconde ? Alors Kafka-first s’impose.
  5. Avez-vous une équipe dédiée au streaming ? Non → préférez CQRS+DB.

REX : erreurs fréquentes lors du choix

  • Choisir Kafka-first pour une application de facturation → incohérence des soldes.
  • Choisir CQRS+DB pour du clickstream 1M events/s → surcoût DB énorme.
  • Oublier l’eventual consistency dans les spécifications métier.
  • Ne pas prévoir de mécanisme de replay dans Kafka-first.
  • Sous-estimer la complexité de l’Outbox dans CQRS+DB.

Architectures complémentaires : Event Sourcing, Outbox, CDC, Microservices synchrones

Au-delà de CQRS+DB et Kafka-first, d’autres patterns répondent à des besoins spécifiques : traçabilité absolue (Event Sourcing), fiabilité de la publication (Outbox), intégration legacy sans code (CDC), ou simplicité immédiate (Microservices synchrones). Ce guide détaille chacun pour vous aider à les combiner ou les substituer.

Astuce DSI : ces patterns ne s’excluent pas mutuellement. Une architecture réelle peut mêler CDC pour le legacy, Outbox pour la fiabilité, Event Sourcing pour l’audit, et des appels synchrones pour l’UX.

1. Event Sourcing + CQRS (Event Store central)

Principe : on ne stocke pas l’état final, mais tous les événements métier immuables. L’état est reconstruit par rejeu. Associé à CQRS, les vues de lecture sont projetées à partir de l’Event Store.

Avantages
  • Traçabilité absolue (audit complet)
  • Rejeu / time travel natif
  • Correction des erreurs par événement compensatoire
  • Parfait pour systèmes réglementés (finance, assurance)
Inconvénients
  • Complexité conceptuelle élevée
  • Versioning des événements difficile
  • Reconstruction lente sur gros volumes (snapshots nécessaires)
  • Courbe d’apprentissage raide
Stack typique
  • EventStoreDB, Axon Server, Kafka (comme event store)
  • PostgreSQL + table d’événements (pattern maison)
  • Akka Persistence, Greg Young’s Event Store
Cas d’usage : systèmes financiers (comptes, virements), assurance (sinistres), traçabilité supply chain, dossiers médicaux.
REX : une banque a réduit ses audits de 80% grâce à Event Sourcing. Leçon : prévoir des snapshots hebdomadaires pour limiter les temps de reconstruction.

2. Outbox Pattern (transaction sécurisée DB → Kafka)

Principe : dans la même transaction ACID, on écrit dans la table métier ET dans une table outbox. Un process asynchrone (polling ou CDC) publie les événements vers Kafka. Garantie "at-least-once".

Avantages
  • Cohérence garantie entre DB et Kafka
  • Pas de double écriture manuelle risquée
  • Standard éprouvé en production
  • S’ajoute facilement à une architecture existante
Inconvénients
  • Latence supplémentaire (polling)
  • Nécessite un process de relay fiable
  • Risque de doublons si relay redémarre
Stack typique
  • Spring Kafka + @Transactional outbox
  • Debezium (CDC) sur table outbox
  • MassTransit (.NET), Outbox pattern in PostgreSQL
Cas d’usage : e-commerce (commande → événement), ERP, CRM, tout système nécessitant une fiabilité absolue de diffusion.
REX : un site e-commerce a éliminé les pertes d’événements de 0.5% en passant à l’Outbox. Leçon : nettoyer régulièrement la table outbox pour éviter l’explosion.

3. CDC (Change Data Capture) – Debezium

Principe : on capture les changements directement dans les logs de la base de données (binlog MySQL, WAL PostgreSQL) et on les transforme en événements Kafka. Aucune modification applicative requise.

Avantages
  • Zéro impact sur le code applicatif
  • Idéal pour legacy ou progiciels
  • Capture même les modifications hors logique métier (scripts, admin)
  • Déploiement rapide (semaines)
Inconvénients
  • Dépendance forte au moteur de DB (binlog/WAL)
  • Charge supplémentaire sur la base (logs)
  • Peu de contrôle sémantique (événements techniques)
  • Complexité de configuration initiale
Stack typique
  • Debezium (connecteur Kafka Connect)
  • Confluent Platform / AWS MSK
  • Sources : PostgreSQL, MySQL, Oracle, SQL Server, MongoDB
Cas d’usage : migration progressive vers microservices, synchronisation d’un CRM legacy avec un data lake, alimentation d’un cache secondaire.
REX : une DSI a synchronisé 10 ans de données legacy en 2 semaines avec Debezium. Leçon : surveiller l’espace disque des logs binaires.

4. Microservices synchrones (REST / gRPC sans event bus)

Principe : architecture classique où les services communiquent directement via appels synchrones (HTTP, gRPC). Pas de bus de messages central. Chaque service possède sa base de données.

Avantages
  • Simplicité extrême (débug, logs)
  • Cohérence immédiate (réponse attendue)
  • Pas de brique supplémentaire (Kafka)
  • Idéal pour MVP, petites applications
Inconvénients
  • Fort couplage (les dépendances tombent en cascade)
  • Propagation de latence
  • Scalabilité limitée
  • Complexité des transactions distribuées (Saga difficile en synchrone)
Stack typique
  • Spring Boot / Micronaut / Quarkus (REST)
  • gRPC (haute performance)
  • API Gateway (Kong, Traefik, Spring Cloud Gateway)
Cas d’usage : prototypes, applications internes peu scalables, systèmes avec faibles volumes ( < 1000 req/s).
REX : un MVP de livraison a été développé en 2 mois avec REST sync. Passé en production avec 500 req/s, les latences ont explosé → migration vers Kafka asynchrone.

Bonnes pratiques & erreurs fréquentes dans les architectures event-driven

L’adoption de patterns event-driven (CQRS, Kafka, Event Sourcing, CDC, Outbox) comporte des pièges classiques. Ce guide recense les bonnes pratiques éprouvées et les erreurs récurrentes observées en DSI, avec des solutions concrètes pour les éviter.

Attention : une mauvaise implémentation peut entraîner des pertes de données, des incohérences ou des surcoûts massifs. Ces bonnes pratiques sont le fruit de retours terrain.

Bonnes pratiques générales

  • Source de vérité explicite – définissez clairement si c’est la DB, Kafka ou l’Event Store.
  • Immuabilité des événements – ne jamais modifier un événement publié ; créer un nouvel événement de correction.
  • Idempotence des consommateurs – tolérez les relectures Kafka (at-least-once) avec des clés de déduplication.
  • Corrélation systématique – utilisez un correlationId pour tracer un flux métier à travers les services.
  • Schema Registry – imposez l’évolution compatible (Avro, Protobuf) pour éviter les ruptures.
  • Monitoring des lags – alerte si un consommateur Kafka accuse un retard > seuil.
  • Dead Letter Queue (DLQ) – gérez les événements non consommables sans bloquer le topic.

Erreurs fréquentes (anti-patterns)

  • Confondre Kafka avec une base de données – faire des requêtes ad-hoc sur les topics (inefficace).
  • Supprimer totalement les DB métier sans stratégie de replay – perte de données définitive.
  • Ne pas gérer le versioning des événements → consommateurs cassés silencieusement.
  • Ignorer la gouvernance des consommateurs → explosion du nombre de topics, lag incontrôlé.
  • Transaction distribuée naïve (2PC) entre DB et Kafka → préférer Outbox ou CDC.
  • Consumer unique pour toutes les Read DB → goulot d’étranglement.
  • Rétention infinie sans tiered storage → coût de stockage explosif.
  • Oublier les tests de replay → incapacité à reconstruire les états après sinistre.

Bonnes pratiques spécifiques par pattern

🔹 CQRS + DB Write
  • Séparez physiquement les bases de lecture et d’écriture (voire moteurs différents).
  • Utilisez un pattern Outbox pour la publication fiable des événements.
  • Ne jamais exposer la Write DB en lecture directe.
  • Mettez en place des snapshots pour les agrégats volumineux.
🔹 Kafka-first
  • Utilisez des clés de partition pertinentes (ex: ID agrégat).
  • Configurez une rétention adaptée (quelques jours pour le temps réel, tiered storage au-delà).
  • Activez l’idempotence des producteurs (enable.idempotence=true).
  • Mettez en place un Schema Registry obligatoire.
🔹 Event Sourcing
  • Utilisez des snapshots périodiques (quotidiens ou hebdomadaires).
  • Versionnez vos événements dès la première version.
  • Prévoyez des événements compensatoires (pas de modification in-place).
  • Documentez le schéma sémantique de chaque événement.
🔹 Outbox Pattern
  • Nettoyez régulièrement la table outbox (après publication).
  • Utilisez un index sur la colonne de statut pour le polling.
  • Préférez CDC sur la table outbox pour un relay plus réactif.
🔹 CDC (Debezium)
  • Surveillez l’espace disque des logs binaires (binlog / WAL).
  • Utilisez des sujets Kafka compacts pour les événements de référence.
  • Testez l’impact sur la source DB avant production.

Erreurs réelles observées en DSI (REX)

❌ Perte d’événements faute d’Outbox
Une entreprise de e-commerce publiait des événements Kafka après un commit DB, mais sans transaction. En cas de panne après le commit et avant la publication, des commandes étaient perdues. Solution : Outbox pattern.

Un bug dans un consumer a corrompu une base de lecture. Impossible de reconstruire l’état car les événements avaient été purgés après 7 jours. Solution : rétention longue + backups des topics.

Glossaire complet des architectures event-driven

Ce glossaire répertorie les termes essentiels pour maîtriser CQRS, Kafka, Event Sourcing, CDC, Outbox, et les patterns associés. Chaque définition est illustrée d’un exemple concret et de liens avec les autres concepts.

Utilisez Ctrl+F pour trouver rapidement un terme.

A. Concepts généraux event-driven

Event-Driven Architecture (EDA)

Architecture où la communication entre composants est déclenchée par la production, la détection et la réaction à des événements. Exemple : un microservice publie “commande.créée”, d’autres réagissent.

Événement (Event)

Enregistrement immuable d’un fait significatif survenu dans le système (ex: “paiement.effectué”, “compteur.relevé”). Contient un en-tête (type, date, corrélation) et un corps de données.

Commande (Command)

Intention d’effectuer une action, généralement synchrone. Se distingue de l’événement qui décrit un fait passé. Exemple : “AnnulerCommande” (commande) vs “CommandeAnnulée” (événement).

Eventual Consistency

Cohérence à terme : après un certain délai, tous les systèmes auront vu la même donnée, mais pas immédiatement. Acceptable pour de nombreux cas (recommandations, analytics).

Strong Consistency

Cohérence forte : toute lecture d’une donnée reflète la dernière écriture. Nécessite des transactions ACID ou des consensus distribués (plus coûteux).

Idempotence

Propriété d’une opération qui peut être appliquée plusieurs fois sans changer le résultat final. Essentielle pour les consommateurs Kafka (at-least-once). Exemple : SET status=“paid” si déjà paid ne change rien.

B. CQRS & Event Sourcing

CQRS (Command Query Responsibility Segregation)

Séparation des modèles de commande (écriture) et de requête (lecture). Permet d’optimiser indépendamment les performances et la scalabilité.

Write DB (Command Model)

Base de données transactionnelle qui reçoit les commandes, garantit l’intégrité ACID et constitue la source de vérité (dans CQRS classique).

Read DB (Query Model)

Base de données optimisée pour la lecture (dénormalisée, indexée). Mise à jour asynchrone via les événements. Peut utiliser Elasticsearch, MongoDB, etc.

Event Sourcing

Pattern où l’état applicatif est dérivé d’une séquence d’événements immuables. On ne stocke que les événements, pas l’état final. Rejeu pour reconstruire.

Event Store

Base de données spécialisée dans le stockage d’événements (EventStoreDB, Axon Server, ou Kafka utilisé comme event store). Permet de rejouer l’historique.

Snapshot

Capture périodique de l’état agrégé pour éviter de rejouer tous les événements depuis l’origine. Améliore les performances de reconstruction.

Aggregate (DDD)

Groupe d’objets métier traités comme une unité pour les modifications. Dans Event Sourcing, un agrégat est reconstruit à partir de ses événements.

Eventual Consistency (rappel)

Acceptation qu’un consommateur voie une donnée avec un léger retard par rapport à l’écriture. Typique en CQRS/Event Sourcing.

C. Apache Kafka & streaming

Topic

Canal de communication nommé dans Kafka. Un topic est partitionné pour la scalabilité. Exemple : topic “orders” contenant tous les événements de commandes.

Partition

Sous-ensemble ordonné et immuable d’un topic. Chaque message a un offset unique dans sa partition. L’ordre n’est garanti qu’au sein d’une partition.

Offset

Position séquentielle d’un message dans une partition. Permet de reprendre la lecture à un point précis (replay).

Consumer Group

Groupe de consommateurs Kafka qui se partagent la lecture des partitions d’un topic. Chaque partition est lue par un seul consommateur du groupe.

Lag (consumer lag)

Différence entre l’offset du dernier message produit et l’offset du dernier message consommé. Un lag élevé indique un consommateur en retard.

Replay

Action de re-lire des événements depuis un offset antérieur, pour reconstruire un état ou corriger un consumer défaillant.

Schema Registry

Service qui gère les versions des schémas d’événements (Avro, Protobuf, JSON Schema). Assure la compatibilité et évite les ruptures.

Kafka Streams

Bibliothèque Java pour traiter des flux en temps réel directement dans Kafka (agrégations, jointures, fenêtres).

ksqlDB

Moteur de streaming SQL pour Kafka, permettant d’écrire des requêtes temps réel (SELECT, JOIN, WINDOW) sur des topics.

Tiered Storage

Stockage des anciens événements sur des supports moins coûteux (ex: S3) tout en les rendant accessibles pour le replay.

D. Patterns Outbox, CDC, Microservices synchrones

Outbox Pattern

Pattern transactionnel : écrire dans la table métier et une table “outbox” dans la même transaction. Un process publie les événements de l’outbox vers Kafka. Garantie de livraison.

CDC (Change Data Capture)

Technique qui capture les modifications d’une base de données à partir de ses journaux (binlog MySQL, WAL PostgreSQL) et les transforme en événements.

Debezium

Plateforme CDC open source basée sur Kafka Connect. Supporte PostgreSQL, MySQL, Oracle, SQL Server, MongoDB, etc.

Polling Publisher

Variante simple de l’Outbox : un process interroge périodiquement la table outbox pour publier les événements non encore envoyés.

Transactional Outbox

Autre nom de l’Outbox pattern, insistant sur le caractère transactionnel (même commit DB).

Microservices synchrones

Architecture où les services communiquent directement via des appels synchrones (REST, gRPC). Pas de bus asynchrone. Plus simple mais plus couplé.

Circuit Breaker

Pattern de résilience pour les appels synchrones : interrompt les appels vers un service défaillant pour éviter les effets cascade.

E. Gouvernance, monitoring, résilience

Dead Letter Queue (DLQ)

Topic ou file où sont placés les événements qu’un consommateur n’a pas pu traiter (erreur de parsing, métier). Permet de ne pas bloquer le flux.

Backpressure

Mécanisme pour ralentir un producteur si le consommateur est trop lent. Dans Kafka, géré par les paramètres de session et les lags.

RPO / RTO

RPO (Recovery Point Objective) : âge maximal des données perdues acceptable. RTO (Recovery Time Objective) : durée maximale d’indisponibilité.

Strangler Pattern

Stratégie de migration progressive d’un système legacy vers une nouvelle architecture, en interceptant et redirigeant petit à petit les appels.

Saga

Pattern de gestion de transactions distribuées sans 2PC. Une série d’événements et d’événements compensatoires. Exemple : Saga orchestrée par Kafka.

Event Carried State Transfer

Les événements contiennent suffisamment de données pour qu’un consommateur n’ait pas besoin d’interroger d’autres services. Réduit le couplage.

REX #1 – Banque internationale : Event Sourcing pour la traçabilité réglementaire (MIFID II)

Contexte : Une banque européenne devait tracer toutes les modifications des comptes clients et des transactions pour répondre aux exigences MIFID II. L’ancien système enregistrait uniquement l’état final, rendant les audits longs et coûteux.

Solution adoptée : Migration progressive d’un module critique (gestion des ordres de bourse) vers Event Sourcing + CQRS, avec EventStoreDB et Kafka pour la diffusion des événements.

📊 Résultats observés (après 9 mois) :
  • Temps de réponse aux audits : passé de 3 jours à 15 minutes (rejeu ciblé).
  • Perte d’événements : 0% (grâce à l’Event Store transactionnel).
  • Charge sur l’équipe : +2 ingénieurs dédiés à l’infrastructure Event Sourcing.
  • Réduction des pénalités réglementaires : 500 k€/an économisés.
⚠️ Leçons apprises :
  • La gestion du versioning des événements a été sous-estimée (mise en place d’un Schema Registry Avro).
  • Les snapshots sont indispensables : au-delà de 10 000 événements par agrégat, les reconstructions devenaient lentes.
  • Former les équipes au pattern Event Sourcing a pris 3 mois.

REX #2 – Leader e-commerce français : Outbox pattern pour fiabiliser la chaîne de commande

Contexte : Une plateforme e-commerce traitant 50 000 commandes/heure subissait des pertes d’événements (environ 0,5% des commandes n’atteignaient pas le système de facturation). Cause : double écriture manuelle (DB puis Kafka) sans transaction.

Solution adoptée : Implémentation du pattern Outbox avec table dédiée dans PostgreSQL et un relay en Java (polling toutes les 200 ms). Les événements sont publiés vers Kafka.

📊 Résultats observés (après 4 semaines) :
  • Pertes d’événements : 0,5% → 0,001% (presque nul).
  • Latence supplémentaire : +150 ms en moyenne (polling).
  • Complexité opérationnelle : légère hausse (nettoyage table outbox).
  • ROI : récupéré en 3 mois grâce à la réduction des litiges clients.
⚠️ Leçons apprises :
  • Nettoyer régulièrement la table outbox (index sur “published” + date).
  • Prévoir une surcharge de la base (écriture supplémentaire).
  • Le polling peut être remplacé par CDC pour réduire la latence.

REX #3 – Fournisseur d’énergie : Kafka-first pour les compteurs intelligents (succès et écueil)

Contexte : 3 millions de compteurs communicants, 150 000 messages/seconde en pic. L’entreprise a choisi une architecture Kafka-first pure (pas de base de données centrale).

Ce qui a bien fonctionné : Ingestion et transformation temps réel (latence < 200 ms), scalabilité horizontale sans effort.

L’écueil : Pour la facturation (besoin de cohérence forte), ils ont tenté de s’appuyer sur les événements Kafka. Plusieurs incohérences sont apparues (doublons, ordre non garanti).

📊 Résultats mitigés :
  • Partie IoT : excellent (disponibilité 99,99%, coût maîtrisé).
  • Partie facturation : 2 incidents majeurs (soldes erronés) dus à l’eventual consistency.
  • Solution finale : architecture hybride – Kafka-first pour l’IoT + CQRS+DB pour la facturation.
⚠️ Leçon clé : Ne pas utiliser Kafka-first pour des sous-domaines exigeant une cohérence forte. Un pattern hybride est préférable.

REX #4 – Opérateur télécom : Migration d’un CRM legacy grâce à CDC (Debezium)

Contexte : Un CRM Oracle vieillissant de 15 ans, difficile à modifier. Besoin d’alimenter un nouveau moteur de recommandation et un data lake en temps réel.

Solution adoptée : Déploiement de Debezium (Kafka Connect) pour capturer les changements sur les tables Oracle via les logs redo. Les événements sont poussés dans Kafka, puis consommés par les nouveaux services.

📊 Résultats :
  • Zéro modification du code legacy (risque minimisé).
  • Délai de mise en œuvre : 3 semaines (vs 6 mois estimés pour une refonte).
  • Latence de capture : 1–2 secondes (acceptable pour le cas d’usage).
  • Charge supplémentaire sur la base Oracle : +5% CPU, maîtrisée.
⚠️ Leçons apprises : Surveiller l’espace disque des logs (redo logs). Prévoir une haute disponibilité du connecteur Debezium.

REX #5 – Startup : Surcharge technique avec Event Sourcing (retour négatif)

Contexte : Une startup de gestion de tâches a voulu adopter Event Sourcing + CQRS dès le début, pensant que cela faciliterait l’évolution future.

Problèmes rencontrés : L’équipe (3 devs juniors) a passé 6 mois à lutter avec le versioning, les snapshots, la gestion des projections. Le time-to-market a été multiplié par 3.

📊 Conclusion :
  • Abandon d’Event Sourcing au profit d’un CRUD classique (REST + PostgreSQL).
  • Délai de livraison final : 4 mois après l’abandon.
  • Leçon : Event Sourcing n’est pas adapté aux applications simples ou aux petites équipes sans expertise.

REX #6 – Enseigne de grande distribution : CQRS + Outbox pour la synchronisation stock entre 500 magasins

Contexte : Un système central de gestion des stocks (monolithe) devenait incapable de supporter les pics de mise à jour (Black Friday). Décision de passer à une architecture event-driven avec CQRS.

Solution : Write DB (PostgreSQL) pour les commandes d’entrée/sortie, Outbox pattern pour publier les événements, Kafka pour la diffusion, et des Read DB régionales (Elasticsearch) pour les consultations.

📊 Résultats :
  • Latence de mise à jour des stocks entre magasins : 2,5 s → 0,9 s.
  • Disponibilité du service de consultation : 99,95% (contre 98% avant).
  • Capacité à monter jusqu’à 200 000 événements/seconde (Black Friday).
  • ROI sur 18 mois grâce à la réduction des ruptures de stock.

Guide de dimensionnement (nombre de brokers, partitions, mémoire)

  • Kafka : 3 brokers minimum (réplication factor=3), partitions = max(10, débit_moyen/10k) par topic. Mémoire heap = 6-8 Go pour des débits > 100k/s.
  • CQRS Write DB : sharding si écriture > 10k tps. Utiliser des réplicas de lecture pour décharger.
  • Outbox : taille de la table outbox à surveiller (index sur `published`). Polling interval = 100-500 ms selon latence requise.
  • CDC : nombre de workers Kafka Connect = nombre de tables capturées / 10. Surveiller les logs binaires (taille, rétention).
  • Event Sourcing : snapshot tous les 1 000 à 10 000 événements selon taille agrégat.

Outils de benchmark pour valider vos performances

  • Kafka : OpenMessaging Benchmark, Kafka Producer/Consumer Perf Test, LinkedIn’s Kafka Bench

  • CQRS/Event Sourcing : Gatling, JMeter, ou tests unitaires de charge avec containers Docker
  • Sync : wrk, vegeta, hey (HTTP)

Interprétation des résultats : courbes de charge

  • Kafka-first : débit linéaire jusqu’à saturation des brokers (disque/réseau).
  • CQRS+DB : plateau rapide sur la Write DB (verrouillage).
  • Outbox : latence augmente avec le volume si polling trop lent.
  • Event Sourcing : les projections sont le goulot.

Synthèse : choisissez en fonction de vos objectifs de performance

  • La plus faible latence → Microservices sync (mais faible débit) ou Kafka-first (bon compromis).
  • Le plus haut débit → Kafka-first (sans conteste).
  • Le meilleur rapport coût/débit à très haut volume → Kafka-first (économie d’échelle).
  • La fiabilité transactionnelle à moindre coût → CQRS+DB + Outbox.
  • La traçabilité absolue → Event Sourcing (surcoût acceptable pour la conformité).
Ces benchmarks vous permettent de dimensionner votre infrastructure et de valider vos choix par des tests sur votre propre charge.
