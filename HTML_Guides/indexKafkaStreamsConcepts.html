Fil conducteur : Pipeline complet de détection de fraude

Scénario réel : Une banque veut détecter les transactions suspectes en temps réel (ex: plus de 1000€ cumulés en moins de 2 minutes sur la même carte).

  1. Producteur : Chaque transaction (montant, carte, horodatage) est envoyée vers le topic transactions-raw avec la clé = numéroCarte.
  2. Kafka Streams – Filtrage : On ne garde que les transactions > 50€ (éliminer les micro-paiements).
  3. GroupByKey : On regroupe par carte (la clé est déjà la carte).
  4. Fenêtrage (Windowing) : Fenêtre glissante de 2 minutes avec TimeWindows.of(Duration.ofMinutes(2)).grace(Duration.ofSeconds(10)).
  5. Aggrégation : On somme les montants par carte dans la fenêtre via aggregate(...) (state store RocksDB).
  6. Filtrage final : On ne garde que les agrégats dont la somme dépasse 1000€.
  7. Production : L’alerte est envoyée dans le topic alertes-fraude avec exactly‑once semantics.
Ce pipeline utilise filter, groupBy, windowing, aggregation, state store et EOS – des sujets incontournables de l’examen CCDAK.
Extrait de code (Kafka Streams DSL) :
KStream<String, Transaction> transactions = builder.stream("transactions-raw");
transactions
    .filter((key, tx) -> tx.getMontant() > 50)   // 🔹 FILTRE
    .groupByKey()                                 // 🔹 GROUP BY
    .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(2)))
    .aggregate(
        () -> 0.0,
        (key, tx, somme) -> somme + tx.getMontant(),
        Materialized.<String, Double, WindowStore<Bytes, byte[]>>as("sommestore")
    )
    .toStream()
    .filter((windowedKey, somme) -> somme > 1000)
    .map((windowedKey, somme) -> KeyValue.pair(windowedKey.key(), new Alerte(...)))
    .to("alertes-fraude", Produced.with(Serdes.String(), new AlertSerde()));

24 concepts Kafka – Version Certification Developer

Chaque carte = analogie puissante + extrait technique CCDAK + lien avec notre fil rouge "fraude".

Topic & Partition
Classeur + sous-dossiers
📌 Analogie : Le topic est un classeur “Transactions”, ses partitions (ex: 6 sous-dossiers) stockent les relevés dans l’ordre. Kafka garantit l’ordre dans une partition.
💡 Examen : Le parallélisme max d’un consumer group = nombre de partitions. Pour le fil rouge, transactions-raw a 12 partitions pour supporter 12 consommateurs simultanés.
🔑 Astuce certif : Toujours dimensionner partitions en fonction du débit cible. Augmenter partitions n’est pas réversible sans recréer le topic.
Producer (idempotent)
Écrivain fiable
📌 Analogie : Guichetier qui enregistre les transactions. Avec enable.idempotence=true, il note un numéro unique sur chaque bordereau → pas de doublon.
💡 Examen : Pour exactly-once, il faut acks=all, retries=MAX_INT, max.in.flight=5 et idempotence.
🔑 Dans notre fraude : Les producteurs évitent les doublons lors des pics réseau.
Consumer Group
Équipe de lecteurs
📌 Analogie : 3 analystes lisent les sous-dossiers (partitions) : chacun sa partition, le travail est réparti. Si un analyste tombe malade, ses dossiers sont réattribués (rebalance).
💡 Examen : Rebalance peut stopper la consommation. Utiliser partition.assignment.strategy (Cooperative Sticky) pour réduire les rééquilibrages.
🔑 Notre Streams : Kafka Streams crée automatiquement des consumers groups pour le sous-traitement.
Broker / ISR
Étagère + réplicas synchrones
📌 Analogie : Chaque étagère (broker) stocke des copies des dossiers. ISR = les assistants qui ont reçu la dernière version.
💡 Examen : min.insync.replicas=2 avec réplication 3 → garantit qu’au moins 2 brokers ont le message avant d’accuser réception.
Leader/Follower
Chef & adjoints
📌 Analogie : Chef de rayon (leader) gère les écritures/lectures. Les adjoints (followers) copient simplement.
💡 Examen : Les followers ne servent pas les clients, uniquement la réplication. unclean.leader.election=false évite la perte de données.
Clé & Headers
Étiquette de tri
📌 Analogie : Chaque transaction a une clé = numéroCarte. Toutes les transactions d’une même carte vont dans la même partition → ordre garanti.
💡 Examen : Les headers (paires clé/valeur) servent à du routage ou métadonnées sans toucher au body.
Schema Registry
Bibliothèque des formulaires
📌 Analogie : Tous les producteurs doivent utiliser le même formulaire (Avro) pour la transaction. Le Schema Registry stocke la version du formulaire.
💡 Examen : Avro + Schema Registry = évolution de schéma sans rupture (compatibilité backward/forward).
Offset / Commit Log
Numéro de page immuable
📌 Analogie : Chaque message a un offset (numéro de page) qui ne change jamais. C’est un journal append-only.
💡 Examen : Le consommateur commit son offset dans le topic interne __consumer_offsets. Maîtriser commit automatique vs manuel.
Rétention / Compaction
Nettoyage ou archivage
📌 Analogie : Delete = jeter les fiches après 7 jours. Compaction = ne garder que la dernière valeur par clé.
💡 Examen : log.cleanup.policy=compact pour les tables de référence (KTable).
Exactly-Once (EOS)
Une et une seule fois
📌 Analogie : Transfert bancaire parfait : ni perte, ni double. Kafka combine idempotence + transactions atomiques.
💡 Examen : processing.guarantee=exactly_once_v2 dans Kafka Streams. Les transactions permettent de committer offsets et messages produits ensemble.
🔑 Dans la fraude : L’alerte est générée exactement une fois même en cas de panne.
KStream / KTable
Flux vs Photo
📌 Analogie : KStream = flux continu d’événements (transactions). KTable = un instantané mutable (ex: solde courant).
💡 Examen : KTable est un changelog compacté. La jointure KStream-KTable est très courante.
State Store
Mémoire persistante locale
📌 Analogie : Un tableau blanc sur lequel chaque analyste note les compteurs intermédiaires (somme par carte).
💡 Examen : State stores sont basés sur RocksDB et sauvegardés dans un topic de changelog. Essentiel pour les opérations avec état.
🔑 Dans le fil rouge : Stockage de la somme des montants par carte sur la fenêtre courante.
Windowing
Découpage temporel
📌 Analogie : Détection de fraude : on regarde les 2 dernières minutes (fenêtre glissante) pour chaque carte.
💡 Examen : Tumbling, Hopping, Session windows. grace() gère les événements en retard.
🔑 Code : windowedBy(TimeWindows.of(Duration.ofMinutes(2))).
Joins & co-partitioning
Fusion de flux
📌 Analogie : Pour joindre deux classeurs, ils doivent avoir le même nombre de sous-dossiers et la même répartition.
💡 Examen : Co-partitioning = même nombre de partitions + même partitionneur. Kafka Streams peut faire un repartition automatique (coût).
Interactive Queries
Interroger l’état
📌 Analogie : Dashboard qui affiche en direct le compteur de fraude par carte en allant chercher l’info dans chaque state store distribué.
💡 Examen : Exposer une API REST pour interroger l’état d’une application Streams (KTable, fenêtres).
Processor API
Low-level contrôle
📌 Analogie : DSL = assemblage de Lego prêts à l’emploi. Processor API = pièces brutes pour construire des traitements sur-mesure.
💡 Examen : Utiliser Processor API pour une gestion fine des événements ou horloge événementielle.
Transactions
Atomicité read-process-write
📌 Analogie : Un chèque de banque : soit tout est validé (consommation + production + commit offset), soit rien.
💡 Examen : initTransaction(), beginTransaction(), sendOffsetsToTransaction().
Kafka Connect
Adaptateur universel
📌 Analogie : Passerelle qui importe automatiquement les données d’une base Oracle vers Kafka (Debezium).
💡 Examen : Connecteurs JDBC, S3, Elastic. Architecture workers distribués, transformations simples (SMT).
KRaft
Pas de ZooKeeper
📌 Analogie : Avant : un secrétaire externe (ZK) gérait les métadonnées. Maintenant : le cluster gère lui-même ses métadonnées (Raft).
💡 Examen : Kafka 3.x+ sans ZooKeeper. KRaft supporte plus de 200k partitions.
Monitoring / Lag
Retard de lecture
📌 Analogie : Le lag = nombre de pages non encore lues par l’analyste. S’il prend du retard, l’alerte fraude risque d’arriver trop tard.
💡 Examen : kafka-consumer-groups --describe. Métriques JMX.
Sécurité & Quotas
Contrôle d’accès
📌 Analogie : Badge d’accès : seuls certains employés peuvent écrire dans le topic transactions. Quotas limitent le débit.
💡 Examen : ACL, SASL/SCRAM, mTLS. Quotas de production/consommation.
Filter (Prédicat)
Tamis
📌 Analogie : Un tamis qui ne laisse passer que les grains plus gros qu’un seuil. En Streams, on ne garde que les événements satisfaisant une condition.
💡 Examen : filter() et filterNot() sont des opérations sans état (stateless). Elles ne nécessitent pas de state store.
🔑 Dans notre fraude : On filtre d’abord les transactions > 50€ pour éliminer les micropaiements avant agrégation.
💻 Code : .filter((key, tx) -> tx.getMontant() > 50).
GroupBy / GroupByKey
Rassemblement par étiquette
📌 Analogie : Toutes les fiches portant la même couleur (clé) sont mises dans le même paquet avant de les compter ou de les additionner.
💡 Examen : groupByKey() utilise la clé existante du message. groupBy() permet de changer la clé (provoque un repartition). L’opération est avec état et nécessite un groupé avant une agrégation.
🔑 Notre pipeline : On groupe par carte (clé) avant d’appliquer la fenêtre et la somme.
Aggregations
Additionneur automatique
📌 Analogie : Un comptable qui tient un registre des sommes par catégorie. Chaque nouvel événement met à jour le total (state store).
💡 Examen : count() (nombre d’événements), reduce() (combine deux valeurs), aggregate() (général). Toutes sont stateful et utilisent RocksDB.
🔑 Dans la fraude : aggregate( () -> 0.0, (key, tx, sum) -> sum + tx.getMontant() ) calcule la somme par carte sur la fenêtre.
💡 Conseil certif : Attention à la sérialisation des objets d’état ; ils doivent être immutables ou bien gérés.

Domaines officiels couverts par notre guide
  • Kafka Core : producers, consumers, brokers, réplication, ISR, offsets
  • Kafka Streams : KStream/KTable, windowing, state stores, exactly-once, joins
  • Opérations Streams : filter, groupBy, aggregation, reduce, count
  • Schema Registry & Avro : évolution de schéma, sérialisation
  • Kafka Connect : sources, sinks, SMT
  • Administration & sécurité : KRaft, ACL, quotas, monitoring lag
Nos conseils pour décrocher l'examen (version francophone)
  • Pratiquez avec Docker : montez un cluster KRaft + Kafka Streams en Java, implémentez le pipeline de détection de fraude ci-dessus.
  • Maîtrisez les CLI : kafka-topics, kafka-consumer-groups, kafka-producer-perf-test.
  • Comprenez les configurations critiques : acks, min.insync.replicas, enable.idempotence, processing.guarantee.
  • Révisez les pièges : rebalance, ordre des messages, co-partitioning, état global vs local, gestion des fenêtres et grace().
  • Ressources recommandées : documentation officielle Confluent, Kafka: The Definitive Guide, et les practice exams (Udemy, ExamTopics).
Mémo final pour l’exam : “Un topic partitionné, un consumer group lié, exactly-once avec transaction, state store pour mémoire, interactive queries pour briller.
Et surtout : filter, groupBy, aggregate sont les trois piliers des transformations stateful/stateless.”
