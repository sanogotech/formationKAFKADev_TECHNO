Fil conducteur : Détection de fraude en temps réel avec ksqlDB

Contexte : Une banque reçoit des transactions bancaires sur un topic Kafka transactions-raw. Chaque transaction contient : numéro de carte (clé), montant, timestamp. Objectif : détecter les cartes qui dépassent 1000€ cumulés sur une fenêtre glissante de 2 minutes, et envoyer une alerte.

Solution : Utiliser ksqlDB pour créer un pipeline de streaming SQL, sans écrire une ligne de Java.

Étape 1 – Créer un STREAM à partir du topic Kafka

On déclare un stream nommé transactions_raw qui lit le topic transactions-raw au format JSON. La clé du message est id_carte.

CREATE STREAM transactions_raw (
    id_carte VARCHAR KEY,
    montant DOUBLE,
    timestamp BIGINT
) WITH (
    KAFKA_TOPIC = 'transactions-raw',
    VALUE_FORMAT = 'JSON'
);

La commande ne produit pas encore de données ; elle définit la structure.

Étape 2 – Filtrer les transactions (garder > 50€)

On crée un nouveau stream transactions_filtrees qui ne contient que les transactions supérieures à 50€. C’est une transformation stateless.

CREATE STREAM transactions_filtrees AS
    SELECT id_carte, montant, timestamp
    FROM transactions_raw
    WHERE montant > 50
    EMIT CHANGES;

EMIT CHANGES signifie que le résultat est mis à jour en continu.

Étape 3 – Fenêtre glissante (tumbling) et agrégation par carte

On agrège les transactions par carte sur des fenêtres fixes de 2 minutes (TUMBLING). On calcule la somme des montants. Le résultat est une TABLE (car GROUP BY).

CREATE TABLE alertes_fraude AS
    SELECT id_carte,
           SUM(montant) AS somme,
           WINDOWSTART AS debut_fenetre,
           WINDOWEND AS fin_fenetre
    FROM transactions_filtrees
    WINDOW TUMBLING (SIZE 2 MINUTES)
    GROUP BY id_carte
    HAVING SUM(montant) > 1000
    EMIT CHANGES;

Chaque fois qu’une nouvelle transaction fait passer le seuil, la ligne de la table est mise à jour et une alerte est émise.

Étape 4 – Interroger les alertes en direct (Push Query)

Pour voir les alertes en temps réel, on exécute une requête push :

SELECT id_carte, somme, debut_fenetre
FROM alertes_fraude
WHERE somme > 1000
EMIT CHANGES;

Cette requête reste active et affiche chaque nouvelle alerte dès qu’elle est produite.

Étape 5 – Interrogation ponctuelle (Pull Query)

Pour connaître l’état actuel d’une carte spécifique (ex: API REST), on utilise une pull query :

SELECT id_carte, somme, debut_fenetre
FROM alertes_fraude
WHERE id_carte = '1234-5678';

Cette requête retourne une seule ligne, instantanément, sans attendre de nouveaux événements.

Résultat : Un pipeline complet de détection de fraude, entièrement déclaratif, sans code, avec exactement-once semantics, fenêtrage, agrégations, et push/pull queries.

18 concepts ksqlDB – Version Certification

Chaque carte = analogie + exemple SQL + lien avec notre fil rouge "fraude".

ksqlDB Server
Moteur SQL continu
📌 Analogie : Un serveur de base de données, mais spécialisé dans les flux.
💡 Examen : Mode interactif (CLI) ou headless (application).
STREAM
Flux d'événements
📌 Analogie : Une rivière continue de transactions.
💡 Examen : CREATE STREAM lit un topic. Données immuables.
🔑 Ici : transactions_raw est un stream.
TABLE
État mutable
📌 Analogie : Un carnet de comptes par carte.
💡 Examen : Issue d’un GROUP BY ou d’un topic compacté.
🔑 alertes_fraude est une table.
CREATE STREAM AS SELECT
Transformation persistante
📌 Analogie : Tuyau qui filtre l’eau.
💡 Examen : Matérialise la requête dans un topic.
🔑 transactions_filtrees est un CSAS.
CREATE TABLE AS SELECT
Agrégation continue
📌 Analogie : Tableau de bord auto‑mis à jour.
💡 Examen : GROUP BY + fenêtre → table.
🔑 alertes_fraude est un CTAS.
Push Query
Alerte en continu
📌 Analogie : Sonnette à chaque transaction.
💡 Examen : SELECT ... EMIT CHANGES. S’exécute pour toujours.
Pull Query
Interrogation ponctuelle
📌 Analogie : Coup de fil pour connaître l’état actuel.
💡 Examen : SELECT ... FROM table WHERE clé = ... (sans EMIT). Très rapide.
WINDOW
Découpage temporel
📌 Analogie : Fenêtre glissante de 2 minutes.
💡 Examen : TUMBLING, HOPPING, SESSION. Grace period.
🔑 Notre requête : TUMBLING (SIZE 2 MINUTES).
GROUP BY / HAVING
Regroupement & filtre post‑agg
📌 Analogie : On regroupe par carte, puis on filtre les groupes > 1000€.
💡 Examen : HAVING s’applique après l’agrégation.
Aggregations
Calculs cumulatifs
📌 Analogie : Calculette qui additionne.
💡 Examen : SUM, COUNT, AVG, MIN, MAX, TOPKDISTINCT.
PARTITION BY
Changement de clé
📌 Analogie : On réétiquette les fiches pour les répartir différemment.
💡 Examen : Provoque un repartition (topic intermédiaire).
WITH clause
Paramètres du connecteur
📌 Analogie : Fiche technique du topic.
💡 Examen : KAFKA_TOPIC, VALUE_FORMAT, KEY_FORMAT, etc.
EMIT CHANGES / FINAL
Émission continue ou finale
📌 Analogie : CHANGES = annonce en direct ; FINAL = attend la fin de la fenêtre.
💡 Examen : Par défaut EMIT CHANGES pour les agrégations windowées.
Vues matérialisées
Cache toujours à jour
📌 Analogie : Tableau blanc auto‑réécrit.
💡 Examen : Les tables ksqlDB sont des vues matérialisées (RocksDB).
INSERT INTO
Alimenter manuellement
📌 Analogie : Ajouter une fiche à la main.
💡 Examen : Permet d’insérer des données de test.
JOIN
Fusion de flux
📌 Analogie : Enrichir une transaction avec les infos client.
💡 Examen : Stream-Stream (windowée), Stream-Table, Table-Table. Nécessite co‑partitioning.
UDF / UDAF
Fonctions personnalisées
📌 Analogie : Nouvelle formule de calcul.
💡 Examen : Écrites en Java, déployées dans le classpath.
CLI / REST API
Console et web service
📌 Analogie : Tableau de bord et télécommande.
💡 Examen : ksql CLI, port 8088 pour REST.

Stratégie gagnante : Certification Confluent (ksqlDB)

Domaines ksqlDB couverts par l’examen CCDAK
  • Création de STREAM et TABLE
  • Push vs Pull queries
  • Fenêtrage (Tumbling, Hopping, Session)
  • Agrégations et GROUP BY / HAVING
  • Jointures (Stream-Table notamment)
  • PARTITION BY et repartition
  • Formats de données (JSON, Avro, Schema Registry)
  • EMIT CHANGES vs EMIT FINAL
Conseils pour réussir l’examen (partie ksqlDB)
  • Pratiquez avec Docker : utilisez l’image confluentinc/ksqldb-server et la CLI ksql.
  • Maîtrisez la différence entre STREAM et TABLE : un stream = historique immuable ; une table = dernier état par clé.
  • Sachez que les fenêtres sont obligatoires pour les agrégations sur les streams (sauf GROUP BY sur une table).
  • Comprenez quand PARTITION BY déclenche un repartition (nécessite un topic intermédiaire).
  • Testez les requêtes avec EXPLAIN pour analyser le plan d’exécution.
  • N’oubliez pas EMIT CHANGES pour les push queries, et son absence pour les pull queries.
Mémo final ksqlDB : “STREAM pour les événements, TABLE pour l’état, PUSH pour l’alerte, PULL pour l’API, WINDOW pour le temps, GROUP BY pour l’agrégation.”
