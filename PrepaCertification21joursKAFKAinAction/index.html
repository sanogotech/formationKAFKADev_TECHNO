Security, Delivery Guarantees & Hierarchy
📖 Definitions:
• Secured Kafka cluster: SSL/TLS encryption, SASL authentication (PLAIN, SCRAM, Kerberos), ACLs.
• Delivery guarantees: At-most-once (fire & forget, possible loss), At-least-once (retries + acks=all, duplicates possible), Exactly-once (idempotent producer + transactions).
• Resource hierarchy: Cluster → Broker → Topic → Partition → Record (offset).
🔑 Key Questions: How to connect to SSL/SASL cluster? (set security.protocol, ssl.truststore.location, sasl.mechanism). Which guarantee minimizes duplicates? Exactly‑once requires transactional.id.
Nuances: Idempotent producer (enable.idempotence=true) guarantees no duplicates per partition sequence. Exactly‑once across partitions needs transactions. Acks=all + min.insync.replicas=2 ensures strong durability.
✅ Best Practices: Always secure production clusters with TLS & SASL; use idempotence for critical pipelines; understand trade-offs: exactly‑once adds latency. Define resource hierarchy for monitoring quotas.
Consumer Groups, Partitions & Assign/Subscribe
📌 Core: Group coordinates partition assignment (rebalance). Each partition assigned to exactly one consumer per group. Assign() : manual partition assignment, no group management. Subscribe() : dynamic rebalancing, group coordination.
Exam Q: Difference? assign() avoids rebalance but lacks fault tolerance; subscribe() supports automatic failover. When to use assign? For stateless deterministic mapping.
⚙️ Nuances: Rebalance triggers revoke/assign callbacks; cooperative rebalancing (partition.assignment.strategy=CooperativeStickyAssignor) reduces stop-the-world. Max.poll.interval.ms impacts liveliness.
💡 Best practice: Use subscribe() for dynamic scaling, assign() for custom sink consumers. Always handle rebalance listeners to commit offsets gracefully.
Topic Configs & Offset Management
⚙️ Topic configs: retention.ms, segment.bytes, cleanup.policy (delete/compact), min.insync.replicas, unclean.leader.election.enable. Consumer offsets: stored in __consumer_offsets topic; auto.offset.reset (latest/earliest/none); commitSync vs commitAsync.
Key question: How to reset consumer group offset? Use kafka-consumer-groups --reset-offsets --to-datetime / --to-earliest. Offsets committed after processing to avoid reprocessing.
⚠️ Nuances: enable.auto.commit true causes at-least-once; manual commit gives control. Topic min.insync.replicas=2 with acks=all prevents data loss. Avoid frequent partition increases.
📌 Practical: Monitor lag using kafka-consumer-groups; set retention.ms based on compliance; compacted topics for keyed state.
Partitioner & Request Lifecycle
🔹 Default partitioner (sticky): If key is null → sticky batch partition (fill one partition before moving). If key present → hash(key) % numPartitions. Request lifecycle: 1) Serialize key/value 2) Interceptor 3) Partitioner determines partition 4) Buffer in RecordAccumulator 5) Sender thread builds ProduceRequest 6) Send to leader broker 7) Acks wait 8) Callback.
🧠 Exam focus: Why sticky partitioner? Improves batching and throughput. Custom partitioner can implement Partitioner interface. Request timeout handling: request.timeout.ms, retries.
Nuances: The metadata fetch may block if partitions unknown. Leader epoch helps consistency.
✅ Best practices: For even distribution, ensure keys have sufficient cardinality; tune linger.ms and batch.size for latency/throughput trade-off.
Configs, Serialization & Deserialization
Producer configs: acks, compression.type, max.in.flight.requests.per.connection, enable.idempotence. Consumer configs: fetch.min.bytes, max.poll.records, session.timeout.ms. SerDes: String, Avro, JSON, Protobuf (Schema Registry). consume(): poll loop, deserialize keys/values.
❓ Key Q: How to serialise custom objects? Implement Serializer/Deserializer or use Avro with Confluent Schema Registry. Importance of schema evolution compatibility.
⚠️ Nuances: Deserialization errors can bring down consumer: use error handling (Recoverable). Consumer subscribe + poll handles group coordination automatically.
📘 Best practice: Use Avro + Schema Registry for evolvable contracts; configure idempotent producer to avoid duplicates; tune fetch.max.bytes for large messages.
Transactions, Headers, Key Design
🔐 Transactions: atomic writes across partitions & topics: initTransactions(), beginTransaction(), sendOffsetsToTransaction(), commitTransaction(). Idempotent producer is required. Headers: optional metadata (routing, tracing). Key strategies: keys determine partition; good key cardinality for load balancing; null keys use sticky partitioner.
Design keys: high cardinality keys (user_id) avoid hot partitions. Headers: add contentType, messageId for filtering; use Compacted topics with keys for log compaction.
🧩 Exactly-once semantics in consumer: combine idempotent producer + transactional reads with isolation.level=read_committed.
✔️ Best practice: transactional API for critical banking/orders; keys for stateful streams; design headers to propagate context (no size limit but keep small).
Kafka Streams: Stateful/Stateless & Exactly‑once
📊 Kafka Streams vs Consumer/Producer: built-in exactly‑once, state stores, event-time processing, continuous topology. No external cluster needed. Stateless ops: filter, map, foreach, branch. Stateful ops: aggregations, joins, windowing, count, reduce. Exactly‑once in KStreams: processing.guarantee=exactly_once_v2 (uses transactions & idempotence).
🎯 Exam nuance: Stateful requires state store (RocksDB). Exactly‑once ensures each record processed once despite failures.
Nuance: exactly_once_v2 reduces performance overhead compared to v1. Standby replicas for high availability. Kafka Streams uses consumer group for scaling.
✅ Best practice: Use stateful for windowed aggregates, configure state dir, and enable exactly-once for idempotent pipelines.
KStream vs KTable & DSL/Processor API
🔁 KStream: full history, append-only stream. KTable: changelog, latest value per key (upsert). GlobalKTable: replicated on all tasks. DSL: high-level, built-in operators (join, aggregate). Processor API: fine-grained control, custom processors, punctuations, access to state stores.
📌 Key Q: When to choose DSL vs Processor? DSL for 90% scenarios; Processor API for custom logic, low-level timers, or custom join windows.
⚡ Stream-Table duality: a table over a stream of changes; KTable materialized view. Joining stream with table for enrichment is powerful.
💡 Best practice: Prefer DSL for readability & built-in optimizations; mix Processor API via .process() for complex event processing.
Kafka Connect (Source/Sink, SMT, Workers, Offsets)
🔌 Connectors: source (import) / sink (export). Worker properties: offset.storage.topic, config.storage.topic, rest.port. Scalability: distributed mode, tasks run in parallel; add workers for horizontal scaling. Connect offsets: offsets stored in Kafka topics (source offsets). SMT: single message transforms (e.g., InsertField, RegexRouter, MaskField). REST interface: GET /connectors, POST /connectors, PUT /connectors/name/config.
Exam: How to scale connectors? Increase tasks.max, then add worker nodes. SMT applied per record, lightweight. Connect offsets enable resume from failure.
🧠 Nuances: Source connectors require manual offset resetting? Use REST to restart tasks. Dead Letter Queue for error handling.
✅ Best practices: Use exactly-once source connectors (e.g., JDBC with incrementing) + idempotent writes. Place SMTs before sink to filter/transform.
Testing, CLI, Scaling & Deployment
🧪 Testing tools: Embedded Kafka (kafka-clients test), MockProducer/MockConsumer, Testcontainers, TopologyTestDriver for Kafka Streams. CLI tools: kafka-console-producer, kafka-console-consumer, kafka-consumer-groups, kafka-topics, kafka-dump-log. Scaling: Horizontal (more consumer instances/Streams threads/Connect workers), Vertical (bigger heap, CPU). Deployment frameworks: Kubernetes (Strimzi), Docker Compose, Confluent Platform, AWS MSK, Confluent Cloud.
🔧 Key: how to test exactly-once? Use idempotent producer + transactional consumer test. Horizontal scaling: increase partitions for parallelism.
📐 Nuance: more consumers than partitions idle; Kafka Streams max tasks = partition count. Vertical scaling helps with state store performance.
🚀 Best practice: Test with TopologyTestDriver for Streams, integration test with EmbeddedKafka. Deploy with Helm charts for observability.
Observability, Lag, Metrics & Alerting
📈 Lag: difference between producer offset and consumer offset. Monitor via kafka-consumer-groups or JMX. Performance metrics: request latency (produce/fetch), bytes-in/out, request queue size, ISR shrinks. Log4j: configure log4j.properties for kafka clients – set ERROR/WARN. Common errors: NotLeaderForPartitionException, TimeoutException, OffsetOutOfRangeException, RebalanceException. Alerts: set lag > threshold, under-replicated partitions, broker down.
🚨 Exam Q: How to config logging in Kafka apps? Use log4j, SLF4J binding; enable debug only for dev. Critical lag detection: use metric consumer.lag.
📊 Nuance: consumer lag is per partition; high lag may require increasing consumers or tuning fetch. Metrics exposed via JMX (Prometheus JMX exporter).
🔔 Best practice: alert on consumer group lag > 10k messages for 5 mins; monitor under-replicated partitions; use Confluent Control Center or Datadog.
Data Formats, Error Handling & Topic Evolution
📄 Common data formats: Avro (schema evolution), JSON (flexible but no schema), Protobuf (efficient). Error handling: retries, callback for producer, Dead Letter Queue for consumer (deserialization errors), exception handlers. Modifying topics: Increase partitions (never decrease), change config via kafka-configs, dynamic updates; avoid changing key semantics. Partitioning requirements: based on throughput & ordering; keys guarantee order per partition.
💡 Key nuance: decreasing partitions not allowed. Best practice: plan partitions for peak throughput (1 partition = ~10 MB/s). Adding partitions can break key ordering if keyed.
⚠️ Error handling best practices: implement ProducerInterceptor for audit; Consumer with recoverable error skip to DLQ using external topic; use Spring Kafka error handlers.
✅ Final best practices: always use schema registry with Avro, plan partition count with future growth, test modifying topic configs in staging, implement idempotency for critical streams, and monitor offsets.
