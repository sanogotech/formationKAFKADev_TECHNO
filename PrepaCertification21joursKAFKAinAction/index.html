Security, Delivery Guarantees & Resource Hierarchy – Deep Dive
🔐 Pourquoi ce module est fondamental ?
La sécurité (authentification, chiffrement) et les garanties de livraison (acks, idempotence, transactions) sont des sujets incontournables de l’examen CCDAK. Près de 30% des questions du domaine « Fundamentals » portent sur ces concepts. La hiérarchie des ressources (cluster → broker → topic → partition → record) est la base de toute réflexion sur les ACLs et le partitionnement.
🔒 Connexion à un cluster Kafka sécurisé
Authentification (SASL)
  • PLAIN : nom d’utilisateur/mot de passe en clair (simple mais nécessite TLS).
  • SCRAM-SHA-256/512 : stockage des identifiants hachés dans ZooKeeper/KRaft, plus sûr.
  • OAUTHBEARER : tokens JWT, idéal pour les environnements cloud (OAuth2).
  • GSSAPI (Kerberos) : héritage des infrastructures big data.
Configuration producteur :
security.protocol=SASL_SSL
sasl.mechanism=SCRAM-SHA-512
sasl.jaas.config=org.apache.kafka.common.security.scram.ScramLoginModule required username="alice" password="alice-secret";
Autorisation (ACLs)
  • Ressources : Topic, Group, Cluster, TransactionalId.
  • Opérations : Read, Write, Describe, Create, Delete, Alter, DescribeConfigs, AlterConfigs.
  • Patterns : LITERAL (exact) ou PREFIXED (ex: my-topic-*).
  • Exemple : kafka-acls --add --allow-principal User:app1 --operation Read --topic my-topic --group my-group
Nuances importantes sur la sécurité :
  • mTLS (Mutual TLS) : Le client et le broker s’authentifient mutuellement via certificats. Utilisé souvent avec security.protocol=SSL (sans SASL).
  • Super users : Comptes contournant les ACLs (ex: super.users=User:admin dans server.properties).
  • Audit logs : Kafka n’a pas de logs d’audit natifs. Utiliser des plugins ou Kafka-native avec les logs des autorisations (authorizer.log).
  • Si vous utilisez SSL sans SASL, l’identité du client est l’objet distinct (CN).

Exemple détaillé – Idempotent Producer 
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("enable.idempotence", "true");   // active l’idempotence (acks=all implicite)
props.put("acks", "all");                  // optionnel mais recommandé
props.put("max.in.flight.requests.per.connection", "5"); // avec idempotence, ≤5 ok
KafkaProducer producer = new KafkaProducer<>(props);
// Les séquences sont numérotées par le producteur, le broker rejette les doublons
💡 L’idempotence repose sur un PID (Producer ID) et des numéros de séquence. Le broker maintient le dernier numéro reçu par partition.
Nuance cruciale pour l’examen : Exactly‑once sur plusieurs partitions nécessite obligatoirement les transactions. L’idempotence seule ne garantit pas l’atomicité entre partitions. Par ailleurs, processing.guarantee=exactly_once_v2 dans Kafka Streams utilise un mécanisme de transactions optimisé (plus performant que v1).
🏛️ Resource Hierarchy – De cluster à record
📌 Niveaux hiérarchiques :
  • Cluster : ensemble de brokers (un ou plusieurs). Identifié par un cluster.id (KRaft) ou généré par ZooKeeper.
  • Broker : serveur Kafka individuel (identifié par un id entier). Chaque broker héberge des réplicas de partitions.
  • Topic : flux logique de messages (ex: “orders”). Configuration : partitions, réplication factor.
  • Partition : unité de parallélisme, ordre garanti. Chaque partition a un leader et des followers.
  • Record (message) : unité de données avec clé, valeur, timestamp, headers, offset.
Utilité pour les ACLs :
Les ACLs peuvent s’appliquer à chaque niveau. Exemple : autoriser un utilisateur à écrire sur le topic “payments” mais pas sur “admin”. Au niveau cluster : autoriser la création de topics (Create).
kafka-acls --add --allow-principal User:app --operation Create --cluster
Questions typiques de l’examen :
  • « Un consommateur peut-il lire à partir d’un topic sans ACL de lecture sur le cluster ? » → Oui, car l’ACL se fait au niveau topic. L’ACL cluster est nécessaire pour les opérations comme Describe sur le cluster.
  • « Quelle ressource est la plus petite unité ayant un ordre garanti ? » → Partition (l’ordre n’est pas garanti entre partitions d’un même topic).
  • « Si un broker meurt, le leader d’une partition peut-il être transféré ? » → Oui, un follower devient leader (gestion des ISR).
⚙️ Comportement des accusés de réception (acks)
acks=0
Le producteur ne reçoit aucun accusé. Latence minimale, risque de perte maximal. Utilisable pour métriques.
acks=1
Le leader écrit le message dans son log local et répond. Si le leader crash avant que les followers ne le répliquent → perte possible.
acks=all (ou -1)
Tous les ISR (in-sync replicas) doivent avoir écrit le message. Indispensable pour la durabilité. Combine avec min.insync.replicas (ex: 2) pour garantir qu’au moins 2 réplicas aient le message.
Bonnes pratiques de durabilité :
  • acks=all + min.insync.replicas=2 + facteur de réplication ≥3 protège contre la perte de données même si un broker tombe.
  • Activez unclean.leader.election.enable=false pour éviter qu’un réplica hors ISR devienne leader (empêche la perte de données).
  • Idempotent producer + acks=all garantit exactly-once par partition.
Erreurs liées à la livraison :
  • TimeoutException : le leader n’a pas accusé dans request.timeout.ms. Solution : augmenter le timeout, vérifier la charge du broker.
  • NotEnoughReplicasException : min.insync.replicas n’est pas atteint (trop de followers en retard).
  • DuplicateSequenceException : tentative de réécriture d’une séquence déjà utilisée (si idempotence activée).

Consumer Groups, Partition Assignment, and assign() vs subscribe() – Complete Guide
🎯 Pourquoi ce module est central pour l’examen CCDAK ?
La compréhension des consumer groups (rééquilibrage, assignation des partitions) et la différence entre assign() et subscribe() est l’un des sujets les plus testés. Environ 25% des questions du domaine « Fundamentals » portent sur la gestion des groupes, les stratégies de partitionnement et la reprise après sinistre. Maîtriser ces concepts est essentiel pour concevoir des applications résilientes.
👥 Consumer groups : architecture interne
Rôle du coordinateur de groupe
  • Un des brokers est élu Group Coordinator pour chaque groupe (basé sur le hash du nom du groupe).
  • Gère les membres du groupe, les heartbeats, et lance les rééquilibrages.
  • Stocke les métadonnées du groupe (membres, partition assignments) dans la partition interne __consumer_offsets.
  • Paramètres : group.initial.rebalance.delay.ms (délai avant de lancer un rebalance), session.timeout.ms (heartbeat maximal).
Cycle de vie d’un consommateur
  • JoinGroup : le consommateur s’enregistre auprès du coordinateur.
  • SyncGroup : le leader du groupe (premier membre) calcule l’assignation, tous les membres reçoivent leur affectation.
  • Heartbeat : maintenir l’appartenance (par défaut toutes les 3 secondes).
  • LeaveGroup : départ volontaire (ex: fermeture propre).
  • Rebalance : déclenché par arrivée/départ de membre, ou changement de métadonnées de topic (nouvelle partition).
Nuance importante : Un consommateur peut être considéré comme mort s’il n’envoie pas de heartbeat pendant session.timeout.ms (défaut 45s). Le rééquilibrage peut prendre jusqu’à max.poll.interval.ms (défaut 5min) si le traitement des messages est long. Utilisez pause()/resume() ou diminuez max.poll.records pour éviter les rebalances intempestifs.

⚔️ assign() vs subscribe() – Tableau de différences
assign() – Manuel
  • Le consommateur n’appartient à aucun groupe.
  • Vous spécifiez exactement les TopicPartition à lire : consumer.assign(Arrays.asList(new TopicPartition("topic", 0), new TopicPartition("topic", 1))).
  • Aucun rééquilibrage automatique (si partition ajoutée, vous devez le faire manuellement).
  • Pas de coordination entre consommateurs → pas de failover automatique.
  • Utile pour les lecteurs « stateless » ou les migrations.
subscribe() – Dynamique (groupe)
  • Le consommateur rejoint un groupe via group.id.
  • Indique les topics (ou pattern regex) : consumer.subscribe(Arrays.asList("topic1", "topic2")).
  • Le groupe se rééquilibre automatiquement : ajout/suppression de consommateurs, ajout de partitions.
  • Si un consommateur meurt, ses partitions sont réassignées à d’autres membres.
  • Permet de paralléliser la charge de manière horizontale.
Questions types que vous verrez à l’examen :
« Un développeur utilise assign() pour lire un sujet ayant 4 partitions, puis lance 2 instances de son application. Que se passe-t-il si l’une des instances s’arrête ? » → Les partitions ne sont pas réassignées, la consommation s’arrête pour les partitions de l’instance morte. C’est l’inconvénient majeur par rapport à subscribe().
« Quelle méthode permet de bénéficier de l’équilibrage de charge automatique ? » → subscribe() uniquement.
💾 Gestion des offsets dans un consumer group
Commit automatique (enable.auto.commit=true)
- Interval : auto.commit.interval.ms (défaut 5s).
- Risque : message traité mais non commité avant le rebalance → relecture (au moins une fois).
- Simple mais déconseillé pour les traitements critiques.
Commit manuel synchrone & asynchrone 
// synchrone (bloquant)
consumer.commitSync();
// asynchrone avec callback
consumer.commitAsync((offsets, exception) -> {
    if(exception != null) log.error("Commit failed", exception);
});
💡 Utilisez commitSync() avant de fermer le consommateur, et commitAsync() pendant le traitement normal pour la performance.
Nuance technique : L’offset est stocké pour chaque partition sous la forme d’un entier (prochain message à lire). Les offsets sont stockés dans la topic __consumer_offsets (nombre de partitions = offsets.topic.num.partitions, défaut 50). 📌 Offsets : le pointeur de lecture du consommateur
Qu’est-ce qu’un offset ?
Un numéro séquentiel attribué à chaque message à l’intérieur d’une partition. Le premier message a l’offset 0 (ou parfois 0). Le consommateur enregistre quel offset a été traité pour chaque partition. Si aucun offset n’est commis, auto.offset.reset prend le relais.
Topic interne __consumer_offsets
Stocke les offsets des consumer groups. Nombre de partitions = offsets.topic.num.partitions (50 par défaut). Brokers y écrivent les commits.
Observation : kafka-console-consumer --bootstrap-server ... --topic __consumer_offsets --formatter "kafka.coordinator.group.GroupMetadataManager\$OffsetsMessageFormatter"
Nuance critique : Les offsets sont engagés par group.id + partition. Si vous supprimez un groupe, les offsets sont effacés du topic interne. La réinitialisation des offsets (--reset-offsets) ne supprime pas les données du topic source, elle modifie simplement le pointeur dans __consumer_offsets.

📈 Monitoring du lag – Pourquoi et comment ?
Lag = différence entre l’offset du producteur et l’offset consommé
Exemple: la dernière écriture dans la partition a l’offset 1500, le consommateur a commis l’offset 1200 → lag = 300 messages.
Commande : kafka-consumer-groups --bootstrap-server localhost:9092 --group my-group --describe

my-group        my-topic        0          1200            1500            300
my-group        my-topic        1          800             800             0
Un lag élevé et persistant indique un consommateur trop lent ou un sous-dimensionnement.
Bonnes pratiques de monitoring :
  • Alertes lorsque le lag dépasse un seuil (ex: 10 000 messages pendant 5 minutes).
  • Utiliser des métriques JMX comme kafka.consumer:type=consumer-fetch-manager-metrics,client-id=* → records-lag-max.
  • Augmenter le nombre de consommateurs (max = nombre de partitions) ou améliorer le traitement.
  • Une valeur de lag négative est impossible (CURRENT-OFFSET > LOG-END-OFFSET indique une anomalie).
✅ Bonnes pratiques recommandées
  • Définissez retention.ms en fonction de l’obligation légale (ex: 90 jours).
  • Utilisez cleanup.policy=compact pour les tables de référence (ex: dernières valeurs de configuration).
  • Pour les données critiques, min.insync.replicas=2 avec replication.factor=3.
  • Augmentez max.message.bytes si vous gérez des messages volumineux, mais attention à la mémoire.
  • Surveillez l’activité des segments (kafka-log-dirs --describe).

Producer Partitioner & Request Lifecycle – Deep Technical Dive
🎯 Pourquoi ce module est déterminant pour l’examen CCDAK ?
Le partitionneur détermine comment les messages sont répartis entre les partitions, impactant directement l’ordre, l’équilibrage de charge et la performance. Le cycle de vie d’une requête producteur (de la sérialisation à la réponse du broker) est un sujet fréquent dans les questions avancées. Environ 15% des questions « Fundamentals » portent sur ces mécanismes internes.
🧭 Qu’est-ce qu’un partitionneur ? Pourquoi est-ce essentiel ?
Objectif principal
Le partitionneur décide dans quelle partition un message sera écrit. Cette décision influence :
  • L’ordre des messages (par partition, l’ordre est garanti).
  • La distribution de la charge entre les brokers (leaders des partitions).
  • La parallélisation côté consommateur (chaque partition est lue par un seul consommateur par groupe).
Cas d’utilisation typiques
  • Clé = ID utilisateur → toutes les commandes d’un même utilisateur vont dans la même partition (ordre préservé).
  • Clé = null → distribution round-robin (ou sticky partitioner pour le batching).
  • Clé = timestamp de l’événement → les événements proches peuvent atterrir dans des partitions différentes.
📌 Default partitioner – Comportement sticky (Kafka ≥ 2.4)
Avant 2.4 : round-robin pur pour les clés null → chaque message alternait entre les partitions, ce qui provoquait une fragmentation des lots (un lot par partition) et une faible efficacité de compression.
Sticky Partitioner (actuel) : Pour les messages sans clé, le producteur choisit une partition et la « colle » (sticky) jusqu’à ce que le lot soit plein (batch.size) ou qu’un délai (linger.ms) soit écoulé. Ensuite, il en choisit une autre aléatoirement. Cela maximise la taille des lots, améliore le débit et la compression.
Implémentation interne (Java) – Simplifiée 
// Pour une clé non nulle : hash(key) % numPartitions
int partition = Utils.toPositive(Utils.murmur2(keyBytes)) % numPartitions;

// Pour clé nulle : sticky partitioner
if (nullKey) {
    if (currentStickyPartition == null || batchFullOrExpired) {
        currentStickyPartition = random.nextInt(numPartitions);
    }
    partition = currentStickyPartition;
}
💡 Le sticky partitioner utilise un générateur aléatoire mais en pratique il alterne entre partitions pour équilibrer la charge à long terme.
⚙️ Implémenter un partitionneur personnalisé
Classe org.apache.kafka.clients.producer.Partitioner : 
public class CustomPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        List partitions = cluster.partitionsForTopic(topic);
        int numPartitions = partitions.size();
        // Exemple : toujours dans la partition basée sur la longueur de la valeur
        return Math.abs(value.toString().length() % numPartitions);
    }
    @Override
    public void close() {}
    @Override
    public void configure(Map configs) {}
}
Configuration : props.put("partitioner.class", "com.example.CustomPartitioner");
Attention : Les partitionneurs personnalisés peuvent contourner l’ordre par clé et causer des déséquilibres. Testez toujours la distribution sur plusieurs millions de messages.
🔄 Request Lifecycle – Du producteur au broker (10 étapes détaillées)
  1. Sérialisation : key.serializer et value.serializer convertissent l’objet en byte[].
  2. Interception (optionnel) : ProducerInterceptor peut modifier le message avant l’envoi (ex: ajouter des headers).
  3. Détermination de la partition : appel au partitionneur (sticky ou personnalisé).
  4. Accumulation (RecordAccumulator) : le message est ajouté à un lot (batch) correspondant à la partition cible. Les lots sont conservés dans une mémoire tampon (buffer.memory, défaut 32 Mo).
  5. Déclenchement de l’envoi : quand le lot est plein (batch.size) ou après linger.ms, le lot est prêt.
  6. Thread Sender : un thread séparé (Sender) prend le(s) lot(s) et construit une requête ProduceRequest.
  7. Métadonnées du cluster : le producteur connaît le leader de chaque partition (via metadata.max.age.ms). Si les métadonnées sont périmées, il rafraîchit d’abord.
  8. Envoi réseau (Selector) : la requête est envoyée au broker leader via le réseau (NIO). Gère le pipeline des requêtes ( max.in.flight.requests.per.connection ).
  9. Traitement par le broker : le broker écrit le message dans le log local, le réplique vers les followers (ISR) selon acks (0,1,all).
  10. Réponse et callback : le producteur reçoit la réponse (succès ou erreur). Si retries > 0, certaines erreurs (timeout, leader not available) déclenchent une nouvelle tentative.

Quelles métadonnées sont gérées ?

✅ Bonnes pratiques (partitioner)
  • Utilisez des clés à haute cardinalité (UUID, ID client) pour une distribution uniforme.
  • Evitez les clés « chaudes » (p. ex. pays = US, FR, DE) sous peine de déséquilibre.
  • Pour l’ordre global strict, une seule partition suffit, mais limite le débit.

  • Une clé constante (ex: “same”) envoie tous les messages dans une seule partition → goulot d’étranglement.
  • Changer de partitionneur en production sans analyse d’impact.

  • Oublier que l’ajout de partitions après coup peut casser la répartition des clés existantes (pas de rééquilibrage automatique des anciens messages).
🚀 Optimisations : batching, linger.ms, compression
  • batch.size (défaut 16 Ko) : taille maximale d’un lot par partition. Un lot plus grand améliore le débit mais augmente la latence.
  • linger.ms (défaut 0) : délai d’attente pour remplir un lot. Avec 5-10 ms, on augmente le batching pour les flux peu denses.
  • compression.type (gzip, snappy, lz4, zstd) : la compression s’applique au lot entier. Le sticky partitionner aide la compression.
  • buffer.memory (32 Mo) : mémoire totale pour les lots non envoyés. Si dépassée, send() bloque ou lève BufferExhaustedException.

📦 Sérialisation et désérialisation (SerDes)
Pourquoi sérialiser ? Kafka transporte des tableaux d’octets (byte[]). Les sérialiseurs convertissent les objets Java en bytes, et les désérialiseurs font l’inverse. Les types primitifs et courants sont fournis : StringSerializer, IntegerSerializer, ByteArraySerializer, etc.
Sérialiseur personnalisé (exemple) 
public class UserSerializer implements Serializer<User> {
    @Override
    public byte[] serialize(String topic, User data) {
        byte[] serialized = new byte[8];
        ByteBuffer buffer = ByteBuffer.wrap(serialized);
        buffer.putInt(data.getId());
        buffer.putInt(data.getName().length());
        buffer.put(data.getName().getBytes());
        return serialized;
    }
}
Désérialiseur associé 
public class UserDeserializer implements Deserializer<User> {
    @Override
    public User deserialize(String topic, byte[] data) {
        ByteBuffer buffer = ByteBuffer.wrap(data);
        int id = buffer.getInt();
        int nameLen = buffer.getInt();
        byte[] nameBytes = new byte[nameLen];
        buffer.get(nameBytes);
        return new User(id, new String(nameBytes));
    }
}

Transactions, Keys & Headers – Deep Dive for Exactly-Once and Smart Messaging
🎯 Pourquoi ce module est décisif pour l’examen CCDAK ?
Les transactions permettent des écritures atomiques sur plusieurs partitions et topics, clé pour exactly‑once end‑to‑end. Les clés déterminent le partitionnement et l’ordre. Les headers ajoutent des métadonnées sans modifier la charge utile. Environ 20% des questions du domaine « Application Development » testent ces concepts. Une maîtrise fine des transactions (initTransactions, commit, isolation.level) est attendue.
🔒 Transactions Kafka : atomicité multi‑partition / multi‑topic
Qu’est-ce qu’une transaction Kafka ?
Une transaction regroupe plusieurs écritures (producer) dans une unité atomique : soit toutes sont visibles, soit aucune. Cela permet d’écrire dans plusieurs partitions (même de topics différents) et de commettre les offsets de consommateurs dans la même transaction (consume‑transform‑produce pattern).
Prérequis : enable.idempotence=true (automatiquement activé), un transactional.id unique pour chaque producteur.
Cas d’usage typiques
  • Atomicité bancaire : virement entre deux comptes (écritures dans deux topics).
  • Lecture d’un topic, transformation, écriture dans un autre avec recul de l’offset consommé.
  • Mise à jour de plusieurs états (ex: commande validée, stock mis à jour).
📚 API transactionnelle (Java) – Pas à pas
// Initialisation (une fois)
producer.initTransactions();

// Dans une boucle de traitement
producer.beginTransaction();
try {
    // Écritures producteur normales
    producer.send(new ProducerRecord<>("topic1", key1, value1));
    producer.send(new ProducerRecord<>("topic2", key2, value2));
    
    // Optionnel : commit des offsets consommés (consume‑transform)
    producer.sendOffsetsToTransaction(offsetsMap, consumer.groupMetadata());
    
    // Validation de la transaction
    producer.commitTransaction();
} catch (Exception e) {
    // Annulation
    producer.abortTransaction();
}
💡 sendOffsetsToTransaction lie le décalage du consommateur aux écritures du producteur, exact‑once end‑to‑end.
Configurations obligatoires
  • enable.idempotence=true (ou sera forcé)
  • transactional.id : identifiant unique (ex: "app1-tx-1")
  • acks=all (implicite avec idempotence)
  • max.in.flight.requests.per.connection=1 (recommandé, mais ≤5 avec idempotence)
⚠️ Durabilité transactionnelle :
  • Le broker transaction coordinator (un des brokers) gère l’état des transactions dans le topic __transaction_state (nombre de partitions = transaction.state.log.num.partitions).
  • transaction.timeout.ms (défaut 1 minute) : temps max pour terminer une transaction.
  • Si le producteur crash, la transaction expire après transaction.timeout.ms et est annulée.
👁️ Isolation.level – Comment lire des messages transactionnels
ValeurComportementCas d’usage
read_uncommitted (défaut) Lit tous les messages, même ceux non encore commis (dirty reads possibles). Performances maximales, données non critiques.
read_committed Lit uniquement les messages appartenant à des transactions commises. Les messages de transactions avortées ou en cours sont invisibles. Exactly‑once, paiements, commandes.
Configuration consommateur pour read_committed : 
props.put("isolation.level", "read_committed");
Remarque : les messages hors transaction (producteur normal) sont toujours visibles.
🔑 Clés Kafka : partitionnement, ordre, compactage
Effet des clés sur le partitionnement
  • Clé non nulle : hash(key) % nombre_partitions → toutes les messages avec la même clé vont dans la même partition (ordre garanti).
  • Clé nulle : sticky partitioner (ou round‑robin avant 2.4).
  • Changer de clé après coup peut casser l’ordre logique.
Compactage (cleanup.policy=compact)
Seuls les messages avec clé sont concernés. Pour chaque clé, seul le dernier message (valeur) est conservé. Les messages sans clé sont ignorés (ne seront jamais compactés).
Utile pour les tables de référence (ex: dernier prix d’un produit).
Bonnes pratiques sur les clés :
  • Choisissez des clés à haute cardinalité (UUID, ID client) pour une distribution homogène.
  • Évitez les clés constantes (ex: "FR") – une seule partition sera utilisée.
  • Ne changez pas la sémantique de la clé en production (les messages anciens ne seront pas répartis à nouveau).
  • Pour les compacted topics, assurez-vous que chaque clé a une valeur, sinon un message "null" (tombstone) supprime la clé.
🏷️ Headers : ajouter du contexte au message
Headers : paires clé‑valeur (tout en bytes). Ils n’affectent ni le partitionnement ni la sérialisation de la valeur. Utiles pour :
  • Routing : ajouter un header "source" (ex: "service=payment").
  • Tracing : propagation de l’ID de transaction (ex: "traceId", "spanId").
  • Métadonnées de sécurité : signature, timestamp custom.
Exemple d’envoi avec headers (producteur) : 
ProducerRecord record = new ProducerRecord<>("topic", "key", "value");
record.headers().add("source", "app1".getBytes());
record.headers().add("timestamp", String.valueOf(System.currentTimeMillis()).getBytes());
producer.send(record);
Lecture des headers (consommateur) : 
for (ConsumerRecord record : records) {
    Headers headers = record.headers();
    Header sourceHeader = headers.lastHeader("source");
    if (sourceHeader != null) {
        String source = new String(sourceHeader.value());
    }
}
💡 Les headers sont accessibles dans Kafka Streams via ProcessorContext.headers().
🧪 Cas d’étude : Transaction avec clé et en-tête
Scénario : Un service de commandes lit les événements du topic "orders", applique une validation, et écrit le résultat dans "validated_orders" avec la même clé (orderId) pour garantir l’ordre. Il ajoute un header "validation_timestamp". Le tout en transaction pour ne créer ni perte ni doublon. 
producer.initTransactions();
consumer.subscribe("orders");
while (true) {
    ConsumerRecords records = consumer.poll(Duration.ofMillis(1000));
    producer.beginTransaction();
    try {
        for (ConsumerRecord rec : records) {
            Order order = rec.value();
            Order validated = validate(order);
            ProducerRecord outRecord = 
                new ProducerRecord<>("validated_orders", order.getId(), validated);
            outRecord.headers().add("validation_ts", 
                                    String.valueOf(System.currentTimeMillis()).getBytes());
            producer.send(outRecord);
        }
        Map offsets = offsetsForRecords(records);
        producer.sendOffsetsToTransaction(offsets, consumer.groupMetadata());
        producer.commitTransaction();
    } catch (Exception e) {
        producer.abortTransaction();
    }
}

Kafka Streams Basics – Stateful vs Stateless, Exactly‑Once, and Core Concepts
🎯 Pourquoi Kafka Streams est‑il incontournable pour l’examen CCDAK ?
Kafka Streams est une bibliothèque client légère pour le traitement de flux en temps réel. Elle représente environ 25% des questions du domaine « Kafka Streams ». Contrairement aux consumers/producers classiques, elle offre des états (state stores), des fenêtres, et une sémantique exactly‑once sans cluster externe (contrairement à Flink ou Spark). Maîtriser les différences entre stateful/stateless, exactly‑once_v2, et les cas d’usage est essentiel.

💡 Les state stores sont persistants par défaut (RocksDB). Ils sont restaurés après redémarrage via les changelog topics.
🎯 Exactly‑once semantics (EOS) dans Kafka Streams
Configuration : 
props.put(StreamsConfig.PROCESSING_GUARANTEE_CONFIG, 
          StreamsConfig.EXACTLY_ONCE_V2);
Versions :
  • exactly_once (v1) : transactions, mais plus lourd.
  • exactly_once_v2 (recommandé) : optimisation, réduction de latence.
⚠️ Ce que exactly‑once garantit :
  • Chaque enregistrement d’entrée est traité exactement une fois (ni duplicate, ni perte).
  • Les mises à jour des state stores et les écritures dans les topics de sortie sont atomiques (transaction).
  • En cas d’échec, aucun doublon n’est produit.
Inconvénient : légère surcharge de latence (~10-15%). Pour des cas où les doublons sont acceptables, utilisez at_least_once.
Conséquence de exactly_once_v2 sur le consumer : Les consommateurs internes de Kafka Streams utilisent isolation.level=read_committed. Les résultats ne sont visibles qu’après commit de la transaction.
🏪 State stores – Gestion des états internes
Types de state stores :
  • RocksDB : par défaut, persistant sur disque (performant pour grandes données).
  • InMemoryKeyValueStore : en mémoire, rapide mais volatile (ne restaure pas après crash).
  • WindowStore : pour les fenêtres (basée sur RocksDB).
  • SessionStore : pour les sessions.
Changelog : Chaque state store est sauvegardé dans un topic de changelog (préfixe appId-) permettant la restauration après redémarrage ou rééquilibrage.
Accès à un state store personnalisé : 
KeyValueStore store = Stores.keyValueStoreBuilder(
    Stores.persistentKeyValueStore("my-store"),
    Serdes.String(),
    Serdes.Long()
).build();
Nuance : La restauration d’un state store peut prendre du temps (replay du changelog). Pour réduire, utilisez num.standby.replicas (répliques chaudes). Une instance avec réplique standby peut prendre le relais immédiatement.
📈 Scalabilité de Kafka Streams
Modèle :
  • num.stream.threads (défaut 1) : nombre de threads au sein d’une même JVM.
  • Chaque thread exécute des tâches (une tâche par partition des topics d’entrée).
  • Plusieurs instances de l’application (même application.id) forment un cluster Streams.
  • Les partitions sont réparties entre les threads et les instances (rebalance coopératif).
Règle : Le parallélisme maximal = nombre total de partitions des topics d’entrée.
📊 KStream vs KTable – Aperçu
  • KStream : flux d’enregistrements (tous les événements, comme un topic).
  • KTable : vue matérialisée par clé (ne retient que la dernière valeur par clé, comme une base de données).
  • GlobalKTable : KTable complètement répliquée sur chaque tâche (utile pour les petites tables de référence).

KStream vs KTable vs GlobalKTable & DSL vs Processor API – Complete Guide
🎯 Why this module is critical for the CCDAK exam?
Understanding the dual nature of streams and tables is the foundation of Kafka Streams. KStream vs KTable vs GlobalKTable determines how data is interpreted (append-only vs upsert). The choice between high‑level DSL and low‑level Processor API defines the flexibility vs complexity trade‑off. Approximately 25% of Kafka Streams questions target these concepts.
📊 KStream, KTable, GlobalKTable – Core abstractions
ConceptNatureChangelog / stateUse case
KStream Flow of records (every event) No state – each record processed independently Raw event processing, filtering, transformation, windowed aggregation
KTable Materialized view per key (latest value only) Stateful changelog topic (compacted) Reference data, aggregates, join enrichment (KStream-KTable)
GlobalKTable Fully replicated KTable on every task Entire changelog replicated across all instances Small lookup tables (e.g., country codes, configuration)
Example: KTable from a topic (compacted) and KStream from a topic (regular) 
// 'user-profile' topic is compacted (each key last value)
KTable profileTable = builder.table("user-profile");

// 'user-clicks' topic is a stream of all clicks
KStream clickStream = builder.stream("user-clicks");

// Join: enrich each click with the latest profile
KStream enriched = clickStream.join(profileTable,
    (click, profile) -> new EnrichedClick(click, profile));
💡 The join uses the same key. KTable lookup is local (state store). GlobalKTable would replicate the whole table, useful for joins where the key is not the same (foreign-key like).
🔍 KStream vs KTable – deep dive
KStream characteristics
  • Every record is an independent event.
  • No compaction – all events remain in the changelog (non‑compacted).
  • Operations: filter, map, flatMap, join (windowed), aggregate (windowed).
  • When you use groupByKey on a KStream, it becomes a KGroupedStream → you must call aggregate, count, reduce which produce a KTable.
KTable characteristics
  • Represents a changelog stream: each key has a single current value (upsert).
  • Underlying topic must be compacted (or compact+delete).
  • Operations: filter, mapValues, join (with KStream or KTable), toStream().
  • KTables can be materialized (state store) for interactive queries.
Critical nuance for the exam: When converting a KStream to a KTable via groupByKey().count(), the result is a KTable that updates over time. The intermediate aggregation uses a state store. If you want a changelog of counts, you get only the latest count per key.
🌐 GlobalKTable – fully replicated table
Characteristics:
  • Full copy of the table's changelog on every Kafka Streams task (every thread/instance).
  • Unequal key join possible: you can join a KStream with a GlobalKTable on a different key (foreign key pattern).
  • Use only for small datasets (e.g., < 10k records) because every instance holds the full table.
Example: join click stream with country table (different key) 
KStream clicks = builder.stream("clicks");
GlobalKTable countries = builder.globalTable("countries");
KStream enriched = clicks.join(countries,
    (key, click) -> click.getCountryCode(), // key extraction for lookup
    (click, countryInfo) -> new EnrichedClick(click, countryInfo));
📘 DSL (Domain Specific Language) – Functional style
Advantages:
  • Concise and expressive – similar to Java streams or Spark.
  • Built-in windowing, joins, aggregations, and state stores.
  • Automatic optimization (Kafka Streams can optimize topology).
  • Less boilerplate code.
Example: Count clicks per user in 5-minute tumbling windows 
KStream clicks = builder.stream("clicks");
KTable, Long> counts = clicks
    .groupByKey()
    .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5)))
    .count();
counts.toStream().to("click-counts");
💡 DSL internally generates a Processor topology. It covers ~90% of real‑time processing needs.
⚙️ Processor API – For advanced use cases
When to use Processor API:
  • Custom state management (e.g., multiple stores).
  • Punctuation (scheduled actions via context().schedule()).
  • Fine‑grained control over forwarding (forward to multiple children).
  • Complex event processing not expressible in DSL.
Example: custom processor that emits a periodic summary 
public class PeriodicForwardProcessor implements Processor {
    private ProcessorContext context;
    private KeyValueStore store;
    
    @Override
    public void init(ProcessorContext ctx) {
        context = ctx;
        store = (KeyValueStore) context.getStateStore("my-store");
        context.schedule(Duration.ofSeconds(10), PunctuationType.WALL_CLOCK_TIME, 
            timestamp -> {
                // Forward aggregate every 10 seconds
                context.forward("alert", store.get("total"));
                context.commit();
            });
    }
    
    @Override
    public void process(String key, Long value) {
        store.put(key, value);
    }
}
Nuances – Using Processor API with DSL
You can mix both: use .transform() or .process() inside a DSL topology to inject a custom processor. This gives the best of both worlds: readable DSL plus low-level control.

📝 Typical CCDAK questions on Streams/Tables & DSL/Processor
  1. Question: You have a compacted topic 'user-last-login'. Which abstraction should you use to capture the latest login per user? Answer: KTable (via builder.table()).
  2. Question: You need to join a stream of orders with a reference table of products. The product table has 10,000 rows. Which approach? Answer: KTable for products and KStream-KTable join. GlobalKTable would work but is overkill.
  3. Question: You need to emit an alert every 30 seconds regardless of incoming messages. Which API? Answer: Processor API with punctuation (schedule).
  4. Question: A KStream and a KTable are joined. What does the join produce? Answer: A new KStream (or KTable depending on join type). Usually KStream-KTable join yields a KStream.
  5. Question: What happens when you call `toTable()` on a KStream? Answer: It turns the KStream into a KTable (last value per key) using a state store.
✅ Best practices
  • Use KStream for raw events, logs, clickstreams.
  • Use KTable for reference data, aggregates, latest state.
  • Use GlobalKTable only for small, mostly static tables that need foreign-key joins.
  • Prefer DSL unless you need punctuation or custom state logic.
  • When mixing DSL and Processor API, encapsulate processors in `.process()` steps.
⚠️ Pitfalls to avoid
  • Using a KTable on a topic that is not compacted will cause infinite growth of the state store.
  • GlobalKTable on a large topic (e.g., 1M records) will memory‑crash your instances.
  • Forgetting that a KTable changelog is also a stream: you can materialize it multiple times.
  • Using Processor API for simple mapping – overkill and harder to maintain.

Kafka Connect – Deep Dive: Connectors, SMT, Workers, REST, Offsets & Scalability
🎯 Why Kafka Connect is essential for the CCDAK exam?
Kafka Connect is a framework for streaming data between Kafka and external systems (databases, files, cloud storage). It’s a core component of the Confluent ecosystem, and exam questions often cover source/sink connectors, Single Message Transforms (SMT), worker configuration, and the REST API. Approximately 15% of the exam focuses on Connect, with emphasis on fault tolerance, offsets management, and scalability.
🏗️ Kafka Connect architecture – Workers, Connectors, Tasks
Worker
A JVM process that runs connectors and tasks. Two modes:
  • Standalone – single worker, configuration via files, simple for development.
  • Distributed – multiple workers, automatic load balancing and fault tolerance, uses Kafka for internal storage.
Connector & Task
  • Connector : defines the configuration and splits work into tasks.
  • Task : actually moves data (source to Kafka or Kafka to sink). Each task runs in its own thread. Number of tasks can be increased for parallelism.
Nuance: In distributed mode, workers coordinate via Kafka topics: config.storage.topic, offset.storage.topic, status.storage.topic. All workers must have the same group.id.

💡 Source connectors track offsets per record (e.g., table row). Sink connectors use consumer offsets stored in Kafka.
⚙️ Worker configuration – essential properties
Required settings in connect-distributed.properties :
  • bootstrap.servers : Kafka brokers for internal topics and data.
  • group.id : Consumer group for workers (default "connect-cluster").
  • config.storage.topic : Stores connector configurations (e.g., "connect-configs").
  • offset.storage.topic : Stores source offsets (e.g., "connect-offsets").
  • status.storage.topic : Stores connector/task status (e.g., "connect-status").
  • key.converter / value.converter : Default converters for all connectors (can be overridden per connector).
  • rest.port : Port for REST API (default 8083).
  • plugin.path : Directory containing connector JARs.
Important: The internal topics must be created automatically if not exists (if auto.create.topics.enable=true). They require replication factor ≥ number of workers. Offsets topics should be compacted (to keep only latest offset per source partition).
📈 Scaling Kafka Connect – horizontal and vertical
Horizontal scaling
  • Add more worker nodes (same group.id).
  • Connectors and tasks are automatically redistributed across workers (rebalance).
  • Maximum parallelism = sum of tasks.max across all connectors.
Increasing tasks per connector 
# Update connector config via REST
PUT /connectors/my-connector/config
{
  "connector.class": "...",
  "tasks.max": "4"
}
For source connectors, tasks are assigned to different tables or partitions. For sink connectors, tasks.max cannot exceed number of source topic partitions.
Scaling best practices: Distribute connectors across workers, but avoid overloading Kafka with too many tasks. Monitor task failures and restart tasks automatically (via REST).
💾 Connect offsets – how they work
Source connectors: Store offsets per source partition (e.g., database table row ID or timestamp). Offsets are stored in the offset.storage.topic. When the connector restarts, it resumes from the last committed offset.
Sink connectors: Use standard Kafka consumer groups (offsets stored in __consumer_offsets). They read from Kafka and write to external system.
Resetting source offsets: Use the Connect REST API (since Kafka 2.3+) or manually delete offset entries from the offset storage topic.

🚨 Error handling – tolerating bad records
Configuration options for error tolerance
  • errors.tolerance=none : fail connector/task on first error.
  • errors.tolerance=all : skip failed records and continue.
  • errors.deadletterqueue.topic.name : send failed records to a DLQ topic.
  • errors.retry.timeout / errors.retry.delay.max.ms : retry on transient errors.
DLQ record includes:
  • Original record (key/value/headers)
  • Error message and stack trace
  • Cause of failure (e.g., deserialization error)
Use DLQ to debug issues without stopping the pipeline.
📝 Typical CCDAK questions – Connect
  1. How to increase parallelism of a source connector reading from a database table? Increase tasks.max and ensure the source connector supports partitionable queries (e.g., JDBC connector with incrementing column). Each task reads a segment of the table.
  2. What happens if a worker in distributed mode dies? Its connectors and tasks are rebalanced to other workers; offsets are stored in Kafka, so processing resumes from the last committed offset.
  3. Can you add an SMT to an existing connector without restarting the whole cluster? Yes, via PUT /connectors/{name}/config – the connector will reconfigure and tasks restart.
  4. Which converter should you use to write Avro data with Schema Registry? AvroConverter with schema.registry.url property.
  5. What is the purpose of the offset.storage.topic? It stores source offsets for source connectors, allowing them to resume from the exact point after a restart.

  • Set tasks.max proportional to source partitions (e.g., number of DB tables or shards).
  • Use JsonConverter or AvroConverter with Schema Registry for evolvable schemas.
  • Enable error tolerance and DLQ for production pipelines.
  • Monitor connector status via REST and set up alerts on FAILED state.
  • Use separate workers for source and sink connectors for performance isolation.

  • Forgetting to set key.converter/value.converter per connector leading to serialization errors.
  • Not configuring the internal topics' replication factor (can lose worker coordination).

Consumer Lag, Performance Metrics, Log4j & Alerting
Lag = producer offset - consumer offset. JMX metrics: request-latency, bytes-in, ISR shrinks. Log4j config for clients. Alerts: lag > threshold, under-replicated partitions.
🚨 Common errors: NotLeaderForPartition, TimeoutException, OffsetOutOfRange.
📈 Lag spikes indicate consumer or processing bottleneck.
🔔 Alert on consumer lag > 5000 for 10 min; use Prometheus + Grafana.
Data Formats (Avro/JSON/Protobuf), Error Handling & Topic Evolution
Avro (schema evolution), JSON (schemaless), Protobuf (efficient). Error handling: retries, DLQ, error handlers. Modifying topics: increase partitions only (never decrease), dynamic config changes.
💡 Can we decrease partitions? No, it breaks ordering. Plan partition count for future.
⚠️ Adding partitions can affect key order; use consistent hashing if needed.
✅ Best practice: schema registry with compatibility BACKWARD; implement dead-letter topic for bad records.
Schema Registry & Compatibility
Schema Registry: central store for Avro/Protobuf/JSON schemas. Compatibility types: BACKWARD (consumers read new schema), FORWARD, FULL, NONE. Schema IDs embedded in messages.
🔑 Exam Q: Which compatibility allows consumers using old schema to read new data? BACKWARD.
🧩 Auto schema registration; subjects: topic-name-value/key. Use REST API for subject versions.
📌 Best practice: use BACKWARD or FULL compatibility, enforce schema evolution rules.
Kafka Security Deep Dive: ACLs, OAuth, mTLS
ACLs (Allow/Deny) on resources: Topic, Group, Cluster. Principal mapping. OAuth/OIDC with OAUTHBEARER; mTLS authentication. Super users bypass ACLs.
❓ How to list ACLs? kafka-acls --list --authorizer-properties. Define resource pattern: LITERAL/PREFIXED.
⚠️ ACLs are evaluated in order; default DENY. OAuth requires token validation.
✅ Grant least privilege; use SCRAM for dynamic user management; audit logs.
MirrorMaker 2 & Cross-Cluster Replication
MM2 replicates topics across clusters, supports active/active, offset sync, ACL sync. Uses Connect framework. Config: source->target, heartbeat/checkpoints topics for consumer offset translation.
🌍 How to handle failover? Use MirrorMaker2 with bidirectional replication, consumer groups mirrored via offset sync.
🧠 Must avoid cyclic replication; configuration: clusters, sync.topic.acls.
✅ Best practice: enable replication of consumer groups, use exactly-once delivery for MM2.
Tiered Storage, Quotas & Reassignment
Tiered storage: offload old segments to object storage (S3). Quotas: byte-rate (network), request rate, CPU throttling. Partition reassignment: kafka-reassign-partitions, throttle to avoid cluster impact.
🔧 How to move partitions? generate reassign JSON, execute with throttle.
⚙️ Tiered storage reduces retention cost; quotas protect noisy neighbors.
📌 Set client quotas to ensure fairness; use incremental reassignment.
Advanced Monitoring: JMX, Log4j patterns, Metrics Analysis
Expose JMX metrics: MBean kafka.consumer:type=consumer-fetch-manager-metrics (records-lag-max). Log4j configuration per logger (kafka.authorizer.logger). Use Burrow or Kafka Lag Exporter.
📊 Metric for consumer group health: `kafka.consumer:type=consumer-coordinator-metrics` (assigned-partitions).
⚠️ High `request‑queue‑size` indicates broker overload; `ISR` shrinks signal replica issues.
✅ Set up structured logging (JSON), monitor critical metrics: under-replicated partitions, request latency, consumer lag, GC pauses.
